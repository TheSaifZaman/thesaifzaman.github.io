The Challenge: Design a URL Shortener at Scale

Let us tie together every concept from this book by designing a URL shortener -- a service like bit.ly that takes long URLs and creates short, memorable links, then redirects users who visit the short link to the original URL.

Functional requirements: (1) Given a long URL, generate a unique short URL. (2) Given a short URL, redirect to the original long URL. (3) Track analytics: click counts, geographic data, referrer information, timestamps.

Non-functional requirements: (1) Handle billions of URLs in storage. (2) Low-latency redirects -- sub-10ms for cache hits, sub-50ms including a database lookup. (3) High write throughput -- thousands of new URLs created per second. (4) High availability -- the redirect path must have 99.99% uptime. (5) Analytics data must be eventually consistent but not lose events.

This exercise touches every chapter: data models and encoding (Ch2-4), replication and partitioning (Ch5-6), transactions (Ch7), distributed system challenges (Ch8-9), and batch/stream processing for analytics (Ch10-12).

2

Data Model and Storage

The core data is a key-value mapping: short code maps to original URL. The schema is simple:

shortCode (primary key, string, 7 chars), originalURL (string, up to 2048 chars), createdAt (timestamp), userId (optional, for authenticated users), expiresAt (optional).

Short code generation: Use Base62 encoding (a-z, A-Z, 0-9) for human-friendly codes. A 7-character Base62 code gives 62^7 = 3.5 trillion possible values -- far more than enough. To generate codes, use an auto-incrementing counter (simple but requires coordination) or pre-generated ID ranges (each application server gets a block of IDs, no coordination needed for each write).

Storage engine choice: A hash-partitioned key-value store is ideal for the core mapping. The access pattern is almost entirely point lookups by key (short code), which is the optimal case for hash indexes. Options include DynamoDB, Cassandra, or even Redis (with persistence) for the hot path.

Encoding: For internal service-to-service communication, use a compact binary format like Protocol Buffers (Protobuf) for efficiency. For the public-facing API, use JSON for developer friendliness and broad compatibility. Schema evolution is handled by Protobuf's field numbering.

Distribution Strategy

Partitioning: Hash the short code to determine which partition stores the URL mapping. Hash partitioning ensures uniform distribution across nodes -- unlike range partitioning, which could create hot spots if codes are sequential. With billions of URLs, you need dozens to hundreds of partitions spread across multiple machines.

Replication: Use leader-follower replication for the write path (creating new short URLs). For the read path (redirecting), use read replicas distributed geographically. Since URL mappings are immutable once created (a short code always maps to the same URL), stale reads from followers are not a problem -- the data does not change.

Caching: The redirect path is extremely read-heavy (reads outnumber writes by 100:1 or more). An in-memory LRU cache (Redis/Memcached) in front of the database serves hot URLs with sub-millisecond latency. The working set (frequently accessed URLs) is much smaller than the total dataset, so a modest cache can handle most traffic. Cache invalidation is trivial because URL mappings are immutable.

Geographic distribution: For global low latency, deploy read replicas and caches in multiple regions. A CDN can cache redirects at edge locations closest to users. DNS-based routing sends users to the nearest datacenter. The write path can be centralized (or use multi-leader across regions) since writes are far less frequent than reads.

Distributed Architecture with Caching

Read path: CDN/edge -> API -> Redis cache -> DB fallback. Write path: API -> ID generator -> partitioned DB. Analytics: click events -> Kafka -> Flink -> analytics DB.

4

Consistency, Transactions, and Streams

Uniqueness of short codes: Each short code must map to exactly one URL. Using a counter or pre-generated ID ranges avoids collisions without expensive distributed coordination. Each application server is assigned a block of IDs (e.g., server A gets IDs 1-10000, server B gets 10001-20000). When a server exhausts its range, it requests a new block from a central coordinator. Within its range, the server can generate codes independently -- no distributed locks or consensus needed per write.

Analytics via stream processing: Every redirect generates a click event sent to Kafka. A Flink stream processor consumes these events and computes real-time analytics: clicks per URL in tumbling time windows, geographic distribution, top referrers. The aggregated results are written to an analytics database (e.g., ClickHouse) for dashboards and API queries.

Batch processing for historical analytics: Periodically (daily or weekly), a batch job processes the complete click event history to compute long-term trends, detect anomalies, and generate reports. The batch results complement real-time analytics -- batch is authoritative, stream is timely.

CDC from write database to analytics: New URL creations are captured via CDC and published to Kafka. The analytics pipeline can join click events with URL metadata (creation date, user who created it) to produce richer analytics -- for example, "URLs created this week received X clicks on average."

5

Full Architecture and Trade-offs

Putting it all together, the complete architecture has two main paths:

Write path (create short URL): Client sends POST to API server. The API server obtains a unique ID from its pre-allocated range, encodes it as Base62, stores the mapping in the partitioned database, and returns the short URL. CDC captures the new row and publishes it to Kafka for downstream consumers.

Read path (redirect): Client visits the short URL. The CDN/edge may serve the redirect from its cache (fastest). Otherwise, the API server checks the Redis cache. On cache miss, it queries the database partition (determined by hashing the short code). The redirect is returned as an HTTP 301 (permanent) or 302 (temporary). A click event is asynchronously sent to Kafka.

Monitoring: Track latency percentiles (p50, p95, p99) for both read and write paths. Monitor error rates by type (404 for unknown codes, 500 for internal failures). Watch for partition hotspots -- if one partition gets more traffic, rebalance. Track Kafka consumer lag to ensure the analytics pipeline keeps up.

Schema evolution: The public API uses versioned JSON endpoints (v1, v2). Internal services use Protobuf with backward/forward-compatible schema changes (adding optional fields). Rolling deployments ensure zero-downtime updates.

Key Trade-offs

Every architecture decision involves trade-offs. The key is matching trade-offs to your specific requirements -- there is no universal "best" design.

Key insight: System design is the art of trade-offs. No single technology solves every problem -- the best architectures compose specialized tools, connected by well-defined data flows, with clear understanding of what each trade-off buys and costs. The principles from every chapter of DDIA -- encoding, storage, replication, partitioning, transactions, consistency, batch, and stream processing -- all come together in real system design.

Capstone Summary

  • URL shortener requirements: Billions of URLs, low-latency redirects, high write throughput, analytics.
  • Data model: Key-value store (shortCode -> originalURL), Base62 encoding for compact 7-character codes.
  • ID generation: Pre-allocated ranges per server to avoid distributed coordination per write.
  • Partitioning: Hash by short code for uniform distribution across database nodes.
  • Replication: Leader-follower for writes, read replicas + Redis cache + CDN for the redirect path.
  • Caching: LRU cache serves hot URLs; immutable mappings mean no invalidation complexity.
  • Analytics: Click events -> Kafka -> Flink (real-time windowed aggregation) -> analytics DB.
  • CDC: New URL creations streamed via CDC to enrich analytics with URL metadata.
  • Monitoring: Latency percentiles, error rates, partition balance, Kafka consumer lag.
  • Trade-offs: Consistency vs latency, storage vs computation, availability vs complexity, timeliness vs accuracy.

Key Terms

Base62
Encoding using a-z, A-Z, 0-9 for URL-safe, compact short codes.
LRU Cache
Least Recently Used eviction -- keeps hot items in memory, evicts cold ones.
Pre-allocated Ranges
Each server gets a block of IDs, enabling local ID generation without coordination.
Consumer Lag
How far behind a stream consumer is from the latest produced event.
301 vs 302
301 = permanent redirect (cacheable); 302 = temporary (allows tracking each click).
Power-Law Distribution
A few URLs get most traffic -- makes caching extremely effective.
