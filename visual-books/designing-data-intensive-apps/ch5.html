Replication

~25 min read Distributed Data

1

Leaders and Followers

Replication means keeping a copy of the same data on multiple machines connected via a network. The most common approach is leader-based replication (also called master-slave or active-passive). One replica is designated the leader: all writes go to the leader first, which writes the data to its local storage and then sends the change to all followers via a replication log or change stream. Followers apply writes in the same order the leader processed them.

Synchronous vs asynchronous replication: With synchronous replication, the leader waits for at least one follower to confirm the write before reporting success to the client. This guarantees the follower has an up-to-date copy, but if that follower is slow or down, writes are blocked. In practice, semi-synchronous is common: one follower is synchronous (guaranteeing an up-to-date copy on at least two nodes), and the rest are asynchronous.

Setting up new followers without downtime: take a consistent snapshot of the leader's database, copy it to the new follower, then have the follower request all data changes that happened since the snapshot (using the snapshot's exact position in the replication log).

Handling node outages: If a follower crashes, it performs catch-up recovery by replaying all changes from its log since its last processed position. If the leader fails, failover promotes a follower to leader. Failover is fraught with pitfalls: the new leader may lack some writes (data loss), split-brain scenarios, and choosing the right timeout for declaring a leader dead.

Leader-Follower Replication

Writes go to the leader only. Reads can go to any replica. Semi-synchronous: one follower is sync, others are async.

2

Problems with Replication Lag

If an application reads from an asynchronous follower, it may see stale data because the follower hasn't caught up with the leader. This is "eventually consistent" -- if you stop writing and wait, followers will eventually catch up. But the "eventually" can be seconds, minutes, or more under load.

Three key problems emerge from replication lag:

Read-your-own-writes: A user writes data (sent to the leader) then reads it back (served by a follower that hasn't replicated yet) -- they see their data vanished. Solution: read the user's own data from the leader, or track timestamps and wait for followers to catch up.

Monotonic reads: A user makes two reads that hit different followers -- the second follower is further behind, so the user sees data "go back in time." Solution: ensure each user always reads from the same replica (e.g., hash user ID to replica).

Consistent prefix reads: If a sequence of causally related writes arrives at different followers in different orders, an observer might see the answer before the question. Solution: ensure causally related writes go to the same partition.

Multi-Leader Replication

In single-leader replication, the leader is a bottleneck and a single point of failure. Multi-leader replication (master-master or active-active) allows more than one node to accept writes. Each leader simultaneously acts as a follower to the other leaders.

Use cases: Multi-datacenter operation (one leader per datacenter, reducing write latency for local users). Offline-capable clients (each device is a "leader" for its local data, syncing later -- think calendar apps). Collaborative editing (each user's changes are like writes to a local leader, with conflict resolution).

The biggest problem is write conflicts. If two leaders concurrently modify the same record differently, you need conflict resolution. Strategies include: last-write-wins (LWW) -- pick the write with the highest timestamp, discarding others (data loss!); merge values -- e.g., concatenate conflicting edits; custom resolution logic -- application-specific code runs on read or write to resolve conflicts.

Multi-Leader Replication Topologies

Circular and star topologies have single points of failure. All-to-all is most fault-tolerant but can suffer from causality ordering issues.

4

Leaderless Replication

Dynamo-style systems (Amazon Dynamo, Cassandra, Riak, Voldemort) abandon the concept of a leader entirely. The client sends writes to several replicas directly (or via a coordinator that does not enforce ordering). Reads also query several nodes in parallel and use version numbers to determine the most recent value.

Quorum: If there are n replicas, every write must be confirmed by w nodes, and every read must query r nodes. As long as w + r > n, you are guaranteed to read at least one node with the latest value. Common choice: n=3, w=2, r=2.

Repair mechanisms: Read repair -- when a client reads from multiple nodes and detects a stale value, it writes the newer value back to the out-of-date node. Anti-entropy process -- a background process constantly scans for differences between replicas and copies missing data.

Sloppy quorums and hinted handoff: If some of the designated n nodes are unreachable, writes can be temporarily accepted by other nodes (sloppy quorum). Once the designated node is back, the temporary node hands off the data (hinted handoff). This increases write availability but means quorum guarantees are weaker.

Concurrent writes: When two clients write to the same key simultaneously, you need a resolution strategy. Last-write-wins (LWW) uses timestamps but silently drops writes. Version vectors track the causal history per replica, allowing the application to merge truly concurrent writes without data loss.

5

Choosing a Replication Strategy

Each replication approach makes different trade-offs:

Single-leader: Simple to understand, no write conflicts, strong consistency guarantees (if reading from the leader). The leader is a bottleneck for writes and a single point of failure. Failover can be complex. Best when strong consistency matters and write throughput is manageable.

Multi-leader: Better write performance (writes accepted at any datacenter), better tolerance of datacenter outages and network problems. But conflict resolution is complex and error-prone. Best for multi-datacenter deployments and offline-first applications.

Leaderless: High availability for writes (no single point of failure), good performance for write-heavy workloads. But eventual consistency only, and concurrent write resolution is complex. Tunable consistency via quorum parameters. Best when availability and partition tolerance are paramount (AP in CAP theorem).

Key insight: There is no universally "best" replication strategy. The right choice depends on your application's requirements for consistency, availability, latency, and operational complexity. Most real-world systems combine strategies -- e.g., single-leader within a datacenter, multi-leader across datacenters.

Chapter 5 Summary

  • Leader-based replication: One leader accepts all writes, followers replicate asynchronously (or semi-synchronously).
  • Replication lag causes three problems: lost own writes, non-monotonic reads, and violated causal ordering.
  • Multi-leader allows writes at multiple nodes -- useful for multi-DC, offline clients, and collaborative editing -- but creates write conflicts.
  • Conflict resolution: LWW (data loss risk), merge values, or custom application logic.
  • Leaderless (Dynamo-style): Reads and writes go to multiple nodes; quorum condition w + r > n ensures overlap.
  • Read repair and anti-entropy fix stale replicas in leaderless systems.
  • Sloppy quorums trade consistency for availability when nodes are unreachable.
  • Version vectors track causality per replica, enabling correct merging of concurrent writes.
  • Trade-offs: Single-leader = simple + consistent; Multi-leader = performant + complex; Leaderless = available + eventually consistent.

Key Terms

Replication Lag
Delay between a write on the leader and its visibility on a follower.
Failover
Promoting a follower to leader when the existing leader fails.
Quorum
Minimum number of nodes that must acknowledge a read/write (w + r > n).
Split-Brain
Two nodes both believe they are the leader, accepting conflicting writes.
Version Vector
Per-replica version numbers tracking causal history of a data item.
Hinted Handoff
Temporarily storing writes on behalf of an unreachable node, delivering them when it recovers.
