Chapter 1

Reliable, Scalable, and Maintainable Applications

~25 min read Foundations

1

Thinking About Data Systems

Modern applications are data-intensive, not compute-intensive. The bottleneck is usually the amount, complexity, and rate of change of data -- not raw CPU cycles. A typical application stitches together several components: databases (store data so it can be found later), caches (remember the result of an expensive operation to speed up reads), search indexes (allow users to search and filter data by keyword), stream processing (send messages to another process for asynchronous handling), and batch processing (periodically crunch large amounts of accumulated data).

The boundaries between these categories are blurring. Redis is used as a message queue, Apache Kafka provides database-like durability guarantees for messages. A single tool no longer satisfies all data processing needs. Instead, work is broken down across multiple tools, stitched together with application code. When you combine several tools to provide a service, the application code's job is to keep caches, indexes, and derived data consistent with the primary data store.

Typical Data System Architecture

Application code stitches together multiple specialized data systems. Keeping them consistent is the developer's responsibility.

2

Reliability

Reliability means the system continues to work correctly even when things go wrong. A fault is a component deviating from its spec; a failure is the system as a whole stopping. We design fault-tolerant systems -- systems that cope with faults to prevent failures.

Hardware faults: Hard disk MTTF is about 10-50 years. In a cluster of 10,000 disks, you should expect on average one disk failure per day. Historically, we added hardware redundancy -- RAID, dual power supplies, hot-swappable CPUs. As data volumes and computing demands grow (especially cloud), multi-machine redundancy using software fault-tolerance becomes essential.

Software faults: Systematic bugs are harder to anticipate. A software bug in one node can cause correlated failures across many nodes simultaneously (e.g., the Linux kernel leap-second bug of June 30, 2012). Cascading failures where one component's failure triggers others. No quick cure -- thorough testing, process isolation, crash-and-restart, monitoring, and alerting help.

Human errors: Humans are the #1 cause of outages. Configuration errors by operators are the leading cause. Approaches: design systems that minimize opportunities for error (good APIs, admin interfaces), provide sandbox environments, test thoroughly at all levels (unit to integration), allow quick rollback, set up detailed monitoring (telemetry), and implement good management practices and training.

3

Scalability — Describing Load

Scalability is the system's ability to cope with increased load. First, we need to describe load succinctly with load parameters: requests per second, the ratio of reads to writes, number of simultaneously active users, hit rate on a cache, etc. The best choice of parameters depends on your system's architecture.

The Twitter fan-out problem is a classic example. Two main operations: posting a tweet (avg 4.6k requests/sec, peak 12k/sec) and viewing the home timeline (300k requests/sec). Two approaches exist:

Fan-out-on-read: When a user requests their home timeline, look up everyone they follow, find their recent tweets, and merge them (sorted by time). Simple but slow at read time for users following many people.

Fan-out-on-write: When a user posts a tweet, pre-compute and insert it into the home timeline cache of every follower. Reads are cheap (just look up the cache), but writes are expensive -- especially for users with millions of followers (celebrities).

Twitter's actual solution: a hybrid approach. Fan-out-on-write for most users. For celebrities with millions of followers, their tweets are fetched at read time and merged with the precomputed timeline. The distribution of followers per user is a key load parameter.

Twitter Fan-Out: Two Approaches

Fan-out on read: cheap writes, expensive reads. Fan-out on write: expensive writes, cheap reads. Twitter uses a hybrid.

4

Scalability — Describing Performance

Once you've described load, you can investigate what happens when load increases: How is performance affected if you keep resources unchanged? How much do you need to increase resources to keep performance unchanged?

In batch processing systems, we care about throughput (records/second or total time for a job). In online systems, we care about response time -- the time between a client sending a request and receiving a response. Response time varies, so think of it as a distribution, not a single number.

Percentiles: The median (p50) is the halfway point -- half of requests are faster, half are slower. To understand outliers, look at p95 (95th percentile), p99, and p999. Amazon observed that customers with the slowest requests often had the most data (i.e., biggest buyers). Tail latencies directly affect revenue.

Tail latency amplification: In microservice architectures, a single end-user request may fan out to multiple backend services. Even if each service is fast at p99, the overall request is only as fast as the slowest backend call. If you make 5 parallel calls each with p99 = 1s, there's a ~5% chance one of them is slow, making the whole request slow.

SLOs and SLAs: Service Level Objectives and Service Level Agreements are often defined in percentiles: "p99 response time under 200ms." Head-of-line blocking: A few slow requests can hold up processing of subsequent requests, making the queue grow and tail latencies worse.

5

Approaches for Coping with Load

Scaling up (vertical scaling): Move to a more powerful machine -- more CPUs, more RAM, more disk. Simple but expensive and has limits.

Scaling out (horizontal scaling / shared-nothing architecture): Distribute load across multiple smaller machines. More complex but potentially more cost-effective and fault-tolerant.

Elastic scaling: Automatically add computing resources when load is detected. Useful for unpredictable load but adds complexity. Manual scaling is simpler and more predictable.

There is no one-size-fits-all scalable architecture. A system designed for 100,000 requests/sec (each 1 KB) looks very different from one designed for 3 requests/min (each 2 GB). The right architecture depends on read volume, write volume, data to store, complexity of data, response time requirements, and access patterns. An architecture that works at one scale may not work at 10x that scale.

Vertical vs Horizontal Scaling

Vertical: one big machine. Horizontal: many small machines (shared-nothing). Most real systems use a pragmatic mix.

6

Maintainability

The majority of the cost of software is not in its initial development, but in its ongoing maintenance -- fixing bugs, keeping systems running, adapting to new requirements, adding new features, paying off technical debt. We should design software to minimize pain during maintenance. Three design principles help:

Operability: Make it easy for operations teams to keep the system running smoothly. Good monitoring, automation support, documentation, self-healing where possible, predictable behavior, good defaults with overrides. "Good operations can often work around limitations of bad software, but good software cannot run reliably with bad operations."

Simplicity: Make it easy for new engineers to understand the system. Remove accidental complexity -- complexity that is not inherent in the problem but arises from the implementation. The best tool for removing accidental complexity is abstraction. A good abstraction hides implementation detail behind a clean, simple facade (e.g., high-level programming languages hide machine code, SQL hides on-disk storage).

Evolvability (also called extensibility or plasticity): Make it easy to make changes. Requirements change constantly. Agile working patterns provide a framework for adapting to change, but we also need agility at the level of the data system design itself -- making it easy to modify data systems and adapt them to changing requirements.

Key insight: Reliability, scalability, and maintainability are not features to be bolted on -- they are fundamental design concerns that must be addressed from the beginning. There is no magic formula; it's all about making the right trade-offs for your specific use case.

Chapter 1 Summary

  • Data-intensive apps combine databases, caches, search indexes, stream processing, and batch processing -- boundaries are blurring.
  • Reliability: The system works correctly despite faults. Hardware faults (add redundancy), software faults (testing, monitoring), human errors (#1 cause -- design for safety).
  • Scalability starts by describing load with parameters (requests/sec, read/write ratios, etc.). Twitter's fan-out problem illustrates how load parameters drive architecture.
  • Performance is described with percentiles (p50, p95, p99, p999). Tail latencies are amplified in microservice architectures.
  • Scaling up (bigger machine) vs scaling out (more machines). Elastic scaling adds resources automatically. No universal scalable architecture exists.
  • Maintainability has three pillars: Operability (easy to operate), Simplicity (abstraction removes accidental complexity), Evolvability (easy to change).

Key Terms

Fault vs Failure
A fault is one component deviating from spec; a failure is the whole system stopping.
Load Parameter
A number describing the current load (e.g., requests/sec, cache hit rate).
Fan-out
The number of downstream requests triggered by one incoming request.
Percentile (p99)
99% of requests are faster than this value; 1% are slower.
Tail Latency Amplification
In parallel calls, the slowest call determines overall latency.
Accidental Complexity
Complexity not inherent in the problem, arising from the implementation.
