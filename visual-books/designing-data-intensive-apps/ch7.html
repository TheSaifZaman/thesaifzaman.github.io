Transactions

~25 min read Distributed Data

1

The Meaning of ACID

Transactions are a mechanism for grouping several reads and writes into a logical unit. The safety guarantees provided by transactions are often described by the acronym ACID: Atomicity, Consistency, Isolation, and Durability. However, in practice, the meaning of these terms is surprisingly vague and implementations vary widely.

Atomicity means a transaction is all-or-nothing: if any part of the transaction fails (crash, network issue, constraint violation), the entire transaction is aborted and all writes are rolled back as if they never happened. It does not mean "happening at the same instant" -- it means "abortability."

Consistency means that certain invariants about your data must always be satisfied (e.g., credits and debits must balance). This is actually the application's responsibility, not the database's -- the database can't know your business rules. The "C" in ACID is somewhat of a stretch.

Isolation means concurrently running transactions don't step on each other. Ideally, the result is the same as if transactions ran serially, one at a time. In practice, serializable isolation has a performance cost, so most databases use weaker isolation levels.

Durability means once a transaction has committed, its data won't be lost even if there's a hardware fault or crash. Implemented via write-ahead logs (WAL) and/or replication to multiple nodes.

BASE (Basically Available, Soft state, Eventually consistent) is sometimes contrasted with ACID for systems that prioritize availability over strong consistency -- but the term is vague and mostly a marketing distinction.

ACID: Four Pillars of Reliable Transactions

ACID properties form the foundation of reliable transactions. Consistency is the odd one out -- it's the application's responsibility, not the database's.

2

Weak Isolation Levels

Serializable isolation (transactions behaving as if they ran one at a time) has a performance cost, so most databases use weaker isolation levels that protect against some but not all concurrency bugs.

Read Committed is the default in many databases (including PostgreSQL). It provides two guarantees: no dirty reads -- you only see data that has been committed (not mid-transaction values), and no dirty writes -- you only overwrite data that has been committed. Implementation: row-level locks for writes (held until commit/abort); for reads, the database remembers the old committed value and returns it to other transactions until the new value is committed.

Snapshot Isolation (also called Repeatable Read in some databases) provides a stronger guarantee: each transaction reads from a consistent snapshot of the database -- the data as it existed at the start of the transaction. Other transactions' writes are invisible, even if they commit during the transaction. This eliminates "non-repeatable reads" (reading the same row twice and getting different results).

Snapshot isolation is implemented using Multi-Version Concurrency Control (MVCC). The database keeps multiple versions of each object. Each version is tagged with the transaction ID that created it (created_by) and the transaction ID that deleted it (deleted_by). Visibility rules: a version is visible to transaction T if it was created by a committed transaction with an ID less than T's, and it has not been deleted (or was deleted by a transaction that hadn't committed when T started).

MVCC: Multi-Version Concurrency Control

MVCC keeps multiple versions of each row. Each transaction sees the version that was current at its snapshot time, enabling consistent reads without blocking writers.

3

Preventing Lost Updates

The lost update problem occurs during a read-modify-write cycle: two transactions read the same value, both modify it, and one overwrites the other's change. Example: two users increment a counter from 42 -- both read 42, both write 43, but the correct answer should be 44.

Atomic operations: The simplest solution. The database provides atomic update operations like UPDATE counters SET value = value + 1. The operation is implemented with an exclusive lock on the object, or simply performed on a single thread. Works well for simple operations (counters, sets) but not for arbitrary read-modify-write logic.

Explicit locking: The application explicitly locks the rows it intends to update using SELECT ... FOR UPDATE. This forces other transactions to wait until the lock is released. Gives the application full control but is error-prone -- forget a lock and you have a bug.

Compare-and-set: Only perform the update if the value hasn't changed since you read it: UPDATE ... SET ... WHERE value = old_value. If another transaction changed it, your update affects zero rows and you retry. Safe if the database checks the current value (not a stale snapshot).

Write skew is a more general form of the lost update problem. Two transactions read the same data, make decisions based on what they read, then write to different objects. Example: two doctors both check that at least two doctors are on call, then both request leave -- now zero doctors are on call. The constraint spans multiple objects, so single-row locks don't help. Solutions: materializing conflicts (introducing a lock object), or using serializable isolation.

4

Serializability

Serializable isolation is the strongest isolation level: the database guarantees that the result of concurrent transactions is the same as if they had executed one at a time (serially). It eliminates all concurrency anomalies. There are three main approaches to implementing it:

(a) Actual Serial Execution: Literally execute transactions one at a time, on a single thread. This became feasible with in-memory datasets (no disk I/O waits) and stored procedures (entire transaction logic submitted at once, no interactive back-and-forth). Used by VoltDB, Redis, and Datomic. Throughput is limited to a single CPU core, so transactions must be very fast. Partitioning can allow parallel execution, but cross-partition transactions are slower.

(b) Two-Phase Locking (2PL): The traditional approach. Shared locks for reads (multiple readers allowed), exclusive locks for writes (blocks all other reads and writes). Locks are held until the transaction commits or aborts (that's the "two-phase" part: expanding phase acquires locks, shrinking phase releases them). Predicate locks protect against phantoms (new rows matching a query condition), but are expensive. Index-range locks are a practical approximation: lock a range of the index rather than individual predicates.

(c) Serializable Snapshot Isolation (SSI): An optimistic approach built on snapshot isolation. Transactions proceed without blocking; at commit time, the database checks whether any conflicts occurred. If a conflict is detected (the transaction's reads are no longer valid due to concurrent writes), the transaction is aborted and must retry. SSI detects two cases: reads of stale MVCC versions, and writes that affect prior reads. Performs well when contention is low; degrades when many transactions must abort.

Three Approaches to Serializability

Serial execution is simplest, 2PL is the traditional approach with locking overhead, and SSI is the modern optimistic approach that detects conflicts at commit time.

5

Summary: Choosing the Right Level

The right isolation level depends on your application's requirements and the trade-offs you're willing to accept:

Read Committed is a good default for most applications. It prevents dirty reads and dirty writes, which are the most egregious concurrency bugs. It's the default in PostgreSQL and many other databases. However, it doesn't prevent non-repeatable reads, phantom reads, or lost updates.

Snapshot Isolation (Repeatable Read) prevents most concurrency anomalies. It's widely available and performs well. It prevents non-repeatable reads and (in some implementations) lost updates. However, it doesn't prevent write skew -- where a transaction makes a decision based on a stale read of data that was concurrently modified.

Serializable isolation is the safest: it prevents all concurrency anomalies. But it comes with performance trade-offs -- either reduced throughput (serial execution, 2PL) or increased abort rates (SSI). Use it when correctness is critical and the workload allows it.

Key insight: Beware that many databases that claim "serializable" isolation actually implement snapshot isolation (e.g., Oracle's "serializable" is actually snapshot isolation). Read the fine print! If your application relies on serializability, verify that your database actually provides it.

Chapter 7 Summary

  • ACID stands for Atomicity (all-or-nothing), Consistency (application invariants), Isolation (concurrent txns don't interfere), Durability (committed data survives crashes).
  • Read Committed: Default in many databases; prevents dirty reads and dirty writes using row-level locks + remembered old values.
  • Snapshot Isolation (MVCC): Each transaction sees a consistent snapshot; multiple row versions tagged with transaction IDs determine visibility.
  • Lost updates occur during read-modify-write cycles. Solutions: atomic operations, explicit locking (SELECT FOR UPDATE), compare-and-set.
  • Write skew: A generalized lost update where a constraint spans multiple objects. Requires serializable isolation or materialized conflicts.
  • Serial execution: Single-threaded with stored procedures. Simple but limited to one CPU core.
  • Two-Phase Locking (2PL): Shared/exclusive locks held until commit. Predicate or index-range locks prevent phantoms.
  • Serializable Snapshot Isolation (SSI): Optimistic approach -- proceed without blocking, detect conflicts at commit, abort if needed.
  • Beware: Many "serializable" implementations are actually snapshot isolation -- verify your database's actual guarantees.

Key Terms

ACID
Atomicity, Consistency, Isolation, Durability -- safety guarantees of transactions.
MVCC
Multi-Version Concurrency Control -- keeping multiple versions of data for snapshot isolation.
Write Skew
Two transactions read overlapping data, make decisions, then write to different objects, violating a constraint.
2PL
Two-Phase Locking -- transactions acquire locks (expanding phase) then release them at commit (shrinking phase).
SSI
Serializable Snapshot Isolation -- optimistic conflict detection at commit time.
Phantom
A new row appears that matches a query's search condition, inserted by a concurrent transaction.
