Chapter 12

The Future of Data Systems

~25 min read Derived Data

1

Data Integration

No single data tool does everything well. A modern application typically needs a relational database for OLTP, a search engine for full-text search, a cache for low-latency reads, a data warehouse for analytics, and perhaps a graph database for connected data. The fundamental challenge is keeping all these systems consistent with each other.

The solution is derived data: designate one system as the system of record (source of truth), and derive all other representations from it. A change to the system of record flows through a log (like Kafka) to update the search index, invalidate the cache, and feed the data warehouse. Total ordering via the log ensures all derived systems process changes in the same order, maintaining consistency.

The Lambda architecture was an early approach: maintain a batch layer (recomputes everything periodically for accuracy) and a stream layer (processes recent events for speed). Queries merge results from both layers. While it works, running and debugging two separate systems is operationally complex. The modern trend is unifying batch and stream: treat batch as a bounded stream. Systems like Flink and Kafka Streams can handle both real-time events and historical reprocessing with the same codebase.

Data Integration with Derived Views

One system of record produces an ordered event log. All derived views (search, cache, analytics) consume the same log, ensuring consistency across the entire data ecosystem.

2

Unbundling Databases

A traditional database bundles many capabilities together: durable storage, secondary indexes, query processing, materialized views, replication, transactions, and access control. This is convenient, but it means you are limited to one system's implementation of each capability.

The unbundled approach separates these concerns: use specialized tools for each (e.g., Kafka for durable log, Elasticsearch for indexing, Redis for caching, PostgreSQL for transactions) and connect them via dataflow. Each tool does one thing well, and the log-based infrastructure ensures they stay consistent.

Creating a new index is like adding a follower: when you create a secondary index in a database, the database scans existing data and builds the index, then keeps it updated as new writes arrive. In the unbundled world, setting up a new derived system (like a search index) works the same way: consume the log from the beginning to bootstrap, then keep consuming new events to stay up-to-date.

This shifts application design from request/response (synchronous queries) to dataflow (asynchronous event processing). Instead of querying a database and joining results in application code, data is pre-joined and materialized by stream processors, ready for low-latency reads.

3

Aiming for Correctness

The end-to-end argument is crucial: guarantees like exactly-once processing are only meaningful at the application level, not just within a single component. For example, if a user clicks "submit" twice, the system must deduplicate at the application level -- a deduplication ID from the client must be carried through to the final database write. Exactly-once within Kafka alone does not prevent the user from seeing duplicate effects.

Enforcing constraints (like uniqueness) in distributed systems is fundamentally hard. It requires some form of consensus or single-partition coordination. For example, ensuring unique usernames requires either routing all username checks to the same partition or using a distributed consensus protocol. Many systems relax uniqueness to "best effort" and handle violations after the fact.

Timeliness vs integrity: These are two different dimensions of correctness. Timeliness means a user sees the most recent data. Integrity means data is not corrupted or contradictory. Integrity is far more critical -- a slightly stale read is tolerable, but permanently losing data or double-charging a customer is not. Systems should be designed to never compromise integrity, even if timeliness suffers.

Auditing provides a safety net: by retaining the event log, you can always verify that derived data is correct by reprocessing the log. If a bug in a stream processor corrupted some derived data, you can fix the bug and recompute from the immutable source events.

4

Trust, but Verify

Mature engineering practice assumes that things will go wrong -- hardware failures, software bugs, operator errors, data corruption. Rather than blindly trusting that systems are correct, we should build in verification mechanisms.

Byzantine fault tolerance (where nodes actively lie or behave maliciously) within a single organization's systems is generally impractical and unnecessary. However, end-to-end verification is both practical and valuable: checksums to detect data corruption, audit logs to track who changed what and when, and cryptographic proofs (like Merkle trees) to verify data integrity.

The event sourcing and CDC approaches naturally support auditing: the immutable event log is a complete record of what happened. You can always compare derived state against a fresh computation from the source events. This is analogous to double-entry bookkeeping in accounting -- discrepancies are detectable.

As systems grow more complex, end-to-end verification becomes more important. Rather than trying to prove that every component is bug-free, we accept that bugs exist and build systems that detect and recover from their consequences.

5

Doing the Right Thing

Technical systems do not exist in a vacuum -- they affect real people's lives. This final section addresses the ethical responsibilities of data engineers and system designers.

Predictive analytics: Machine learning models trained on historical data can amplify existing biases. A model trained on biased hiring data will perpetuate discrimination. Using algorithmic decisions for credit scoring, insurance pricing, or criminal sentencing can have severe, unjust consequences for individuals -- and the affected person may have no way to understand or challenge the decision.

Surveillance: Data collected for one purpose is often repurposed for another. Location data collected for navigation can be used for surveillance. Purchase history collected for recommendations can be sold to advertisers. The technical capability to collect and process data at scale creates surveillance capabilities that can be abused.

Privacy: Meaningful informed consent is difficult at scale -- nobody reads terms of service. Data, once collected, can be breached, subpoenaed, or sold. The asymmetry of power between data collectors and data subjects is a structural problem that technology alone cannot solve.

Data engineers have a responsibility to consider these implications: to push back against unethical data collection, to design systems that minimize data retention, to build in user controls, and to be transparent about how data is used. Building systems that respect users is not just an ethical imperative -- it is increasingly a legal one (GDPR, CCPA).

Key insight: The future of data systems lies in composing specialized tools via dataflow, achieving correctness through end-to-end verification rather than component-level guarantees, and building systems that respect both technical integrity and human values.

Chapter 12 Summary

  • Data integration: Designate a system of record and derive all other representations (search, cache, warehouse) via an ordered event log.
  • Lambda architecture: Batch layer + stream layer; modern trend unifies both by treating batch as a bounded stream.
  • Unbundling databases: Compose specialized tools connected by dataflow rather than relying on one monolithic database for everything.
  • Dataflow vs request/response: Pre-join and materialize data via stream processors for low-latency reads.
  • End-to-end argument: Exactly-once semantics are only meaningful at the application level (e.g., deduplication IDs from client to database).
  • Timeliness vs integrity: Integrity (no corruption) is more critical than timeliness (seeing the latest value). Never compromise integrity.
  • Auditing: Immutable event logs enable reprocessing to verify and correct derived data.
  • Trust, but verify: Use checksums, audit logs, and cryptographic proofs for end-to-end data verification.
  • Ethics: Predictive analytics can amplify bias; surveillance risks arise from data repurposing; engineers must design systems that respect users.

Key Terms

Derived Data
Data computed from a system of record -- search indexes, caches, materialized views.
Lambda Architecture
Dual-layer approach: batch for accuracy, stream for speed, queries merge both.
Unbundling
Decomposing a monolithic database into specialized tools connected by dataflow.
End-to-End Argument
Correctness guarantees must be enforced at the application level, not just within components.
Integrity
Data is not corrupted or contradictory -- more critical than timeliness.
Auditing
Verifying derived data by reprocessing the immutable event log from the source of truth.
