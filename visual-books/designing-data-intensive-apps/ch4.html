Encoding and Evolution

~22 min read Foundations

1

Formats for Encoding Data

Programs work with data in at least two representations: in memory (objects, structs, arrays, hash tables -- optimized for CPU access) and on disk or network (self-contained sequences of bytes). The translation from in-memory to bytes is called encoding (serialization, marshalling); the reverse is decoding (deserialization, unmarshalling).

Two Representations of Data

In-memory structures use pointers, padding, and language-specific layouts. Encoded data is a flat byte sequence that any language can read.

Language-specific formats (Java's java.io.Serializable, Python's pickle, Ruby's Marshal): Convenient but deeply flawed. They tie you to one language, often have security vulnerabilities (deserializing untrusted data can execute arbitrary code), poor versioning/forward-backward compatibility, and often poor performance.

Problems with Language-Specific Formats

JSON and XML: Widely supported, human-readable, language-independent. But they have problems. JSON doesn't distinguish integers from floating-point numbers (everything is a "number"). JSON doesn't support binary strings (workaround: Base64, which increases size by 33%). XML is verbose. Neither has a schema mechanism that's widely adopted or enforced.

Binary JSON variants (MessagePack, BSON, BJSON, UBJSON, BISON): Encode JSON in binary for compactness and faster parsing. But the space savings are modest -- a 81-byte JSON document might become 66 bytes in MessagePack. The data model is still JSON, so you still have the number ambiguity and no schema.

Encoding Format Spectrum

The trade-off: human readability decreases as compactness and parsing speed improve. Schema-based formats offer the best of both worlds with the schema file providing readability.

2

Thrift, Protocol Buffers, and Avro

These are schema-based binary encoding formats. You define a schema with field names, types, and numeric field tags. The encoded data uses field tags instead of field names, making it more compact.

Apache Thrift (Facebook) has two encodings: BinaryProtocol (field tag + type + length + value) and CompactProtocol (packs field tag and type into a single byte, uses variable-length integers). Protocol Buffers (Protobuf) (Google) is similar to Thrift's CompactProtocol -- very efficient, variable-length integers, packed into minimal bytes.

Both Thrift and Protobuf use field tags (numbers) as the identity of each field. The field name is only for human readability in the schema file -- it never appears in the encoded data. This is critical for schema evolution: you can change a field's name freely, but you must never change its tag number.

Schema to Binary: How Field Tags Work

Apache Avro takes a different approach: no field tags at all. The encoded data is just values concatenated together. To decode it, you need both the writer's schema (the schema used when encoding) and the reader's schema (the schema the reading code expects). Avro resolves the two schemas at read time, matching fields by name. This makes Avro the most compact, and the writer's schema can be included once per file (for Hadoop) or negotiated per connection (for RPC).

Encoding Comparison: Same Record

JSON: 81 bytes (field names as strings). Protobuf: 33 bytes (numeric field tags). Avro: 32 bytes (no tags at all, relies on schema resolution).

3

Schema Evolution

Schemas inevitably change over time. The key question is whether old and new code can coexist -- typically during rolling upgrades where not all servers are updated simultaneously.

Forward vs Backward Compatibility

Forward compatibility: Old code can read data written by new code. When old code encounters an unknown field tag, it simply skips it (the type annotation tells it how many bytes to skip). This works automatically in Thrift and Protobuf.

Backward compatibility: New code can read data written by old code. When new code expects a field that's missing from old data, it uses the default value defined in the schema. This means you can add new fields freely, as long as they have defaults (or are optional). You must never add a new required field -- old data won't have it, breaking backward compatibility.

Safe vs Unsafe Schema Changes

Rules for safe evolution: Never remove a required field. Never reuse a field tag number (even after removing the field). Only add optional fields with default values. In Protobuf/Thrift, removing a field means its tag can never be used again.

Avro's approach is different: the reader's schema and writer's schema don't need to be identical -- they just need to be compatible. Avro resolves differences by matching fields by name. If the writer has a field the reader doesn't expect, it's ignored. If the reader expects a field the writer didn't include, the default value is used. This makes Avro particularly friendly for schema evolution in data pipelines.

4

Modes of Dataflow

Data gets encoded and decoded in three main contexts:

Through databases: The writing process encodes data; the reading process decodes it. The reader might be a different version of the application (or even a different application entirely). Data in a database can outlive the code that wrote it by many years -- you might decode data that was written 5 years ago by long-obsolete code. Forward compatibility is crucial: an older version of the code might read and re-write a record, and must not destroy unknown fields added by newer code.

Data Outlives Code in Databases

Through services (REST and RPC): A client encodes a request, the server decodes it, processes it, encodes a response, and the client decodes the response. REST uses HTTP with JSON (or other formats), is widely adopted, and works well for public APIs. RPC (Remote Procedure Call) tries to make network calls look like local function calls -- but this is fundamentally flawed. Network calls are unpredictable (timeouts, retries, variable latency), not idempotent by default, and involve serialization overhead. Modern RPC frameworks (gRPC with Protobuf, Finagle with Thrift) acknowledge these differences. Service compatibility requires maintaining old and new API versions simultaneously during rolling upgrades.

RPC vs Local Function Call: The Hidden Differences

Through asynchronous messaging: Message brokers (RabbitMQ, Kafka) act as intermediaries. The sender encodes a message, the broker stores it temporarily, and the recipient decodes it. Advantages: decoupling (sender doesn't need to know who receives the message), buffering (broker handles backpressure), and re-delivery on failure. The actor model (Akka, Erlang OTP) is a programming model where actors communicate exclusively by sending asynchronous messages, with each actor processing one message at a time. Distributed actor frameworks (across machines) combine message brokering with the actor model.

Three Modes of Dataflow

Data flows through databases (long-lived), through services (request-response), or through message brokers (asynchronous, decoupled).

5

The Merits of Schemas

Binary encoding formats with schemas (Protobuf, Thrift, Avro) have several significant advantages over text-based formats like JSON:

Compactness: They can be much more compact than JSON since field names are omitted from the encoded data (replaced by field tags or schema-order). Avro in particular is extremely compact.

Schema as documentation: The schema is always up-to-date because it's required for encoding and decoding. Unlike documentation or comments, it can't become stale -- if the schema doesn't match the data, encoding/decoding fails.

Compatibility checking: You can check forward and backward compatibility of schema changes before deploying. The schema registry can reject incompatible changes automatically, preventing data corruption or application crashes.

Code generation: For statically typed languages (Java, C++, Go), schemas enable generating classes or structs with type-safe accessors. This gives you compile-time type checking for the data structures, catching errors early.

Four Pillars of Schema Advantages
Key insight: Schema evolution enables rolling upgrades without downtime. By ensuring forward and backward compatibility in your encoding format, you can deploy new code gradually -- some servers run the new version while others still run the old version. This is essential for large-scale systems where you can't update everything simultaneously.

Chapter 4 Summary

  • Encoding translates in-memory data structures to bytes for storage/network. Language-specific formats (pickle, Java serialization) are dangerous and inflexible.
  • JSON/XML: Human-readable and widely supported but verbose, ambiguous (number types), and lack enforced schemas.
  • Thrift & Protobuf: Schema-based binary encoding using numeric field tags. Compact and efficient. Field tags are critical for evolution.
  • Avro: No field tags -- encodes values in schema order. Reader and writer schemas are resolved at read time. Most compact.
  • Forward compatibility: Old code can read new data (skips unknown fields). Backward compatibility: New code can read old data (uses defaults for missing fields).
  • Rules: Never remove required fields, never reuse field tags, only add optional fields with defaults.
  • Dataflow through databases: Data may outlive code by years. Both forward and backward compatibility needed.
  • Dataflow through services: REST (simple, HTTP-based) and RPC (gRPC, Thrift -- tries to make network look local, but shouldn't hide the differences).
  • Dataflow through messaging: Brokers (Kafka, RabbitMQ) decouple sender and receiver. Actor model (Akka, Erlang) combines messaging with computation.
  • Schemas provide compactness, documentation, compatibility checks, and code generation.

Key Terms

Encoding / Decoding
Translating between in-memory objects and byte sequences (serialization / deserialization).
Field Tag
A numeric identifier for a schema field, used instead of the field name in binary encoding.
Forward Compatibility
Old code can read data written by new code (unknown fields are ignored).
Backward Compatibility
New code can read data written by old code (missing fields use defaults).
Writer/Reader Schema
Avro concept: the schema used at write time vs the schema the reading application expects.
RPC
Remote Procedure Call: making a network request look like a local function call (with caveats).
