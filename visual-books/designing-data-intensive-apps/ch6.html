Partitioning

~22 min read Distributed Data

1

Partitioning of Key-Value Data

The goal of partitioning (also called sharding) is to spread data and query load evenly across multiple nodes. If the partitioning is unfair -- some partitions have much more data or receive more queries than others -- it is called skewed, and the overloaded partition is called a hot spot.

Key range partitioning: Assign a continuous range of keys to each partition, like volumes of an encyclopedia (A-D, E-H, ...). This preserves key ordering, enabling efficient range queries. The downside: if keys are accessed sequentially (e.g., timestamp-prefixed keys), all writes hit the same partition, creating a hot spot. Partition boundaries must be chosen carefully -- often adaptively based on the data distribution.

Hash partitioning: Apply a hash function to each key and assign hash ranges to partitions. A good hash function distributes keys uniformly, eliminating hot spots from sequential keys. But you lose the ability to do efficient range queries -- adjacent keys are scattered across partitions. Some systems (like Cassandra) allow compound primary keys where the first part is hashed (for distribution) and the rest is range-sorted within the partition.

2

Skewed Workloads and Hot Spots

Even with hash partitioning, hot spots can still occur. Imagine a social media platform where a celebrity with millions of followers posts an update: all reads and writes for that single key (the post ID or user ID) will hammer one partition. Hashing distributes different keys evenly, but it cannot help when a single key receives disproportionate traffic.

Today, most data systems cannot automatically compensate for a single highly skewed key. It is the application's responsibility to handle this. A common technique: append a random number (e.g., two-digit suffix, giving 100 variants) to hot keys, spreading writes across 100 partitions. But reads now need to query all 100 keys and merge results. This trade-off only makes sense for the small number of keys known to be hot -- applying it to all keys would add unnecessary overhead.

Future systems may detect and compensate for skew automatically, but for now it remains an application-level concern that requires explicit engineering.

3

Partitioning and Secondary Indexes

Key-value data is straightforward to partition -- just map the key to a partition. But most applications also need secondary indexes (e.g., "find all cars where color = red"), and secondary indexes don't map neatly to partitions. Two approaches exist:

Document-partitioned (local) indexes: Each partition maintains its own secondary index covering only the documents in that partition. Writing is simple (update one partition's index), but reads require scatter/gather -- querying all partitions and merging results. This is expensive and prone to tail latency amplification.

Term-partitioned (global) indexes: The secondary index is itself partitioned -- e.g., colors a-m on partition 1, n-z on partition 2. Reads are efficient (query one partition of the index), but writes may need to update indexes across multiple partitions. These cross-partition writes are often implemented asynchronously.

Rebalancing Partitions

Over time, data grows, query throughput changes, or machines fail. You need to move data between nodes -- this is rebalancing. The key requirement: after rebalancing, load should be shared fairly, the database must continue accepting reads and writes during the process, and no more data than necessary should be moved.

Don't use hash mod N: If you assign keys by hash(key) mod N, then adding or removing a node (changing N) reshuffles most keys. This is catastrophically expensive.

Fixed number of partitions: Create many more partitions than nodes at the start (e.g., 1000 partitions for 10 nodes). When a node is added, it steals a few partitions from existing nodes. The partition count doesn't change -- only the assignment of partitions to nodes. Used by Riak, Elasticsearch, Couchbase, Voldemort.

Dynamic partitioning: Partitions split when they grow too large and merge when they shrink. HBase and RethinkDB use this approach. It adapts well to data volume but starts with a single partition (pre-splitting can help).

Proportional to nodes: Keep a fixed number of partitions per node. When a new node joins, it randomly splits existing partitions and takes half. Used by Cassandra.

Automatic vs manual rebalancing: Fully automatic rebalancing is convenient but can be dangerous -- it can overload the network or cascade failures if combined with automatic failure detection. Many production systems use a hybrid: the system suggests partition moves, but a human confirms them.
5

Request Routing

When a client wants to read or write a key, how does it know which node to contact? This is an instance of a general problem called service discovery. Three common approaches:

1. Contact any node (gossip protocol): The client connects to any node. If that node owns the partition, it handles the request; otherwise, it forwards to the correct node. Nodes use a gossip protocol to propagate partition-to-node mapping. Used by Cassandra and Riak. No external dependency, but adds complexity to every node.

2. Routing tier: All requests go through a partition-aware routing layer (load balancer) that determines the correct node. The routing tier uses a coordination service like ZooKeeper to track partition assignments. Used by HBase, SolrCloud, and Kafka. Clean separation of concerns but adds a network hop.

3. Client-aware: The client itself knows the partition-to-node mapping and connects directly to the correct node. The client subscribes to ZooKeeper to stay updated. Most efficient but requires smarter clients.

Three Request Routing Approaches

Parallel query execution: Massively parallel processing (MPP) databases like Teradata, Vertica, and Presto support complex analytical queries that touch many partitions. The query optimizer breaks the query into execution stages, and each stage runs in parallel across partitions before results are combined.

Chapter 6 Summary

  • Partitioning goal: Spread data and query load evenly across nodes to avoid skew and hot spots.
  • Key range partitioning: Preserves sort order for range queries, but risks hot spots from sequential access patterns.
  • Hash partitioning: Distributes keys evenly, eliminates sequential hot spots, but sacrifices range query efficiency.
  • Hot spots from popular keys cannot be solved by hashing -- application must split hot keys (e.g., random suffix).
  • Local (document) indexes: Each partition has its own index. Fast writes, expensive scatter/gather reads.
  • Global (term) indexes: Index is itself partitioned. Efficient reads, but writes must update multiple partitions.
  • Don't use hash mod N for rebalancing -- adding nodes reshuffles nearly everything.
  • Rebalancing strategies: Fixed partition count, dynamic split/merge, or proportional to nodes.
  • Request routing: Gossip (any node), routing tier (ZooKeeper), or client-aware (smart clients).

Key Terms

Partition (Shard)
A subset of the total dataset, stored on one node.
Hot Spot
A partition receiving disproportionately high load.
Scatter/Gather
Querying all partitions and merging results -- expensive for secondary index reads.
Consistent Hashing
A hash-based approach that minimizes data movement when partitions change.
Rebalancing
Moving data between nodes to maintain even load distribution.
ZooKeeper
Coordination service that tracks partition-to-node assignments for routing.
