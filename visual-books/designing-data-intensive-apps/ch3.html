Storage and Retrieval

~25 min read Foundations

1

Log-Structured Storage

The simplest possible database: two bash functions. db_set key value appends a key-value pair to the end of a file. db_get key greps for the last occurrence of the key. Writes are O(1) (just append), but reads are O(n) (scan the whole file). To speed up reads, we need an index.

Hash indexes (Bitcask): Keep an in-memory hash map where every key maps to a byte offset in the data file pointing to the most recent value. Writes are still fast (append + update hash map). Reads are fast (one disk seek). Limitation: all keys must fit in memory. Works well when you have many writes per key but a manageable number of distinct keys.

To avoid running out of disk space, break the log into segments. When a segment reaches a certain size, close it and start a new one. Then run compaction: throw away duplicate keys in the log, keeping only the most recent update for each key. Compacted segments can be merged together.

SSTables (Sorted String Tables): Like the segment files above, but the key-value pairs are sorted by key. Merging segments is efficient (like merge sort). You don't need to keep all keys in memory -- a sparse in-memory index works because you can scan between two known positions to find intermediate keys.

LSM-Trees (Log-Structured Merge-Trees): Writes go to an in-memory balanced tree (the memtable, typically a red-black tree or AVL tree). When the memtable exceeds a threshold, it's flushed to disk as a new SSTable segment. Reads check the memtable first, then each SSTable on disk from newest to oldest. Background compaction periodically merges and compacts SSTables. Bloom filters are used to avoid pointless disk reads for keys that don't exist.

LSM-Tree Write Path

Writes go to the in-memory memtable (sorted tree). When full, it's flushed as an SSTable. Background compaction merges SSTables into larger sorted files.

2

B-Trees

B-trees are the most widely used indexing structure, standard in almost all relational databases. Like SSTables, B-trees keep key-value pairs sorted by key, enabling efficient lookups and range queries. But the design is fundamentally different.

B-trees break the database into fixed-size pages (or blocks), traditionally 4 KB. Each page can be identified by an address, allowing pages to reference other pages -- like pointers on disk. One page is designated as the root. The root contains several keys and references to child pages. Each child is responsible for a contiguous range of keys. A page that contains individual keys (not references to further children) is called a leaf page.

The number of references to child pages in one page is called the branching factor -- typically several hundred. A four-level B-tree of 4 KB pages with a branching factor of 500 can store up to 256 TB.

Updates: Find the leaf page containing the key, change the value, write the page back to disk. Inserts: Find the page whose range encompasses the new key. If there's space, add it. If not, split the page into two half-full pages and update the parent. This keeps the tree balanced -- a B-tree with n keys always has O(log n) depth.

Crash recovery: B-trees modify pages in place, which is dangerous if a crash occurs mid-write (e.g., during a page split). A write-ahead log (WAL) records every modification before it's applied to the tree pages, enabling recovery after a crash.

Concurrency: Multiple threads accessing the B-tree require careful locking. Latches (lightweight locks) protect individual tree pages during modifications.

B-Tree Structure

B-tree pages: root references internal pages, which reference leaf pages holding actual values. Balanced -- O(log n) depth.

3

Comparing B-Trees and LSM-Trees

Both B-trees and LSM-trees are widely used, and each has trade-offs.

LSM-tree advantages: Writes are sequential (append-only), which is much faster than random writes for both HDDs and SSDs. LSM-trees have lower write amplification (the ratio of bytes written to disk vs bytes written by the application) compared to B-trees, which rewrite entire pages. LSM-trees can be compressed better because there's no page fragmentation -- compaction produces tightly packed files. This means LSM-trees often use less disk space for the same dataset.

B-tree advantages: Each key exists in exactly one place in the index. This makes B-trees attractive for transactional workloads needing strong isolation via locks -- you can lock a specific key range at the tree level. B-trees offer more predictable performance because there's no background compaction competing for disk I/O.

LSM-tree downsides: Compaction can interfere with read/write performance, especially at high percentiles. If the rate of incoming writes consistently outpaces compaction, unmerged segments pile up on disk, slowing reads (which must check more files) and eventually running out of disk space. You need careful monitoring.

4

OLTP vs OLAP

OLTP (Online Transaction Processing): The access pattern of most user-facing applications. Many small transactions, each reading or writing a small number of records based on a key. Examples: inserting a sale, looking up a user profile, updating an inventory count. High volume, low latency, user-interactive.

OLAP (Online Analytical Processing): Business analysts run queries that scan huge numbers of records, aggregating data to compute statistics. Examples: "What was total revenue per store in January?" These queries are few but each one touches millions of rows. Read-heavy, not user-interactive.

Data warehousing: Running OLAP queries directly on the OLTP database would harm its performance. The solution: ETL (Extract-Transform-Load) -- extract data from OLTP databases, transform it into an analysis-friendly schema, and load it into a separate data warehouse optimized for analytics. The warehouse can be indexed and tuned differently from the OLTP system.

Star schema (dimensional modeling): A central fact table records events (e.g., each sale). Each row has foreign keys to dimension tables -- who (customer), what (product), where (store), when (date), how (promotion). Dimension tables are usually much smaller than the fact table. Snowflake schema: Like star, but dimensions are further normalized into sub-dimensions (e.g., brand and category tables referenced by the product dimension).

Star Schema

Star schema: a central fact table (events/transactions) surrounded by dimension tables (who, what, where, when). The most common data warehouse schema.

5

Column-Oriented Storage

Fact tables can grow to trillions of rows with hundreds of columns. A typical OLAP query accesses only 4-5 columns but across millions of rows. In a row-oriented store, the database must read entire rows from disk, loading all 100+ columns even if only 3 are needed -- an enormous waste of I/O.

Column-oriented storage: Instead of storing all values from one row together, store all values from one column together. Each column is stored in a separate file. A query that needs only 3 columns reads only those 3 column files. This dramatically reduces the amount of data read from disk.

Column compression: Columns often have many repeated values (e.g., a "product_id" column in a billion-row fact table might only have 100,000 distinct products). Bitmap encoding: Create a bitmap for each distinct value -- one bit per row, 1 if the row has that value, 0 otherwise. These bitmaps compress extremely well with run-length encoding. Compressed columns also fit more data in CPU cache and enable vectorized processing -- operating on chunks of compressed column data in a single CPU cycle.

Sort order: Even though columns are stored separately, rows must maintain correspondence across column files (the 23rd entry in each column file belongs to the same row). The DBA can choose to sort by certain columns -- like a compound index. Sorting helps compression (sorted columns have long runs of repeated values) and enables fast range queries on the sort column.

Row-Oriented vs Column-Oriented Layout

Row store reads entire rows. Column store reads only the columns the query needs, reducing disk I/O dramatically for analytical queries.

6

Materialized Views

A materialized view is a precomputed cache of query results stored on disk. Unlike a standard (virtual) view which re-executes the query every time, a materialized view is an actual copy of the query results. When underlying data changes, the materialized view needs to be updated (either eagerly on write or lazily/periodically).

Data cubes (OLAP cubes): A special case of materialized view. Imagine a 2D grid with one axis being product and the other being date. Each cell contains an aggregate (e.g., total revenue for that product on that date). You can roll up by summing across a dimension -- sum all products to get revenue per date, or sum all dates to get revenue per product. With more dimensions (store, customer segment, promotion), you get an n-dimensional hypercube.

The trade-off: data cubes make certain aggregation queries very fast (just look up the precomputed cell). But they are inflexible -- you can only query the exact dimensions and aggregates the cube was built for. Ad-hoc queries that don't fit the cube's structure must still go to the raw data. Most data warehouses keep the raw data and use cubes/materialized views as a performance optimization for known query patterns.

Key insight: Storage engines are optimized for different access patterns. OLTP (B-trees, LSM-trees) optimizes for reading/writing individual records by key. OLAP (column storage, compression, materialized views) optimizes for scanning large numbers of rows and aggregating. Understanding these internals helps you choose the right database for your workload.

Chapter 3 Summary

  • Hash indexes (Bitcask): In-memory hash map pointing to byte offsets in append-only log files. Fast writes and reads, but all keys must fit in memory.
  • SSTables & LSM-Trees: Sorted segments on disk. Writes go to in-memory memtable, flush to SSTables. Background compaction merges segments. Used by LevelDB, RocksDB, Cassandra, HBase.
  • Bloom filters efficiently test if a key might exist, avoiding unnecessary disk reads.
  • B-Trees: Fixed-size pages, branching factor ~500, O(log n) depth. In-place updates with WAL for crash recovery. Standard for relational DBs.
  • LSM vs B-Tree: LSM = faster writes, better compression, sequential I/O. B-Tree = predictable latency, one key location (good for locks), no compaction interference.
  • OLTP vs OLAP: OLTP = many small read/write transactions. OLAP = few large analytical scans. Data warehouses use ETL to separate them.
  • Star schema: Central fact table + dimension tables. Snowflake = further normalization of dimensions.
  • Column storage: Store each column separately. Enables reading only needed columns, bitmap compression, vectorized processing.
  • Materialized views / data cubes: Precomputed aggregates. Fast for known queries, inflexible for ad-hoc analysis.

Key Terms

LSM-Tree
Log-Structured Merge-Tree: write to memtable, flush sorted segments, compact in background.
SSTable
Sorted String Table: an immutable file of key-value pairs sorted by key.
Write Amplification
Ratio of bytes written to disk vs bytes written by the application (lower is better).
WAL
Write-Ahead Log: every B-tree modification is logged before applying, for crash recovery.
Star Schema
Warehouse schema: central fact table referencing dimension tables (who, what, where, when).
Bitmap Index
One bit per row per distinct value in a column; compresses well with run-length encoding.
