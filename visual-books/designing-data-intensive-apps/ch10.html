Batch Processing

~25 min read Derived Data

1

Three Types of Systems

Systems that process data can be broadly categorized into three types based on their latency characteristics. Services (online systems) handle requests as they arrive -- a client sends a request and expects a response with low latency. Response time and availability are the primary measures of performance. Web servers, REST APIs, and databases serving user-facing applications are typical examples.

Batch processing systems (offline) take a large amount of input data, run a job to process it, and produce output data. Jobs typically take minutes to hours. Throughput (data processed per second) is the key metric, not latency. There is no user waiting for the result in real time.

Stream processing systems (near-real-time) sit somewhere between online and batch. They consume events shortly after they are produced, with lower latency than batch but without the request/response model of services. Message queues and event processing pipelines are examples.

The Unix philosophy is a powerful precursor to modern batch processing: each program does one thing well, programs can be connected via pipes, and the uniform interface is text streams. Simple log analysis with awk, sort, uniq -c, and head demonstrates this composability -- a chain of simple tools analyzing gigabytes of logs efficiently.

Three Types of Data Systems

Services optimize for latency, batch for throughput, and stream processing falls in between. The Unix philosophy of composable tools inspired modern batch processing frameworks.

2

MapReduce and HDFS

HDFS (Hadoop Distributed File System) is a distributed filesystem based on Google's GFS paper. Data is stored across many machines, with each file split into blocks (typically 128 MB) that are replicated across multiple nodes (usually 3 copies). Any machine can read any file, and HDFS handles replication and fault tolerance transparently. HDFS provides the "shared-nothing" architecture's shared storage layer.

MapReduce is a programming framework for processing large datasets in parallel across a cluster. A MapReduce job proceeds in four phases:

(1) Read the input files, breaking them into records. (2) The map function is called once for each record, extracting key-value pairs. The mapper is stateless -- it processes one record at a time. (3) The framework sorts and groups (shuffles) all key-value pairs by key, so that all values for the same key end up at the same reducer. (4) The reduce function processes each group of values for a given key and produces output.

This is conceptually similar to a distributed version of Unix commands: the mapper is like awk extracting fields, the shuffle is like sort, and the reducer is like uniq -c. Fault tolerance comes from the fact that inputs are immutable and deterministic -- if a mapper or reducer fails, the framework simply re-runs it on the same input chunk. Failed tasks are retried automatically without affecting the rest of the job.

MapReduce: input is split across mappers, shuffled by key, then reduced. Immutable inputs enable fault tolerance -- failed tasks are simply re-run.

3

Join Algorithms in MapReduce

Many batch processing jobs need to join two datasets -- for example, joining a log of user activity with a user profile database. MapReduce supports several join strategies, each suited to different data characteristics.

Reduce-side joins (sort-merge join): Both datasets are mapped to emit records keyed by the join key. The shuffle phase brings all records with the same key to the same reducer, which then performs the actual join. This works for any data sizes but requires shuffling all data across the network.

Map-side joins avoid the expensive shuffle step but require specific input properties. A broadcast hash join works when one input is small enough to fit in memory: each mapper loads the small dataset into a hash table and joins it with its partition of the large dataset. A partitioned hash join works when both inputs are partitioned the same way -- each mapper only needs the corresponding partition of each input.

Handling skew (hot keys): If some keys have far more records than others (e.g., a celebrity's user ID in an activity log), a single reducer gets overloaded. Solutions include: sending records for hot keys to multiple randomly chosen reducers (spreading the load), or doing a map-side join for hot keys specifically.

4

Beyond MapReduce

MapReduce has a significant materialization overhead: each job writes its complete output to HDFS before the next job can read it. A chain of MapReduce jobs means reading and writing to distributed storage at every step, which is slow.

Dataflow engines like Apache Spark, Tez, and Flink address this by treating the entire computation as a directed acyclic graph (DAG) of operators. Intermediate data is piped between operators without writing to HDFS, similar to Unix pipes. Operators can be mappers, reducers, or more flexible functions. This dramatically reduces I/O overhead.

Graphs and iterative processing: Many algorithms (PageRank, shortest paths, machine learning) need to iteratively process a graph. The Pregel/BSP (Bulk Synchronous Parallel) model lets each vertex send messages to adjacent vertices in rounds, enabling distributed graph computation.

High-level APIs have made batch processing more accessible: Hive provides SQL-on-Hadoop, Pig offers a dataflow scripting language, Spark SQL supports SQL and DataFrames, and Cascading provides a Java API for pipelines. These compile down to MapReduce or dataflow engine operations, optimizing the execution plan automatically.

5

Key Principles

Several foundational principles underpin batch processing systems:

Immutable inputs, new outputs: Batch jobs never modify their input -- they read input and produce new output files. This functional approach means you can always re-run a job safely, and you can keep the input around for debugging or reprocessing. It also enables easy rollback: if a job produces bad output, just delete it and re-run.

Fault tolerance via recomputation: Because inputs are immutable and map/reduce functions are deterministic, any failed task can be re-executed on the same input to produce the same output. This is much simpler than trying to recover partially-completed work.

Good throughput on large datasets: Batch processing is optimized for scanning through terabytes or petabytes of data efficiently. It trades latency for throughput -- individual records may take a long time to process end-to-end, but the system processes many records per second.

Blurring boundaries: The line between batch processing and MPP (Massively Parallel Processing) databases like Impala, Drill, and Presto is increasingly blurry. MPP databases now run on commodity hardware clusters (like Hadoop), while batch systems now support SQL queries. The key difference is that batch systems are more flexible (arbitrary code, not just SQL) while MPP databases are more optimized for interactive queries.

Key insight: Batch processing's power comes from its simplicity: immutable inputs + deterministic processing + recomputation for fault tolerance. This "functional" approach enables reliable processing at massive scale, and the same principles carry forward into stream processing.

Chapter 10 Summary

  • Three system types: Services (online, low latency), batch processing (offline, high throughput), and stream processing (near-real-time).
  • Unix philosophy: Small composable tools connected by pipes inspired the design of MapReduce and modern batch frameworks.
  • HDFS: Distributed filesystem with block replication across commodity machines, providing the storage layer for batch processing.
  • MapReduce: Read input, map to key-value pairs, shuffle/sort by key, reduce each group. Fault tolerant via recomputation.
  • Join strategies: Reduce-side (sort-merge), map-side (broadcast hash join for small inputs, partitioned hash join for co-partitioned inputs).
  • Hot key skew: Spread records for popular keys across multiple reducers to avoid bottlenecks.
  • Dataflow engines (Spark, Flink, Tez): Pipe data between operators without materializing to HDFS, improving performance over MapReduce chains.
  • High-level APIs: Hive, Pig, Spark SQL translate queries into optimized batch execution plans.
  • Key principles: Immutable inputs, deterministic functions, fault tolerance via recomputation, and throughput optimization.

Key Terms

MapReduce
Programming model: map (extract key-value pairs) then reduce (aggregate by key) across a cluster.
HDFS
Hadoop Distributed File System -- stores files as replicated blocks across commodity machines.
Shuffle
The phase between map and reduce that sorts and groups records by key across the network.
Broadcast Hash Join
Map-side join where the small dataset is loaded into memory as a hash table on every mapper.
Materialization
Writing intermediate results to storage between processing steps (MapReduce's main overhead).
DAG
Directed Acyclic Graph -- dataflow engines model computation as a DAG of operators.
