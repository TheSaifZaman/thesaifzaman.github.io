Consistency and Consensus

~25 min read Distributed Data

1

Linearizability

Linearizability (also called atomic consistency or strong consistency) is the strongest consistency guarantee: the system behaves as if there is only one copy of the data, and all operations on it are atomic. Once a read returns a value v, all subsequent reads (by any client) must return v or a more recent value -- you never see a value "go back in time."

Use cases: Leader election (all nodes must agree on who the leader is at any given moment). Uniqueness constraints (e.g., two users must not be able to register the same username simultaneously). Cross-channel timing dependencies (e.g., you upload an image, then send a message referencing it -- the message consumer must be able to see the image).

The cost of linearizability: The CAP theorem states that during a network partition, you must choose between consistency (linearizability) and availability. If two datacenters are partitioned, a linearizable system must reject requests on one side (losing availability), while a non-linearizable system can continue serving requests on both sides (but with potentially stale data). Even without partitions, linearizability has a performance cost due to coordination overhead.

Many systems choose not to be linearizable for performance reasons. Even RAM on a modern multi-core CPU is not linearizable (due to CPU caches) unless you use memory barriers.

Linearizable vs Non-Linearizable Timeline

In a linearizable system, once a read returns x=1, no subsequent read can return x=0. In a non-linearizable system, a later read might see an older value.

2

Ordering Guarantees

Causal consistency is a weaker but more practical guarantee than linearizability: if event A caused event B (e.g., A is a question, B is the answer), then everyone must see A before B. Events that are not causally related (concurrent) can be seen in any order. Causal consistency is the strongest model that doesn't sacrifice performance for network delays.

Causal order is a partial order, not a total order. In a linearizable system, all operations form a total order (you can always say which happened first). In a causally consistent system, concurrent events are incomparable -- neither happened "before" the other. Only causally related events must be ordered.

Lamport timestamps provide a total ordering across all events: each node maintains a counter, and each event gets a timestamp of (counter, nodeID). The counter is incremented on each event and updated to the maximum seen value when a message is received. This gives a total order consistent with causality, but it has a limitation: you can't tell from Lamport timestamps alone whether two events are concurrent or causally related (they always give a total order, even for concurrent events).

Total order broadcast is a protocol that delivers messages to all nodes in the exact same order. It is equivalent to consensus: if you can solve total order broadcast, you can solve consensus, and vice versa. Total order broadcast is the foundation for state machine replication (all replicas process the same operations in the same order, staying consistent).

3

Distributed Transactions and 2PC

Two-Phase Commit (2PC) is the most common algorithm for achieving atomic transaction commits across multiple nodes. It uses a coordinator (also called transaction manager) that orchestrates the commit:

Phase 1 (Prepare): The coordinator sends a "prepare" request to all participants. Each participant checks whether it can commit (validates constraints, acquires locks) and responds with "yes" (I promise I can commit) or "no" (I must abort). A "yes" vote is a binding promise -- the participant must be able to commit if asked, even after a crash and restart.

Phase 2 (Commit/Abort): If all participants vote "yes," the coordinator writes a commit decision to its own log (the commit point), then sends "commit" to all participants. If any participant votes "no," the coordinator sends "abort" to all.

The critical vulnerability: If the coordinator crashes after sending "prepare" but before sending the commit/abort decision, participants are stuck in a state of uncertainty -- they voted "yes" and hold locks, but don't know the outcome. They must wait for the coordinator to recover. This makes 2PC a blocking protocol.

In practice: XA transactions implement 2PC across heterogeneous systems (different databases, message brokers). They carry a performance penalty because locks must be held during the entire protocol, and a failed coordinator can block the entire system.

Two-Phase Commit (2PC)

2PC ensures atomicity across nodes, but if the coordinator crashes after "prepare" and before "commit," participants hold locks indefinitely -- a blocking protocol.

4

Fault-Tolerant Consensus

Consensus means getting several nodes to agree on something. It sounds simple but is one of the most important and difficult problems in distributed computing. The well-known algorithms are: Paxos (the classic, notoriously difficult to understand), Raft (designed for understandability), Zab (used by ZooKeeper), and Viewstamped Replication.

A correct consensus algorithm must satisfy four properties: Uniform agreement -- no two nodes decide differently. Integrity -- no node decides twice. Validity -- if a node decides value v, then v was proposed by some node. Termination (liveness) -- every non-crashed node eventually decides. The first three are safety properties (always hold); termination is a liveness property (eventually holds, given a sufficient number of working nodes).

Most consensus algorithms use epoch numbers (called ballot numbers in Paxos, terms in Raft). Within each epoch, there is a unique leader. Before a leader can make a decision, it must collect votes from a quorum (majority) of nodes. If a leader is suspected of being dead, a new election begins with a higher epoch number, and the new leader's epoch supersedes the old one.

Limitations of consensus: It requires a majority of nodes to be functioning (a 3-node cluster tolerates 1 failure, a 5-node cluster tolerates 2). Membership changes (adding/removing nodes) are complex. Frequent leader elections (due to network instability) hurt performance because the system makes no progress during elections. Consensus algorithms are designed for small clusters of nodes (3-7), not thousands.

5

Coordination Services

ZooKeeper, etcd, and Consul are coordination services that provide a small set of powerful primitives, implemented on top of consensus algorithms internally. They look like a key-value store, but their guarantees are much stronger than a regular database.

What they provide: Linearizable atomic operations (e.g., compare-and-set). Total ordering of operations (every operation gets a monotonically increasing transaction ID). Failure detection (ephemeral nodes that disappear if the client's session times out). Change notifications (watches that notify clients when data changes).

Use cases: Leader election -- nodes try to acquire an ephemeral lock; the one that succeeds is the leader. If it dies, its session expires, the lock is released, and another node takes over. Distributed locks -- similar to leader election but for mutual exclusion. Service discovery -- services register their address in the coordination service; clients look up which nodes provide a given service. Membership services -- tracking which nodes are alive and part of the cluster.

Allocating work to nodes: When new nodes join or leave, work (partitions, tasks) must be rebalanced. ZooKeeper can track the current assignment and, via watches, notify nodes when the assignment changes. The actual rebalancing logic runs in the application, but ZooKeeper provides the coordination primitives to do it safely.

Key insight: You don't need to implement consensus yourself. Use ZooKeeper, etcd, or Consul for the coordination-heavy parts of your system (leader election, service discovery, distributed locks), and build your application logic on top. These tools encapsulate the hard problems of distributed coordination so you don't have to.

Chapter 9 Summary

  • Linearizability: Strongest consistency -- behaves as if one copy of data, all operations atomic. Sacrificed for performance by many systems.
  • CAP theorem: During a network partition, choose between consistency (linearizability) and availability.
  • Causal consistency: Weaker than linearizability but preserves cause-and-effect ordering. Best performance trade-off.
  • Lamport timestamps: Total ordering via (counter, nodeID) pairs, but can't distinguish concurrent from causal events.
  • Total order broadcast: All nodes deliver messages in the same order. Equivalent to consensus.
  • 2PC (Two-Phase Commit): Coordinator-based atomic commit across nodes. Blocking if coordinator crashes after prepare.
  • Consensus algorithms (Paxos, Raft, Zab): Achieve agreement with epoch-based leaders and quorum voting. Require majority of nodes.
  • Coordination services (ZooKeeper, etcd, Consul): Provide linearizable ops, failure detection, and change notifications on top of consensus.
  • Used for: Leader election, distributed locks, service discovery, membership, and work allocation.

Key Terms

Linearizability
System appears to have one copy of data; once a value is read, it never goes backward.
CAP Theorem
During a partition, choose consistency or availability -- you can't have both.
Causal Consistency
If A caused B, everyone sees A before B. Concurrent events may be in any order.
Two-Phase Commit
Prepare/commit protocol for atomic distributed transactions. Blocking if coordinator fails.
Consensus
Getting multiple nodes to agree on a value. Solved by Paxos, Raft, Zab algorithms.
Epoch Number
Monotonically increasing number identifying a leader's term in a consensus protocol.
