Stream Processing

~25 min read Derived Data

1

Event Streams and Message Brokers

An event is a small, self-contained, immutable, timestamped record of something that happened -- a user click, a sensor reading, a database write. A producer (publisher/sender) generates events, and a consumer (subscriber/receiver) processes them. The challenge is connecting producers to consumers reliably and efficiently.

Traditional message brokers (RabbitMQ, ActiveMQ, JMS) deliver messages to consumers and typically delete the message after acknowledgment. They are designed for task queues: messages are transient, the broker tracks which messages have been delivered, and consumers process messages in roughly the order received. If consumers are slow, messages queue up in the broker's memory.

Log-based message brokers (Apache Kafka, Amazon Kinesis) take a fundamentally different approach. Messages are appended to a partitioned, append-only log. Each partition is an ordered, immutable sequence of messages. Consumers track their position (offset) in the log and can replay messages from any point. Messages are retained for a configurable period (days or weeks), not deleted after consumption. Kafka achieves parallelism by splitting a topic into partitions -- each partition is consumed by one consumer in a consumer group.

Traditional vs Log-Based Brokers

Traditional brokers delete messages after consumption. Log-based brokers retain messages and let consumers replay from any offset.

2

Databases and Streams

Change Data Capture (CDC) bridges the gap between databases and streams by treating every database write as an event in a stream. When a row is inserted, updated, or deleted, that change is captured and published to a message broker. Downstream systems -- search indexes, caches, data warehouses -- consume the change stream and update their own representations accordingly.

CDC turns the database's write-ahead log into a source of truth for derived systems. The order of changes is preserved, so derived systems stay consistent with the source database. Log compaction makes this practical for long-term use: the broker periodically removes older records for a key, keeping only the most recent value. This means a new consumer can bootstrap its state by reading the compacted log from the beginning without needing a full database dump.

Event sourcing is a related but distinct pattern. Instead of storing the current state and capturing changes after the fact, event sourcing stores immutable events at the application level as the primary record (e.g., "user added item X to cart" rather than "cart now contains X"). Current state is derived by replaying the event log. Unlike CDC, which captures low-level database changes, event sourcing captures high-level domain events that carry business meaning.

Change Data Capture Pipeline

CDC captures every database write as an event. Derived systems (search, cache, analytics) consume the change stream and stay in sync with the source.

3

Processing Streams

Once you have a stream of events, there are many ways to process them:

Complex Event Processing (CEP): Search for patterns across a stream of events -- for example, detecting a sequence of "login failed" events followed by "login success" from a different IP (potential account takeover). CEP engines use a query language to define patterns and trigger alerts when patterns match.

Stream analytics: Compute aggregations over time windows -- e.g., the average response time per minute, or the 99th percentile latency over the last 5 minutes. Unlike CEP (which looks for specific patterns), analytics compute running statistics.

Materialized views: Maintain a derived dataset that is updated incrementally as new events arrive -- for example, a real-time count of items per category. Unlike batch recomputation, the view is always up-to-date.

A critical challenge is time reasoning. The event time (when the event actually occurred) can differ significantly from the processing time (when the system processes it) due to network delays, batching, or reprocessing. Stream processors must handle late-arriving events gracefully.

Window types define how events are grouped for aggregation:

Four Window Types

Tumbling windows partition time into non-overlapping intervals. Hopping windows overlap. Sliding windows center on each event. Session windows group by activity gaps.

4

Stream Joins

Joining data in streams is more complex than in batch processing because data arrives continuously and you cannot "go back" to read old data without maintaining state.

Stream-stream join (window join): Join events from two streams that occur within a time window of each other. For example, matching ad impressions with ad clicks within 1 hour. The processor must maintain state for all events within the window duration for both streams. When a new event arrives on one stream, it checks the other stream's buffered events for matches.

Stream-table join (enrichment): Enrich events in a stream with data from a table. For example, joining user activity events with user profile data to add demographic information. The table can be loaded into the stream processor's memory (if small enough) or queried remotely. If the table changes, CDC can keep the local copy up-to-date.

Table-table join (materialized view): Both inputs are CDC streams from different tables. The stream processor maintains the result of the join and updates it as either input changes. For example, maintaining a denormalized view that joins orders with products -- when a product price changes, all affected orders in the view are updated.

Each join type requires different state management. Stream-stream joins need windowed buffers. Stream-table joins need a snapshot of the table. Table-table joins need the full state of both tables.

5

Fault Tolerance

Unlike batch processing (where you can simply re-run a failed job from the beginning), stream processing runs continuously and must recover without reprocessing all historical data. Several approaches exist:

Microbatching (Spark Streaming): Divide the stream into small time intervals (e.g., 1 second) and process each interval as a mini-batch job. Fault tolerance is inherited from the batch framework: if a microbatch fails, replay the input and re-run it. The trade-off is that latency is bounded by the batch interval size.

Checkpointing (Apache Flink): Periodically take consistent snapshots of all operator state. If a failure occurs, restart from the last checkpoint. Flink uses a variant of the Chandy-Lamport algorithm to take consistent distributed snapshots without stopping processing. This enables true event-at-a-time processing with lower latency than microbatching.

Exactly-once semantics are the holy grail of stream processing. Three approaches: (1) Idempotent writes -- make it safe to process the same event twice by ensuring the output is the same regardless (e.g., writing to a key-value store with a deterministic key). (2) Atomic commit -- use distributed transactions to commit output and consumer offsets together. (3) Transactions -- Kafka's exactly-once support uses transactions to atomically write to multiple partitions and commit consumer offsets.

In all cases, the fundamental enabler is being able to rebuild state from the input log. If a processor fails, it can replay events from its last committed offset and reconstruct its state.

Key insight: The log-based approach unifies batch and stream processing. A batch job is simply a stream processor that starts at the beginning of the log and finishes when it reaches the end. Stream processing is the same operation, but it keeps running and processes new events as they arrive.

Chapter 11 Summary

  • Events: Small, immutable, timestamped records produced by one system and consumed by others.
  • Traditional brokers (RabbitMQ): Delete messages after acknowledgment; no replay.
  • Log-based brokers (Kafka): Append-only partitioned log; consumers track offsets and can replay from any point.
  • CDC: Capture database writes as a stream to keep derived systems (search, cache, warehouse) in sync.
  • Event sourcing: Store immutable application-level events as the source of truth; derive current state by replaying.
  • Window types: Tumbling (fixed, non-overlapping), hopping (fixed, overlapping), sliding (per-event), session (gap-based).
  • Event time vs processing time: Must handle late-arriving events due to delays between when events occur and when they are processed.
  • Stream joins: Stream-stream (windowed), stream-table (enrichment via CDC), table-table (materialized join view).
  • Fault tolerance: Microbatching (Spark), checkpointing (Flink), exactly-once via idempotent writes or atomic commits.

Key Terms

CDC
Change Data Capture -- streaming database changes to derived systems.
Event Sourcing
Storing immutable domain events as the primary data record; deriving state by replay.
Log Compaction
Retaining only the latest value per key in a log, enabling consumer bootstrapping.
Offset
A consumer's position in a log partition; determines which messages to read next.
Tumbling Window
Fixed-size, non-overlapping time intervals for grouping events.
Exactly-Once
Guarantee that each event's effect is applied exactly once, even with failures and retries.
