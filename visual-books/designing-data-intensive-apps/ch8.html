Chapter 8

The Trouble with Distributed Systems

~25 min read Distributed Data

1

Faults and Partial Failures

A single computer is fundamentally different from a distributed system. When a program runs on one machine, it is largely deterministic: the same operation produces the same result. If something goes wrong, the entire machine typically crashes -- a total failure. You don't usually get a "partially working" computer.

In a distributed system, things are fundamentally different. Some parts of the system may be working while other parts are broken. These partial failures are nondeterministic: the system may behave in unpredictable ways because you cannot know for sure what state remote nodes are in.

Cloud vs HPC: High-Performance Computing (HPC) treats the cluster more like a single computer -- if a node fails, the entire job is restarted from a checkpoint. Cloud computing takes the opposite approach: systems are built from commodity hardware with high failure rates. Rather than trying to prevent failures, cloud systems are designed to tolerate partial failures. In a shared-nothing architecture, nodes communicate only via an unreliable network.

Building reliable systems from unreliable components is possible (like TCP providing reliable transport over unreliable IP), but you must understand and handle the failure modes explicitly.

2

Unreliable Networks

When you send a request over the network to another node, many things can go wrong. The request may be lost (dropped by the network). It may be waiting in a queue (delayed at a switch or the recipient's OS buffer). The remote node may have crashed. The remote node may have processed the request but the response was lost. Or the response may be delayed (queued on the way back).

The fundamental problem: you can't tell the difference between a lost request and a slow one. The only tool you have is a timeout. But choosing the right timeout is a dilemma: too short, and you declare nodes dead when they're merely slow (false positives, leading to unnecessary failovers and cascading failures). Too long, and it takes a long time to detect actual failures.

Network congestion is a major source of variable delays: packets queue at network switches when many machines send to the same destination; the receiving OS queues packets if the application is busy; TCP flow control (backpressure) throttles the sender; VM pauses can delay packet processing. There is no theoretically "correct" timeout value -- it must be determined experimentally.

Failure Modes of a Network Request

A network request can fail in many ways, but the sender cannot distinguish between them. The only option is to wait for a timeout.

3

Unreliable Clocks

Each machine in a distributed system has its own clock, and they are never perfectly synchronized. NTP (Network Time Protocol) can synchronize clocks, but only to limited accuracy -- network delays, clock drift, and misconfiguration all introduce errors. Relying on clock synchronization across nodes is dangerous.

Time-of-day clocks (wall clocks) return the current date and time (e.g., System.currentTimeMillis()). They are synchronized with NTP, but they can jump: forward or backward when NTP adjusts, or during leap seconds. This makes them unsuitable for measuring elapsed time.

Monotonic clocks (e.g., System.nanoTime()) always move forward and are guaranteed never to jump backward. They are perfect for measuring durations (how long something took) but their absolute value is meaningless -- you can only compare two readings from the same machine.

Using timestamps to order events across nodes is dangerous because of clock skew. If Node A's clock is slightly ahead of Node B's, events on A will appear to happen "later" than events on B, even if B's event actually happened later in real time. Last-write-wins (LWW) based on timestamps can silently drop writes due to clock skew.

Google's TrueTime API takes a different approach: instead of returning a single timestamp, it returns a confidence interval [earliest, latest]. The actual time is guaranteed to be within that interval. When ordering events, the system waits until the intervals don't overlap, ensuring correct ordering. This requires GPS receivers and atomic clocks in each datacenter.

Clock Skew Between Nodes

Clock skew means timestamps across nodes cannot be trusted for ordering events. B's event happened later in real time but has an earlier timestamp.

4

Process Pauses

Even on a single machine, a thread can be paused for an arbitrarily long time and there is nothing it can do about it. Sources of unexpected pauses include: garbage collection (stop-the-world GC pauses can last seconds in large heaps), virtual machine suspension (VM live migration, snapshots), disk I/O (thrashing when memory is overcommitted, or synchronous disk writes), OS context switches, and SIGSTOP/SIGCONT signals.

The danger: a thread might be paused in the middle of a critical section without knowing it was paused. Consider a lease-based leader: a node obtains a lease (time-limited lock) saying "I am the leader for the next 10 seconds." It checks the lease, sees it's still valid, then proceeds to write. But if a GC pause happened between the check and the write, the lease may have already expired and another node became the leader -- both nodes now believe they are the leader.

Fencing tokens solve this problem. Every time a lock or lease is granted, the lock service issues a monotonically increasing fencing token (a number). The client includes this token in every write. The storage server rejects any write with a token lower than the highest token it has already seen. Even if a paused client sends a delayed write with an old token, the storage server rejects it because it has already seen a higher token from the new leader.

5

Knowledge, Truth, and Lies

In a distributed system, a node cannot trust its own judgment alone. A node might think it is the leader, think its lease is valid, or think another node is dead -- and be wrong about all of these. Truth is defined by quorum: a majority of nodes must agree on a fact (e.g., which node is the leader, whether a node is alive).

Byzantine faults occur when nodes actively lie or send corrupted data -- either due to bugs, hardware errors, or malicious actors. Most datacenter systems do not protect against Byzantine faults because it is expensive and most nodes are trusted. Byzantine fault tolerance is mainly relevant in peer-to-peer networks and blockchain systems.

System models formalize our assumptions about what can go wrong. For timing: synchronous model assumes bounded network delay and bounded clock drift (unrealistic for most systems). Partially synchronous model assumes the system is usually well-behaved but occasionally has unbounded delays (most realistic). Asynchronous model makes no timing assumptions at all (theoretical, very conservative).

For crash behavior: crash-stop assumes a node that crashes never comes back. Crash-recovery assumes nodes may crash and later restart with durable state. Byzantine assumes nodes may behave arbitrarily, including sending malicious messages. Most practical systems assume crash-recovery in a partially synchronous network.

Chapter 8 Summary

  • Partial failures are the defining challenge of distributed systems: some parts work while others are broken, and you can't always tell which.
  • Unreliable networks: Requests can be lost, delayed, or duplicated. You can't distinguish "lost" from "slow." Timeouts are your only tool, but there is no correct timeout value.
  • Unreliable clocks: Time-of-day clocks can jump; monotonic clocks only measure durations. Clock skew makes cross-node timestamps unreliable for ordering events.
  • Google TrueTime returns confidence intervals rather than exact timestamps, enabling correct event ordering at the cost of special hardware.
  • Process pauses (GC, VM suspension, disk I/O) can cause a thread to be paused for arbitrarily long, making time-based assumptions dangerous.
  • Fencing tokens: Monotonically increasing tokens that prevent stale writes from paused clients.
  • Truth by quorum: A node cannot trust its own judgment -- truth is determined by majority agreement.
  • System models: Synchronous (bounded delays), partially synchronous (usually bounded), asynchronous (no timing assumptions).
  • Crash models: Crash-stop, crash-recovery, and Byzantine (nodes may lie).

Key Terms

Partial Failure
When some components of a distributed system work while others are broken.
Clock Skew
The difference in time readings between two imperfectly synchronized clocks.
Fencing Token
A monotonically increasing number issued with each lock grant, used to reject stale writes.
Byzantine Fault
A node that lies, sends corrupted data, or behaves maliciously.
Quorum
Majority agreement among nodes to determine the truth in a distributed system.
NTP
Network Time Protocol -- synchronizes clocks across machines with limited accuracy.
