Data Models and Query Languages

~25 min read Foundations

1

Relational vs Document Model

The relational model was proposed by Edgar Codd in 1970: data is organized into relations (tables), where each relation is an unordered collection of tuples (rows). It dominated the database world for 25-30 years, generalizing well across business data processing, transaction processing, and batch processing.

The NoSQL movement emerged around 2009, driven by: a need for greater scalability (very large datasets or high write throughput), preference for free/open-source software, specialized query operations not well supported by relational models, and desire for more dynamic/expressive data models than rigid relational schemas.

Object-relational mismatch (impedance mismatch): Most application code today is written in object-oriented languages. If data is stored in relational tables, there's an awkward translation layer between application objects and database rows. ORMs like ActiveRecord and Hibernate reduce boilerplate but can't completely hide the mismatch.

Consider representing a LinkedIn profile: in a relational model, you'd have separate tables for positions, education, and contact info with foreign keys. In a document model (JSON), the entire profile is a single self-contained document with nested arrays. The document model has better locality -- all data for one profile is in one place, retrieved in a single query.

Relational vs Document: A LinkedIn Profile

Relational: normalized across tables with foreign keys. Document: self-contained JSON with nested structures. Better locality for one-to-many data.

The Impedance Mismatch Problem

Application objects are trees/graphs. Relational tables are flat. The ORM must translate between these two worlds -- the "impedance mismatch."

2

Relationships in Data Models

One-to-many relationships (e.g., one user has many positions) are handled well by the document model -- just nest them as arrays inside the document.

Many-to-one relationships need IDs referencing shared data. For example, instead of storing "Greater Seattle Area" as a free-text string in every profile, you store a region_id pointing to a normalized regions table. This avoids duplication and makes updates easy. But document databases often lack good join support, pushing join logic into application code.

Many-to-many relationships (e.g., recommendations between users, organizations referenced by multiple people) are where the document model struggles most. You end up needing joins, and without database-level support, the application gets more complex.

Historical context -- the network model (CODASYL): Before the relational model won, the network model allowed records to have multiple parents (many-to-many). But accessing a record required navigating along "access paths" through chains of links. It was like navigating a maze -- queries were complex, inflexible, and any change to the data structure could break queries.

The relational model won because the query optimizer automatically determines the most efficient way to execute a query. You declare what data you want, and the optimizer figures out how to get it -- effectively auto-generating the access paths that CODASYL programmers had to write by hand.

Three Relationship Types

One-to-many: document model excels. Many-to-one: needs references and joins. Many-to-many: document model struggles -- consider relational or graph.

3

Query Languages

SQL is declarative: You specify the pattern of data you want -- what conditions the results must meet, how the data should be transformed (sorted, grouped, aggregated) -- but not how to achieve that goal. The database's query optimizer decides the execution strategy. This is powerful because the database can optimize under the hood without you changing your queries.

Imperative code tells the computer to perform operations in a specific order: "iterate through the list, check each element, add matching ones to the result." This is harder for the database to parallelize or optimize automatically.

Declarative languages also work in other contexts. CSS is declarative: you specify "all <p> elements should be blue" and the browser figures out how to render that. Imagine if you had to imperatively walk the DOM and set each element's color!

MapReduce is a hybrid. The map and reduce functions must be pure (no side effects), which is declarative-like, but the overall chaining of map and reduce steps is imperative. MongoDB originally used MapReduce for analytics queries, but later added the aggregation pipeline as a more declarative alternative.

Declarative vs Imperative Query Styles

Declarative: specify what results you want. Imperative: specify step-by-step how to get them. Declarative is easier to optimize and parallelize.

4

Graph Data Models

When many-to-many relationships are the dominant feature of your data, it becomes natural to model the data as a graph. Graphs consist of vertices (nodes/entities) and edges (relationships/arcs).

Property graphs (used by Neo4j, Titan, InfiniteGraph): Each vertex has a unique ID, a set of outgoing/incoming edges, and a collection of key-value properties. Each edge has a unique ID, head and tail vertices, a label describing the relationship type, and key-value properties. This is very flexible -- any vertex can have an edge to any other vertex, with no schema restrictions.

Cypher is a declarative query language for property graphs (created for Neo4j). You describe patterns like (person)-[:BORN_IN]->(location)-[:WITHIN*0..]->(us:Location {name:'United States'}) to find people born in the US.

Triple-stores store data as three-part statements: (subject, predicate, object). For example: (Jim, likes, bananas). The subject is a vertex, the predicate is an edge label, and the object is either another vertex or a primitive value. SPARQL is the query language for triple-stores, using RDF (Resource Description Framework).

Datalog is an older query language (1980s) that expresses data as predicate(subject, object) facts. Queries are built by combining rules. It influenced later graph query languages.

Property Graph: People, Locations, Relationships

Vertices are entities (people, locations). Edges are labeled relationships (BORN_IN, WITHIN, MARRIED). Flexible and expressive for interconnected data.

Triple-Store: (Subject, Predicate, Object)

Triple-stores express all facts as (subject, predicate, object). The object is either an entity reference or a primitive. This maps naturally to RDF and SPARQL.

5

Schema-on-Read vs Schema-on-Write

Schema-on-write (relational databases): The schema is explicit and enforced at write time. When you insert data, the database checks it conforms to the schema. If you want to change the structure, you issue an ALTER TABLE statement. This is analogous to static typing in programming languages -- errors are caught at compile/write time.

Schema-on-read (document databases): The structure of the data is implicit and only interpreted when the data is read. Nothing prevents you from writing any structure. The application code assumes a certain structure when reading and handles it accordingly. This is analogous to dynamic typing -- flexibility at the cost of runtime errors.

Schema-on-read is advantageous when: items in the collection don't all have the same structure (heterogeneous data), the structure is determined by external systems you can't control, or you want to change the format of your data frequently. Schema-on-write is better when all records are expected to have the same structure and you want the database to enforce consistency.

In practice, ALTER TABLE on large tables can be very slow in some databases (MySQL copies the entire table). Document databases handle "migrations" at the application level -- new code handles both the old and new format.

Schema-on-Write vs Schema-on-Read

Schema-on-write: database enforces structure (safe, rigid). Schema-on-read: application handles structure (flexible, error-prone). Like static vs dynamic typing.

6

Convergence of Models

The relational and document models are converging. Most relational databases now support JSON and XML columns -- you can store semi-structured data inside a relational database (PostgreSQL, MySQL, IBM DB2, SQL Server all support JSON). This allows document-like locality and flexibility within a relational framework.

Document databases are adding relational-like features: RethinkDB supports joins in its query language, MongoDB resolved references with its $lookup aggregation stage (similar to a left join).

Graph queries can be expressed in SQL using recursive common table expressions (CTEs), though the syntax is more verbose than purpose-built graph query languages like Cypher.

The lesson: choose based on access patterns, not hype. If your data is mostly self-contained documents with one-to-many relationships, the document model works well. If you need many-to-many relationships and complex joins, the relational model shines. If interconnected data with variable structure is your core, consider graph databases. And remember: a single application may use multiple data models for different purposes.

Model Convergence: Borrowing Each Other's Strengths

Relational adds JSON support. Document adds joins. Graph fills the many-to-many niche. Choose based on your dominant access patterns.

Key insight: Data models profoundly affect how we think about problems. Each model makes some operations easy and some awkward. The best choice depends on your data's relationship patterns and your application's access patterns -- not on which technology is trendiest.

Chapter 2 Summary

  • Relational model (Codd, 1970): data as tables with rows, dominant for 30+ years. Strengths: joins, many-to-many, query optimizer handles access paths.
  • Document model (NoSQL): self-contained JSON/BSON documents. Strengths: schema flexibility, data locality, natural fit for one-to-many. Weakness: poor joins.
  • Impedance mismatch: Application objects don't map cleanly to relational tables. Documents reduce this for hierarchical data.
  • Many-to-many relationships are hard in the document model. The network model (CODASYL) tried to handle them with manual access paths but lost to the relational model's query optimizer.
  • SQL is declarative (what, not how). MapReduce is a hybrid. MongoDB's aggregation pipeline is declarative.
  • Graph models (property graphs + Cypher, triple-stores + SPARQL) excel when many-to-many relationships dominate.
  • Schema-on-read (document) vs schema-on-write (relational): dynamic vs static typing analogy.
  • Models are converging: RDBMs add JSON; document DBs add joins. Choose based on access patterns.

Key Terms

Impedance Mismatch
The disconnect between object-oriented application code and relational database tables.
Data Locality
Storing related data together so it can be retrieved in a single read.
Query Optimizer
The database component that automatically picks the best execution plan for a query.
Property Graph
A graph model where vertices and edges have typed labels and key-value properties.
Schema-on-Read
Data structure is implicit and only interpreted at read time (like dynamic typing).
Triple-Store
A database storing (subject, predicate, object) triples, queried with SPARQL.
