Chapter 6

Fine-tuning for Classification

8 Lessons 7-Question Quiz ~22 min read Fine-tuning Transfer Learning

Chapter 5 concluded with a pretrained GPT model and the skill to load OpenAI's GPT-2 weights into your own architecture. At that point the model can generate fluent text — but it cannot make decisions. Classification fine-tuning bridges the gap between a general-purpose language model and a task-specific predictor. Rather than teaching the model a new language from scratch, you redirect its already-rich linguistic understanding toward drawing a specific, bounded conclusion from every input it sees. The result is a model that is orders of magnitude more accurate than any approach you could train from random initialisation on a small labelled dataset.

This chapter walks through every step of that transformation: choosing the right output representation, assembling a labelled dataset and data loader, loading pretrained weights, grafting on a lightweight classification head, selecting appropriate loss and evaluation metrics, running a tight fine-tuning loop, and finally deploying the classifier in a real inference pipeline. The running example is spam detection on SMS messages — a deliberately simple binary task that lets you focus on the mechanics without getting lost in domain complexity. Every pattern you learn here applies directly to sentiment analysis, topic classification, toxicity detection, or any other discrete-label NLP problem.

6.1

Classification vs Instruction Fine-tuning

Fine-tuning a pretrained LLM falls into two broad families that differ fundamentally in what the model is asked to produce. Classification fine-tuning requires the model to emit one label from a finite set — "spam" or "ham", "positive" or "negative", "sports" or "politics" or "finance". The output space is discrete and closed; every possible answer is known upfront. This simplicity makes the architecture change minimal: you discard or bypass the vocabulary-size output projection and replace it with a small linear layer that outputs a score for each class. The model learns to map any input to a point in that restricted label space, guided by a cross-entropy loss computed over a handful of classes rather than over tens of thousands of vocabulary tokens.

Instruction fine-tuning (covered in Chapter 7) demands free-text output. The model must generate an open-ended response to a natural language prompt — summarise this article, translate this sentence, answer this question. The output space is the full vocabulary at every decoding step, so the language modelling head is kept intact. The loss is still next-token cross-entropy, but now it is computed over long target sequences rather than a single label. Instruction fine-tuning requires substantially more labelled data and compute than classification fine-tuning, because the model must learn both what to say and how to express it coherently across many tokens. For well-defined decision problems, classification fine-tuning is almost always the right choice — it is faster, cheaper, and more reliable.

The key insight that makes both approaches work is transfer learning: the pretrained model already understands language. It knows that "free prize winner" and "claim your reward now" signal something promotional; it knows that subject and verb must agree; it understands negation and context. Fine-tuning does not teach the model language — it teaches the model which aspects of its existing language knowledge to weigh when making a particular decision. That is why even a tiny labelled dataset of a few thousand examples can yield a high-accuracy classifier when the backbone was pretrained on billions of tokens.

6.2

Spam Detection Dataset

The book uses the UCI SMS Spam Collection as its running example — a publicly available dataset of 5,572 short text messages, each labelled "ham" (legitimate) or "spam". Every message is a single SMS of up to a few hundred characters, making the dataset small enough to iterate on quickly while still being realistic enough to reveal the challenges of applied classification. After loading, the string labels are converted to integers: 0 for ham, 1 for spam. The conversion is a one-liner using pandas boolean indexing, and the resulting column feeds directly into the DataLoader as the target tensor.

The most important characteristic of this dataset is severe class imbalance: roughly 4,825 messages are ham (87%) and only 747 are spam (13%). This ratio reflects the real world — most messages people receive are legitimate — but it creates a dangerous trap for evaluation. A model that predicts "ham" for every single input with no learning whatsoever achieves 87% accuracy. This sounds impressive until you realise the model has completely failed at its actual purpose: catching spam. Class imbalance means raw accuracy is essentially meaningless as a primary metric for this task. Any honest evaluation must track precision, recall, and F1 score for the minority class — and ideally report them separately for each label.

Two standard remedies for class imbalance are oversampling the minority class (duplicating spam examples until the classes are roughly balanced) and undersampling the majority class (randomly removing ham examples). Oversampling is simpler to implement and avoids discarding data; undersampling reduces total training time. The book addresses imbalance by oversampling the spam class to match the ham count before splitting into train/validation/test sets, which ensures the model sees equal exposure to both classes during gradient updates. A weighted loss function is another option: assign a higher per-sample loss weight to the minority class, effectively making the model pay more attention to spam misclassifications.

6.3

Padding and DataLoaders

PyTorch requires that all sequences within a batch share the same length so they can be stacked into a rectangular tensor. SMS messages range from a handful of tokens to well over a hundred, so a naive batch would fail with a shape mismatch. Two strategies handle this: truncation discards tokens beyond a maximum length (a reasonable choice when messages are already short); padding extends shorter sequences to the length of the longest in the batch by appending a designated padding token. GPT-2 uses the <|endoftext|> token (ID 50256) as padding — a token that appears at document boundaries in the training data and is therefore semantically neutral when it shows up at the end of an input.

When padding tokens are present, the self-attention mechanism should ideally ignore them — padding should carry no signal about the message's content. This is achieved with an attention mask: a binary tensor of the same shape as the input, where 1 means "real token, attend normally" and 0 means "padding, do not attend to this position". The mask is passed alongside the input IDs and used inside the attention computation to set padding positions to -infinity before the softmax, which zeroes out their contribution. For classification using only the last real token's representation, well-padded batches also require tracking the position of the last non-padding token per example — a bookkeeping step the book handles inside the forward pass.

The DataLoader handles shuffling and batching; a custom collate_fn is the right hook for padding. By deferring padding to collate time rather than preprocessing, you avoid wasting memory on globally padded sequences — each batch is padded only to the length of its longest member. This matters most when the dataset has high variance in sequence length, as the SMS collection does. The collate function encodes each text on the fly, applies optional truncation, then pads within the batch and returns a pair of tensors ready for the model.

def collate_batch(batch, pad_token_id=50256, max_length=None):
    texts, labels = zip(*batch)
    # Encode each text
    encoded = [tokenizer.encode(t) for t in texts]
    if max_length:
        encoded = [e[:max_length] for e in encoded]
    # Pad to the length of the longest in this batch
    max_len = max(len(e) for e in encoded)
    padded  = [e + [pad_token_id] * (max_len - len(e)) for e in encoded]
    return torch.tensor(padded), torch.tensor(labels)
6.4

Loading Pretrained Weights

The power of the fine-tuning approach comes entirely from starting with weights that already encode a rich model of language. Loading those weights requires instantiating your custom GPT architecture and then filling it with values from a saved checkpoint — either one you saved yourself after running Chapter 5's pretraining loop, or the publicly released GPT-2 Small weights from OpenAI (which Chapter 5 covers how to map). The embedding tables, all twelve transformer blocks, and the final LayerNorm are loaded exactly as-is. Only the language modelling head (the lm_head linear layer projecting from 768 dimensions to 50,257 vocabulary logits) is replaced — it would produce nonsensical gradients if kept, since its outputs correspond to vocabulary tokens rather than class labels.

A critical architectural decision is how many layers to freeze during fine-tuning. Freezing a layer means setting requires_grad = False on all its parameters; the forward pass still uses those values, but no gradient is computed for them and the optimizer leaves them untouched. There are three common strategies: (a) freeze everything and train only the new classification head — fastest but the lowest accuracy ceiling, because all the representations are fixed; (b) freeze the first N transformer blocks and unfreeze the last M plus the classifier — a good default compromise; (c) train all parameters end-to-end — the most powerful but also the slowest and most prone to catastrophic forgetting. For the SMS dataset the book unfreezes the last transformer block and the final LayerNorm in addition to the new head, which gives near-full accuracy at a fraction of the compute cost of option (c).

The intuition behind selective unfreezing is that early transformer layers tend to learn low-level linguistic features — token morphology, part-of-speech patterns, syntactic structure — that are useful for almost any task and should be kept intact. Later layers learn higher-level semantic representations that are more task-specific and benefit most from being steered toward the new objective. By concentrating gradient updates on the upper layers, you get the best of both worlds: stable low-level features from pretraining and task-adapted high-level representations from fine-tuning.

model = GPTModel(GPT2_CONFIG_124M)
model.load_state_dict(torch.load('gpt2_weights.pt', map_location='cpu'))

# Freeze all parameters first
for param in model.parameters():
    param.requires_grad = False

# Unfreeze last transformer block + final LayerNorm
for param in model.trf_blocks[-1].parameters():
    param.requires_grad = True
for param in model.final_norm.parameters():
    param.requires_grad = True
6.5

Adding a Classification Head

Once the pretrained weights are loaded and the appropriate layers are frozen, you attach a classification head — a single nn.Linear(d_model, num_classes) layer. For GPT-2 Small, d_model = 768. For binary classification, num_classes = 2. This tiny module is the only part of the network trained from random initialisation; everything else starts from the pretrained checkpoint. The head takes as input a single 768-dimensional vector extracted from the GPT backbone's final hidden state and outputs two logits, one per class. The argmax of those logits is the predicted label.

The critical decision is which position in the sequence to extract as the classification representation. GPT-2 is a causal (left-to-right) model: at each token position, the hidden state is a function of all tokens to the left, including the current one. This means the hidden state at the last real token position is the only one that has attended to the complete input — it has seen every preceding token and incorporated their information through the multi-head attention mechanism across all twelve layers. Using the first token (the BERT convention with a dedicated [CLS] token) would work poorly for GPT, because causal masking means the first token can only see itself. Mean-pooling all token representations is reasonable but adds noise from padding positions. The standard choice for GPT-style classifiers is therefore the last non-padding token.

The resulting architecture is conceptually very clean: the GPT backbone acts as a deep feature extractor, transforming a variable-length token sequence into a single fixed-size vector that is semantically rich. The classification head is then a logistic regression layer operating on top of those features. This modular view — backbone plus head — is the fundamental pattern underlying not only classification but also question answering, named entity recognition, and other structured prediction tasks built on top of pretrained transformers.

class SpamClassifier(nn.Module):
    def __init__(self, base_model, num_classes=2):
        super().__init__()
        self.gpt        = base_model
        self.classifier = nn.Linear(768, num_classes)

    def forward(self, input_ids):
        hidden    = self.gpt.transformer_output(input_ids)  # [B, T, 768]
        last_tok  = hidden[:, -1, :]                        # [B, 768]
        return self.classifier(last_tok)                      # [B, 2]

6.6

Loss and Accuracy Metrics

Classification fine-tuning uses cross-entropy loss — the same formula as pretraining, but now applied over num_classes outputs instead of 50,257 vocabulary tokens. The model outputs two logits per example; cross-entropy interprets these as unnormalised log-probabilities, applies softmax internally, and computes -log(p_correct_class) for each example in the batch, then averages. Because there are only two classes, this is equivalent to binary cross-entropy. The gradient of this loss strongly penalises confident wrong predictions and gently penalises uncertain correct ones, which is exactly the right training signal for a classifier.

Accuracy — the fraction of predictions that match the true label — is easy to compute and intuitive to report, but dangerously misleading for imbalanced datasets. The F1 score is far more informative: it is the harmonic mean of precision (of all examples predicted as spam, what fraction actually are spam?) and recall (of all actual spam examples, what fraction did the model catch?). The harmonic mean severely punishes any model that collapses one of these two values, which is exactly what happens when a model learns to always predict the majority class. In production spam filters, recall is typically prioritised — missing real spam is more costly than the occasional false alarm — but both must be tracked. The confusion matrix — a 2×2 table of true positives, true negatives, false positives, and false negatives — gives the full picture at a glance.

A well-calibrated classifier should also provide a meaningful confidence score alongside its label prediction. Applying softmax to the raw logits yields class probabilities that sum to 1.0; the probability of the predicted class is a rough confidence estimate. In practice, raw neural network softmax probabilities tend to be overconfident — the model often assigns 98% to its top class even when it is wrong. Techniques like temperature scaling (applying a temperature > 1 to soften the softmax before deployment) or Platt scaling (fitting a logistic regression on top of the raw logits using a calibration set) can correct this. For the purposes of this chapter's spam filter, the raw softmax probability is sufficient, but the book flags calibration as an important consideration for production systems.

def compute_accuracy(model, loader):
    model.eval()
    correct, total = 0, 0
    with torch.no_grad():
        for input_ids, labels in loader:
            logits = model(input_ids)
            preds  = logits.argmax(dim=-1)
            correct += (preds == labels).sum().item()
            total   += labels.size(0)
    return correct / total

6.7

Fine-tuning Training Loop

The fine-tuning training loop looks nearly identical to the pretraining loop from Chapter 5, but several hyperparameters must be adjusted significantly. The most important is the learning rate: pretraining typically uses a warmup schedule that peaks around 1e-4 to 3e-4. Fine-tuning should use a much smaller value — typically 5e-5 or lower — and often without warmup, because the model starts from a high-quality pretrained state rather than random weights. A large learning rate during fine-tuning risks catastrophic forgetting: the gradient updates are large enough to overwrite the rich linguistic representations acquired during pretraining, leaving the model good at spam detection but incoherent at language in general. The low learning rate ensures only small, targeted adjustments are made to the already-useful representations.

The number of training epochs also shrinks dramatically. Pretraining runs for many thousands of steps over enormous corpora to build up language knowledge from scratch. Fine-tuning on a few thousand labelled examples typically converges in one to five epochs. Running for longer risks overfitting: the labelled dataset is too small to justify more gradient steps, and the model will start memorising the specific examples it has seen rather than generalising the discriminative patterns. Monitor validation accuracy after each epoch and stop when it plateaus or declines — early stopping is both the practical and theoretically correct approach here.

The optimizer is still AdamW with weight decay, but only the unfrozen parameters are passed to it. Constructing the optimizer with filter(lambda p: p.requires_grad, model.parameters()) ensures that frozen parameters are not included in the optimizer's state dictionaries, saving memory and avoiding any accidental updates. Gradient clipping remains useful at the same threshold of 1.0, though exploding gradients are less likely here because the model starts from a stable pretrained state rather than random initialisation.

optimizer = torch.optim.AdamW(
    filter(lambda p: p.requires_grad, model.parameters()),
    lr=5e-5, weight_decay=0.1
)

for epoch in range(num_epochs):

6.8

Using the Classifier

Inference with the fine-tuned model is straightforward: encode the input text with the tokeniser, wrap it in a batch dimension, run a forward pass, take the argmax of the output logits. The key operational difference from training is the model.eval() call, which disables dropout (if any) and switches BatchNorm layers to use their running statistics rather than per-batch statistics. Wrapping the forward pass in torch.no_grad() is equally important — it prevents PyTorch from recording the computation graph for backpropagation, which saves significant memory and speeds up inference by roughly 10–20% for a model of this size.

Applying softmax to the logits converts them to proper probabilities, enabling a confidence threshold strategy. Rather than always outputting the argmax label, you can instruct the model to abstain when its maximum probability falls below some threshold — say, 0.8. Messages that fall below the threshold get routed to a manual review queue or are classified conservatively as ham. This is a simple but highly practical pattern for production deployments where the cost of a wrong answer is asymmetric: missing spam is bad, but incorrectly filtering a legitimate message (a false positive) erodes user trust rapidly. Threshold tuning on a held-out calibration set lets you explicitly trade off precision against recall to match your application's tolerance for each type of error.

When deploying the classifier beyond the SMS domain — for example, applying the same model to email subject lines or social media posts — it is worth re-evaluating performance on a representative sample from the new domain before going live. Fine-tuning produces a highly task-specific model, but it is still grounded in the original training distribution. A model trained exclusively on SMS messages may miss spam patterns that are common in emails (HTML links, large embedded images, formal salutations) while flagging short transactional messages that look "spammy" in SMS context but are perfectly normal in email. Domain shift is one of the most common causes of production performance degradation in deployed NLP systems, and it is often invisible until users start complaining.

def classify_text(model, tokenizer, text, device='cpu'):
    model.eval()
    input_ids = torch.tensor([tokenizer.encode(text)]).to(device)
    with torch.no_grad():
        logits = model(input_ids)
    probs      = F.softmax(logits, dim=-1)
    label      = logits.argmax(dim=-1).item()
    confidence = probs[0][label].item()
    return ('spam' if label == 1 else 'ham'), confidence

Chapter 6 — Key Takeaways

  • Classification fine-tuning replaces lm_head with a small Linear(d_model, num_classes) layer — everything else stays from pretraining.
  • Last token's hidden state is used as the classification representation in GPT-style models — causal attention means it has seen the whole input.
  • Loading pretrained weights provides a massive head start — a few hours of fine-tuning on thousands of examples beats weeks of training from scratch.
  • Freezing early layers speeds up training and preserves low-level linguistic features; unfreezing the last block + head is the standard compromise.
  • Low learning rate (5e-5) is critical — too high causes catastrophic forgetting, destroying the pretrained knowledge that makes fine-tuning valuable.
  • F1 score is the right metric for imbalanced datasets like spam detection — raw accuracy is misleading when the majority class dominates.
  • Confidence thresholds at inference allow the classifier to abstain on uncertain cases, enabling precision/recall trade-offs suited to the application's risk tolerance.
