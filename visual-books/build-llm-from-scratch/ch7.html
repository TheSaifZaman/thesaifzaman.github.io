Chapter 7

Fine-tuning to Follow Instructions

9 Lessons 8-Question Quiz ~25 min read Instruction Tuning RLHF

Chapter 6 demonstrated how to convert a pretrained GPT model into a text classifier by swapping its language head for a compact class-score layer. That approach is powerful for bounded decision problems where the output belongs to a known, finite set. But the most impactful real-world use of large language models is something richer and more open-ended: responding helpfully to arbitrary natural-language instructions. "Translate this sentence to Spanish." "Summarise this research paper in three bullet points." "Explain why this Python function raises a KeyError and fix it." A classification head cannot produce these answers — only the full language model, generating one token at a time, can.

This chapter completes the LLM development pipeline introduced throughout the book by walking through instruction fine-tuning from first principles. You will learn how to structure and format an instruction dataset, build a custom collate function that masks prompt tokens so the model trains only on response generation, load a pretrained GPT-2 as the starting point, run the fine-tuning loop with the identical cross-entropy objective used in pretraining (but with masked labels), extract clean responses at inference, and evaluate quality using another LLM as an automated judge. Along the way you will see exactly how models like ChatGPT and Claude learn to be helpful assistants rather than raw text completers — and exactly what you would need to build one yourself.

7.1

Instruction Fine-tuning Introduction

A pretrained language model is an extraordinarily capable pattern-matcher: given a prompt, it can continue the text in a fluent, contextually appropriate way. But text completion is not the same as instruction following. If you type "Summarise the following article:" into a raw pretrained GPT, it is just as likely to generate another article as it is to produce a summary — because its training objective was simply to predict the next token, with no distinction between instructions and the responses they are meant to elicit. Instruction fine-tuning teaches the model to recognise the structural pattern of a task description followed by a response, so that when it sees an instruction at inference time it generates a helpful completion rather than wandering off into unconstrained text continuation.

The training data for instruction fine-tuning consists of carefully curated instruction–response pairs: thousands (or hundreds of thousands) of examples where each example contains a natural-language instruction and the ideal response. The training objective is the same cross-entropy loss used in pretraining — predict the next token correctly — but applied only over the response tokens. The prompt is fed to the model so it has full context, but the loss is masked to zero over the prompt portion. This forces the model to learn what constitutes a good response to each type of instruction without also memorising how to reconstruct the prompt it was already given. The result, after even a modest fine-tuning run, is a model that behaves like an assistant rather than a text completer.

Critically, no architectural changes are needed compared to the pretrained base model. The language modelling head — a linear projection from hidden dimension to full vocabulary size — is kept intact. The model still generates text one token at a time using autoregressive decoding with the same sampling or greedy strategies used for ordinary text generation. What changes is purely the distribution of training data: rather than seeing raw internet text during pretraining, the model now sees instruction–response pairs in a controlled format. Stanford's Alpaca project demonstrated just how powerful this is: fine-tuning LLaMA-7B on only 52,000 GPT-generated instruction pairs produced a capable instruction-following assistant competitive with much larger models at the time. The leverage comes entirely from the rich representations already built during pretraining — instruction tuning redirects them rather than rebuilds them.

Key Insight

The instruction-tuned model learns to be an assistant, not just a text completer. Stanford Alpaca showed that 52,000 instruction pairs fine-tuned on LLaMA-7B were enough to transform a raw base model into a capable assistant — demonstrating that quality and format of training data matters more than sheer quantity once pretraining representations are already strong.

7.2

Instruction Dataset Format

Each sample in an instruction fine-tuning dataset contains three fields. The instruction describes the task in natural language — it is what the user says to the model. The input is an optional context field that provides additional material the instruction refers to: the article to be summarised, the code to be debugged, the sentence to be translated. When no additional context is needed (for example, "What is the capital of France?"), the input field is left empty. The response is the expected output — what a helpful, accurate assistant would produce. Building a high-quality dataset requires either human annotators producing these pairs, or using a capable model like GPT-3.5 to generate responses at scale from a seed set of hand-written instructions (the approach Alpaca used).

Dataset diversity is as important as dataset size. A model trained exclusively on factual question-answering will not learn to write code, summarise documents, or follow creative writing prompts — even if it sees millions of QA pairs. The instruction fine-tuning dataset should cover the range of tasks the deployed model will encounter: factual queries, code generation, code debugging, creative writing, translation, summarisation, mathematical reasoning, and open-ended conversation. Each category requires different response styles and structures, so the model must see representative examples from all of them to develop genuine versatility. The book uses a relatively small illustrative dataset; production models like ChatGPT use millions of diverse, human-curated instruction pairs.

The JSON format below is the simplest and most common representation for instruction datasets. It is easy to load with Python's built-in json module, easy to inspect and validate, and directly compatible with tools like Hugging Face Datasets. Notice that the input field is an empty string when no additional context is required — this signals the collate function that no context block should be inserted into the formatted prompt template.

7.3

Alpaca vs Phi-3 Prompt Styles

Instruction–response pairs cannot be fed to the model as bare text — the model needs a consistent structural pattern that clearly delineates where the instruction ends and where it should begin generating. Different projects have established different prompt templates for this purpose, and each is essentially a mini-language that the model learns during fine-tuning. The Alpaca format, developed at Stanford, wraps each sample in a structured template with clearly labelled sections ("### Instruction:", "### Input:", "### Response:"). These section markers act as landmarks that the model learns to recognise — when it sees "### Response:", it knows it is time to generate the answer. The simplicity of this format makes it easy to implement and widely adopted for smaller, research-oriented fine-tuning runs.

The Phi-3 format (and related chat formats like ChatML) uses special tokens to delimit turns rather than text-based markers. The special tokens <|user|>, <|assistant|>, and <|end|> are embedded directly in the token vocabulary as reserved IDs. This approach is more robust against edge cases where the instruction itself happens to contain the string "### Response:" — a text-based delimiter is ambiguous in that scenario, whereas a true special token ID never appears in natural text. Production models from Microsoft, OpenAI, and Anthropic all use special-token-based formats for exactly this reason. The trade-off is that the tokeniser must support these special tokens, which complicates using an off-the-shelf GPT-2 tokeniser without modification.

The most important operational rule: the template used at inference must be identical to the template used during training. The model learns to associate the structural pattern of the template with the task of generating a response. If you train with the Alpaca format but query with the Phi-3 format at inference, the model sees an unfamiliar structure and the quality degrades dramatically — it may produce responses in the wrong location, fail to stop at the right token, or generate incoherent output. Template consistency is one of the most common sources of bugs when adapting fine-tuned models from notebooks to production.

7.4

Batching with Custom Collate

Instruction–response pairs vary enormously in length. A factual QA sample may be 40 tokens total; a code debugging sample with a long function and a detailed explanation may be 500 tokens. Batching these together naively would either fail (due to shape mismatch) or waste compute (due to massive padding). The custom collate function is the right place to handle this: it processes each sample in the batch, applies the prompt template, encodes the full string, truncates to a configured maximum length, pads within the batch to the length of the longest sample, and returns two tensors — input_ids for the forward pass and labels for the loss.

The critical detail that distinguishes instruction fine-tuning from pretraining is the label masking step. After encoding the full string (prompt + response + end token), the collate function computes the length of the prompt portion alone. It then constructs the labels tensor by filling the prompt positions with -100 — PyTorch's convention for "ignore this position in the loss computation." Cross-entropy's ignore_index=-100 argument ensures these positions contribute zero gradient, so the model never receives a training signal to reconstruct the instruction it was given. It only receives gradients from the response tokens, learning to generate good answers rather than to memorise question phrasing. Padding positions are also set to -100 for the same reason.

Getting the prompt length computation right is essential. If you compute it from the formatted string rather than the raw instruction, you must be careful to include all template markers (the "### Instruction:\n" prefix, any "### Input:\n" block, and the "### Response:\n" suffix) in the count — otherwise you will accidentally train on or accidentally mask the wrong tokens. A single off-by-one in the prompt length shifts the mask by one token and silently corrupts the training signal. The book recommends always encoding the prompt alone, measuring its token count, and using that count to slice the labels array — this is the most direct and least error-prone approach.

def instruction_collate(batch, tokenizer, max_len=256, pad_id=50256):
    inputs, labels = [], []
    for item in batch:
        prompt = format_prompt(item['instruction'], item['input'])
        full   = prompt + item['response'] + '<|endoftext|>'
        enc    = tokenizer.encode(full)[:max_len]
        prompt_len = len(tokenizer.encode(prompt))
        # Mask prompt tokens in labels
        lbl = [-100] * prompt_len + enc[prompt_len:]
        # Pad both to max_len
        pad_len = max_len - len(enc)
        enc = enc + [pad_id] * pad_len
        lbl = lbl + [-100]   * pad_len
        inputs.append(enc)
        labels.append(lbl)
    return torch.tensor(inputs), torch.tensor(labels)

7.5

Loading the Pretrained LLM

Unlike classification fine-tuning, instruction fine-tuning requires no architectural modification. The pretrained GPT-2 model is loaded exactly as it was used for text generation in Chapter 5 — token embeddings, positional embeddings, all transformer blocks, the final LayerNorm, and the full lm_head projecting from hidden dimension to vocab_size. The language head is kept precisely because the task is still text generation: the model must produce a probability distribution over the vocabulary at every decoding step. This makes the weight-loading process identical to Chapter 5's procedure — load the checkpoint, map the parameter names, and verify shapes match.

Model size matters more for instruction fine-tuning than for classification fine-tuning. Classification fine-tuning adds a tiny head on top of fixed or partially-fixed representations, so even GPT-2 Small (117M parameters) can be an effective classifier for many tasks. Instruction fine-tuning, however, requires the base model to have strong enough pretrained representations to generalise from a few thousand instruction examples to arbitrary unseen instructions. Larger models — GPT-2 Medium (345M), GPT-2 Large (774M), or GPT-2 XL (1.5B) — have substantially better emergent reasoning and follow diverse instructions more reliably. The book uses GPT-2 Small for illustration, but notes that GPT-2 Medium is a better practical choice if compute allows. For production-grade assistants, even larger starting points such as LLaMA or Mistral are standard.

If training from scratch on your own hardware is not feasible, the Hugging Face transformers library provides a one-liner: AutoModelForCausalLM.from_pretrained('gpt2-medium'). The book implements weight loading manually to expose every mapping decision — which OpenAI checkpoint key maps to which layer index, how the attention weight matrices are split, how the layer normalisations are matched. Understanding this mapping is valuable beyond the book itself: it is the same skill needed to load weights from any external checkpoint format into a custom architecture, which comes up constantly in LLM research and production engineering.

7.6

The Instruction Fine-tuning Loop

The instruction fine-tuning training loop is structurally identical to the pretraining loop from Chapter 5. The optimizer is AdamW; the loss is cross-entropy; gradient clipping at 1.0 prevents instability; the model is set to train mode at the top of each epoch and eval mode for validation. The critical difference is a single argument in the loss computation: ignore_index=-100. This tells PyTorch's cross-entropy implementation to skip any position in the labels tensor that is set to -100, computing the loss and gradient only over the response tokens. Without this argument, the model would receive gradients to reconstruct the prompt — wasting capacity and potentially teaching the model to mirror instructions back rather than respond to them.

The learning rate for instruction fine-tuning is typically lower than for pretraining — 5e-5 is the standard starting point. Unlike classification fine-tuning, where you might freeze most of the model and only unfreeze the head, instruction fine-tuning generally benefits from fine-tuning all parameters. The model needs to adjust its generation behavior throughout the full stack — not just learn a narrow classification boundary — and the pretrained representations are strong enough that even full-parameter fine-tuning with a modest learning rate does not cause catastrophic forgetting. Training typically runs for two to five epochs over the instruction dataset, which is far shorter than pretraining's many thousands of steps over massive corpora. Early stopping based on validation loss is recommended.

Monitoring the training loss alone is insufficient for instruction fine-tuning. The cross-entropy loss (computed only over response tokens) can decrease smoothly even as the model learns to generate plausible-sounding but factually incorrect or off-topic responses. The most informative signal is periodic qualitative inspection: every N steps, run inference on a fixed set of probe instructions — one factual QA, one code task, one creative writing task — and examine the actual generated text. This qualitative check catches mode collapse (the model starts generating the same generic response to everything), repetition loops, and format failures (the model stops using the template) far earlier than the loss curve does.

optimizer = torch.optim.AdamW(model.parameters(), lr=5e-5)

for epoch in range(num_epochs):
    model.train()
    for input_ids, label_ids in instruction_loader:
        optimizer.zero_grad()
        logits = model(input_ids)          # [B, T, vocab_size]
        loss = F.cross_entropy(
            logits.view(-1, logits.size(-1)),
            label_ids.view(-1),
            ignore_index=-100              # skip masked prompt tokens
        )
        loss.backward()
        torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
        optimizer.step()
    print(f"Epoch {epoch+1} loss: {loss.item():.4f}")
7.7

Extracting Responses

At inference time, the procedure is: format the instruction using the same template as training, encode it to token IDs, run the autoregressive generation loop (identical to the generation function from Chapter 5), decode the output token IDs to a string, and then extract just the response portion from the full generated text. The last step is often overlooked but is essential. The model's output is the full formatted string — template markers and all. Returning the full string to the user exposes the internal template structure and may confuse downstream parsing. A simple string split on "### Response:" followed by stripping whitespace and truncating at the first <|endoftext|> is all that is needed for the Alpaca format.

Generation should stop when the model produces the <|endoftext|> token — this is what the model was trained to emit at the end of each response. If you run the generation loop for a fixed number of steps without checking for this stop token, the model will typically start generating a new instruction–response pair after the first one ends (since it has learned that the end-of-text token is followed by more formatted samples from the training distribution). Proper early stopping at the end-of-text token prevents this. Most generation functions already support a stop_token or eos_token_id argument; if yours does not, add a break condition inside the loop.

Response extraction quality has an outsized impact on perceived model quality. A perfectly fine-tuned model can appear broken if the extraction function returns the wrong portion of the generated text, strips too aggressively (removing trailing punctuation), or fails to handle edge cases (instructions that accidentally contain "### Response:" in the input context). Testing the extraction function independently on a range of generated outputs — including pathological ones — before deploying the inference pipeline is an important engineering step that saves significant debugging time later.

def generate_response(model, tokenizer, instruction, input_text=""):
    prompt = format_prompt(instruction, input_text)
    input_ids = torch.tensor([tokenizer.encode(prompt)])
    with torch.no_grad():
        output_ids = generate(model, input_ids, max_new_tokens=200)
    full_text = tokenizer.decode(output_ids[0].tolist())
    # Extract just the response
    response = full_text.split("### Response:")[-1].strip()
    return response.split("<|endoftext|>")[0].strip()
7.8

Evaluating with Another LLM

Evaluating instruction-tuned models is substantially harder than evaluating classifiers. A classifier's output is a label — you compare it to the ground truth and count correct predictions. An instruction-tuned model's output is open-ended text. The "correct" response to "Explain quantum entanglement for a 10-year-old" does not have a single canonical answer; it has a space of valid responses that vary in vocabulary, length, analogies used, and pedagogical approach. Automated string-matching metrics like BLEU and ROUGE, designed for machine translation and summarisation, measure lexical overlap between the generated text and a reference response. They consistently rank verbose, repetitive responses as high quality simply because they share more words with the reference — and they completely miss the difference between a response that is technically word-for-word identical but factually wrong versus one that uses different words to express the same correct idea.

Human evaluation is the gold standard — have raters score responses on helpfulness, accuracy, and fluency — but it is expensive, slow, and difficult to scale. A practical middle ground that has emerged in recent years is LLM-as-evaluator: use a capable model like GPT-4 or Claude as an automated judge. You provide the judge with the instruction, the model's response, and an evaluation rubric, then ask it to assign a score or compare two responses and pick the better one. LLM judges correlate well with human preference in controlled studies, can process thousands of examples per hour, and can be asked to explain their ratings — giving you actionable feedback about what the model is getting wrong (too verbose, factually incorrect, wrong tone). The main caveat is that LLM judges have their own biases, particularly toward longer, more formal-sounding responses and toward outputs that match the judge model's own style.

A practical evaluation workflow: generate responses on a fixed held-out set of instructions spanning all task categories in the training data, run them through an LLM judge with a structured rubric, and track the average score across epochs and model versions. This lets you compare fine-tuning configurations objectively — different learning rates, different dataset sizes, different numbers of epochs — and catch regressions when you update the training data. Tracking score by task category separately (factual QA, code, creative writing) often reveals that changes that improve one category degrade another, a signal that the training distribution balance needs adjustment.

# Example judge prompt for LLM-as-evaluator
judge_prompt = """Given the instruction: "{instruction}"
The model responded: "{response}"
Rate the quality on a scale of 1-10 and explain why.
Score: [1-10]"""

# Call GPT-4 / Claude API with judge_prompt
# Parse the Score: line from the response
def parse_score(judge_output):
    for line in judge_output.split('\n'):
        if line.startswith('Score:'):
            return int(line.split()[1])
    return None
7.9

Conclusions and Next Steps

This chapter closed the loop on the full LLM development pipeline introduced in Chapter 1 and built step by step throughout the book. Pretraining (Chapters 5) gave the model its general language understanding; classification fine-tuning (Chapter 6) redirected that understanding toward bounded decisions; instruction fine-tuning (this chapter) redirected it toward open-ended assistant behavior. The entire pipeline — from a randomly initialised model to a capable instruction-following assistant — is now implementable by anyone with access to a GPU and a few thousand hours of training compute. Beyond what the book covers, the field has developed several important techniques for further improving aligned language models. RLHF (Reinforcement Learning from Human Feedback) adds a second fine-tuning stage: human raters compare pairs of model responses and indicate which they prefer, a reward model is trained to predict human preference, and then the LLM is further fine-tuned using PPO (Proximal Policy Optimization) to maximize the reward model's score. RLHF is how GPT-3 became InstructGPT and how the earliest versions of ChatGPT were aligned.

Constitutional AI, developed by Anthropic for Claude, replaces expensive human preference labels with AI-generated critiques. The model is given a set of principles (the "constitution") and instructed to critique its own outputs against those principles, then revise them. This dramatically reduces reliance on human labellers and can produce more consistent alignment with a broader set of values than pairwise human preferences alone can capture. Direct Preference Optimization (DPO) is a more recent alternative to RLHF that eliminates the separate reward model entirely, training the LLM directly on preference pairs using a closed-form objective that is simpler to implement and often more stable. These methods are advancing rapidly; the core intuition — teach the model what humans prefer, not just what the next token is — remains constant across all of them.

Parameter-efficient fine-tuning methods like LoRA (Low-Rank Adaptation, covered in Appendix E) make instruction fine-tuning accessible at any scale. Rather than updating all parameters of a 7B or 70B model — which requires multiple high-end GPUs just to store the gradients — LoRA adds small trainable rank-decomposition matrices alongside the frozen original weights. Only these tiny adapter matrices are updated during fine-tuning, reducing the number of trainable parameters by a factor of 100x or more while achieving performance close to full fine-tuning. A single consumer GPU can instruction-fine-tune a 7B model with LoRA. This democratisation of fine-tuning is one of the most significant practical developments in the LLM field since the release of GPT-2.

Key Insight

RLHF, Constitutional AI, and DPO all build on top of instruction fine-tuning — they assume the model already knows how to follow instructions and push it further toward human preferences. Instruction fine-tuning is not the end of alignment training; it is the necessary foundation for every advanced alignment technique that follows.

Chapter 7 — Key Takeaways

  • Instruction fine-tuning teaches models to respond helpfully to natural-language instructions — not just complete arbitrary text. No architectural change is needed from the pretrained base model.
  • The training objective is still cross-entropy, computed over the full vocabulary at each step — but only over the response tokens. Prompt tokens are masked with -100 so F.cross_entropy(ignore_index=-100) skips them entirely.
  • Prompt templates (Alpaca, Phi-3, ChatML) must be identical between training and inference — the model learns to recognise the template boundary as the cue to start generating the response.
  • Custom collate functions handle variable-length instruction–response pairs: encode, truncate, pad within the batch, mask prompt tokens in labels, and return ready-to-train tensor pairs.
  • LLM-as-evaluator (using GPT-4 or Claude as a judge) is a practical alternative to expensive human evaluation — it scales to thousands of examples per hour and correlates well with human preference.
  • Larger base models respond better to instruction fine-tuning — instruction tuning redirects existing capabilities rather than building them from scratch, so stronger pretrained representations lead to stronger instruction following.
  • RLHF and Constitutional AI extend instruction fine-tuning with preference learning — they assume the model already knows how to follow instructions and further align it with human values.
