1.1

What Is a Large Language Model?

A Large Language Model (LLM) is a type of deep neural network trained on enormous quantities of text data to understand and generate human language. At their core, LLMs are sophisticated statistical machines: they learn the probability distribution of text by observing billions of tokens — words, subwords, or characters — and learning which sequences tend to follow which. The "large" in the name refers both to the sheer number of learnable parameters (often hundreds of millions to hundreds of billions) and to the scale of data used during training.

Modern LLMs almost universally rest on the transformer architecture, a design introduced in 2017 that replaced earlier recurrent approaches with a mechanism called self-attention. Rather than reading text one word at a time like a person skimming left-to-right, a transformer considers every token in a sequence simultaneously, computing rich relationships between all pairs. This parallel, context-aware processing is what gives LLMs their remarkable ability to maintain long-range coherence and nuanced meaning.

The training objective sounds deceptively simple: given a sequence of tokens, predict the next one. But running this objective across trillions of tokens forces the model to develop internal representations of grammar, facts, logical reasoning, stylistic patterns, and even rudimentary world knowledge — capabilities that were never explicitly programmed. This phenomenon, where abilities that were not directly trained for suddenly appear at sufficient scale, is known as emergent capability.

Well-known examples of LLMs include GPT-4 (OpenAI), Claude (Anthropic), Gemini (Google DeepMind), and the open-weight LLaMA family (Meta). Each differs in architectural details, training data, and fine-tuning approach, but all share the same foundational recipe: transformer backbone + massive pretraining corpus + scale.

Key Insight: The emergent capabilities of LLMs — reasoning, coding, translation, summarisation — are not hand-coded. They arise naturally from learning to predict the next token at scale. Scale + data = capabilities that surprise even their creators.

1.2

Applications of LLMs

LLMs have rapidly moved from research curiosities to production systems powering a remarkable breadth of real-world applications. Their ability to flexibly encode and decode language makes them suitable for tasks that previously required entirely separate, purpose-built models. Understanding the landscape of applications also helps clarify why the field distinguishes between different architectural flavours — encoder-only models versus decoder-only models — each optimised for a different style of task.

Encoder-based models like BERT (Bidirectional Encoder Representations from Transformers, Google, 2018) read an entire input sequence in both directions before producing a contextual representation. This makes them excellent at understanding and classification tasks: sentiment analysis, named entity recognition, question answering over a context passage, and natural language inference. They do not generate free-form text.

Decoder-based models like GPT (Generative Pre-trained Transformer) read tokens left-to-right in a causal (autoregressive) fashion and are optimised for generation. Every output token depends only on the tokens that precede it. This design excels at open-ended text generation, code completion, machine translation (when fine-tuned), and conversational agents. The book you are studying focuses exclusively on the GPT (decoder-only) lineage.

Many enterprise use-cases combine both flavours — a retrieval system (encoder) paired with a generative model (decoder) — to produce systems that are both factually grounded and fluent writers.

BERT vs GPT: BERT sees the whole sentence (bidirectional) → great for understanding. GPT sees only the past (causal) → great for generation. Both are transformers; they differ in which part of the architecture they use and how they are trained.

1.3

The Three Stages of Building an LLM

Constructing a usable LLM is not a single step — it is a structured pipeline composed of three broad phases, each with its own data requirements, compute budget, and objectives. Understanding this pipeline is essential because the book you are reading takes you through all three stages, implementing each from scratch in Python and PyTorch.

Stage 1 — Data Collection and Preprocessing. Everything begins with raw text scraped from the internet (web crawls like Common Crawl), books, code repositories, scientific papers, and other sources. This text must be cleaned (removing HTML tags, deduplicating, filtering low-quality content), then tokenised — split into a vocabulary of subword units using an algorithm such as Byte-Pair Encoding (BPE). The result is a long sequence of integer token IDs that the model will actually operate on. The vocabulary size and tokenisation scheme have significant downstream effects on model performance.

Stage 2 — Pretraining. The tokenised corpus is used to train the transformer on the next-token prediction objective: given the first k tokens of a sequence, predict token k+1. This process is entirely unsupervised — no human-annotated labels are needed. The model iterates over the corpus, computes a cross-entropy loss between its predictions and the actual next tokens, and updates its parameters via backpropagation. At internet scale, this requires clusters of thousands of GPUs running for weeks or months. The resulting "base model" has absorbed broad world knowledge but has no specific task alignment.

Stage 3 — Fine-tuning. A pretrained base model is a capable but unrefined learner. Fine-tuning adapts the model to specific behaviours using a much smaller, curated, labelled dataset. There are two main fine-tuning flavours explored in this book: classification fine-tuning (adding a classification head and training on labelled examples) and instruction fine-tuning (training the model to follow natural-language instructions using prompt–response pairs). Techniques like RLHF (Reinforcement Learning from Human Feedback) and RLAIF add a further alignment layer, rewarding models that produce helpful and safe outputs.

Cost Perspective: Pretraining GPT-3 (175B parameters) is estimated to have cost $4–12 million in compute. Fine-tuning a 7B model for a specific task can be done on a single GPU in hours. This is why transfer learning — reusing pretrained weights — is transformative.

1.4

The Transformer Architecture

The modern era of LLMs began with the 2017 paper "Attention Is All You Need" by Vaswani et al. (Google Brain), which proposed the Transformer — a neural network architecture that relies entirely on an attention mechanism instead of recurrence or convolution. Prior sequence-to-sequence models based on Recurrent Neural Networks (RNNs) and Long Short-Term Memory (LSTM) networks processed tokens one at a time, creating an information bottleneck and making parallelisation during training nearly impossible. The Transformer solved this fundamentally by computing attention over all positions simultaneously.

The original Transformer is an encoder-decoder architecture designed for tasks like machine translation, where the encoder reads the input sentence and the decoder generates the output. BERT uses only the encoder stack (bidirectional), while GPT uses only the decoder stack (causal/unidirectional). This book builds a GPT-style decoder-only model from scratch.

The self-attention mechanism is the heart of the transformer. For each token, it computes three vectors — a Query (Q), a Key (K), and a Value (V) — by multiplying the token's embedding by three learned weight matrices. Attention scores are computed by taking the dot product of each query with all keys, scaling by the square root of the key dimension to stabilise gradients, applying a softmax to get a probability distribution, and then using that distribution to take a weighted sum of the value vectors. The result is a context-enriched representation for each token that "knows" about every other token in the sequence.

Multi-head attention runs this process in parallel across several independent "heads," each potentially attending to different relationship types. The outputs of all heads are concatenated and projected back to the model's hidden dimension. Stacked on top of this are feed-forward networks, layer normalisation (applied before the attention and FFN sublayers in the pre-norm variant used by GPT-2), residual connections, and a final linear + softmax output layer that maps to vocabulary probabilities.

Why Not RNNs? RNNs process tokens sequentially — to understand token t they must pass information through all tokens 1 through t–1. This creates a vanishing gradient problem over long sequences and prevents parallelisation. Self-attention computes all pairwise relationships at once, with constant path length between any two tokens, regardless of distance.

Both stacks share the embedding, positional encoding, FFN, and normalisation blocks. The key difference is the attention mask — GPT uses causal masking so each token can only attend to past tokens.
# Scaled dot-product attention (simplified)
import torch
import torch.nn.functional as F

def scaled_dot_product_attention(Q, K, V, mask=None):
    d_k = Q.size(-1)
    scores = Q @ K.transpose(-2, -1) / d_k**0.5  # [B, H, T, T]
    if mask is not None:
        scores = scores.masked_fill(mask == 0, float('-inf'))
    weights = F.softmax(scores, dim=-1)
    return weights @ V
1.5

Pretraining vs Fine-tuning

One of the most powerful ideas in modern machine learning is transfer learning: train a model on a large, general-purpose task, then reuse the learned representations for a specific, narrower task. In the LLM world, this manifests as the pretraining-then-fine-tuning paradigm that dominates the field.

Pretraining is the expensive, foundational stage. The model is exposed to a massive, diverse corpus — think the English-language Wikipedia, decades of news articles, books, academic papers, and a broad sample of the public web — and trained to predict the next token, token after token, for trillions of total tokens. No human labels are required; the text itself is both the input and the supervision signal. The model must learn to be a good language model, which forces it to implicitly encode grammar, factual knowledge, stylistic conventions, and even some reasoning patterns.

GPT-2, the model this book implements, was pretrained on WebText — a 40 GB dataset created by scraping all outbound links from Reddit posts that received at least 3 upvotes (a rough quality filter). This dataset, while modest by today's standards, was sufficient to demonstrate remarkably coherent long-form text generation and convinced the field that scale was the key variable.

Fine-tuning takes the pretrained weights as a starting point and continues training on a much smaller, task-specific dataset. Because the model already "knows" language deeply, fine-tuning requires far less data and compute to produce a capable task-specific model. The intuition is that the pretrained representations capture general linguistic structure; fine-tuning merely steers these capabilities toward the target task.

There are two main fine-tuning strategies covered in this book. Classification fine-tuning replaces the language modelling head with a classification head and trains the model to predict a class label (e.g., spam or not spam). Instruction fine-tuning keeps the language modelling head but trains the model to follow natural-language instructions encoded in a prompt–completion format, producing a model that behaves like a helpful assistant.

1.6

GPT Architecture Overview

GPT stands for Generative Pre-trained Transformer. The model family began with GPT-1 (OpenAI, 2018, 117M parameters), and each subsequent version dramatically scaled both the model size and training data, culminating in GPT-4 (architecture details undisclosed). The book implements the GPT-2 architecture — small enough to run on a laptop yet large enough to exhibit impressive generative capabilities and to teach every important architectural concept.

GPT is a decoder-only transformer. It is autoregressive: at inference time, the model receives a context (a sequence of input tokens), produces a probability distribution over the entire vocabulary for the next token position, samples or takes the argmax to pick a token, appends it to the context, and repeats. This loop continues until a stop condition is met (maximum length or an end-of-sequence token). Because each new token is conditioned on all previous tokens — and the model was trained to predict exactly this — the generated sequences are fluent and contextually coherent.

Internally, the forward pass of GPT proceeds as follows: input token IDs are passed through a token embedding table (a learned matrix of shape [vocab_size, d_model]) to convert each integer into a dense vector. A separate positional embedding (also learned in GPT-2, unlike the sinusoidal encoding in the original Transformer paper) is added to inject order information, since self-attention is inherently permutation-invariant. The combined embeddings flow through N identical transformer blocks in sequence. Each block contains a layer normalisation → masked multi-head self-attention → residual connection → layer normalisation → feed-forward network → residual connection structure. After the final block, a linear projection maps from the model's hidden dimension back to vocabulary size, and a softmax converts the logits to token probabilities.

GPT-2 was released in four sizes, each progressively larger. The book implements the smallest, GPT-2 Small (124M), but the code is parameterised so that running the larger configurations requires only changing a config dictionary. Understanding GPT-2 Small in full detail provides nearly complete coverage of what makes GPT-4 tick — the differences are predominantly in scale, not architecture.

Chapter 1 — Key Takeaways

  • LLMs are large neural networks trained on massive text corpora to predict the next token; they emerge from the transformer architecture introduced in Attention Is All You Need (2017).
  • GPT uses a decoder-only (causal) transformer architecture; BERT uses an encoder-only (bidirectional) architecture — each is optimised for a different style of task.
  • Building an LLM follows three stages: data preparation & tokenisation → pretraining on unlabelled text → fine-tuning on labelled task data.
  • Pretraining is expensive (weeks on GPU clusters, terabytes of text) but unlocks broad language understanding. Fine-tuning is affordable and adapts that understanding to specific tasks.
  • Emergent capabilities — reasoning, translation, code generation — are not programmed directly. They arise from learning the next-token prediction objective at scale.
  • This book implements GPT-2 Small (124M parameters) in Python/PyTorch from scratch, covering every layer and every training step.
