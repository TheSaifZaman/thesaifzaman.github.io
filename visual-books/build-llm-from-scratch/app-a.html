Appendix A

Introduction to PyTorch

5 Lessons 5-Question Quiz ~20 min read PyTorch Tensors Autograd

PyTorch is the dominant deep learning framework for research and, increasingly, production. It provides two essential building blocks: tensors (multi-dimensional arrays that can run on GPU) and automatic differentiation (the ability to compute gradients through arbitrary computational graphs). Everything in this book — tokenisation, attention, pretraining, fine-tuning — is implemented in PyTorch. This appendix covers the five foundational concepts you need to read and write PyTorch code confidently: tensor operations, autograd, the training loop pattern, GPU usage, and the Dataset/DataLoader abstraction.

If you have used NumPy before, PyTorch tensors will feel immediately familiar. The key additions are GPU support (move a tensor to the GPU with .to('cuda') and all operations run on the GPU transparently) and automatic differentiation (set requires_grad=True and PyTorch tracks every operation for gradient computation). These two features, combined with a rich neural network API in torch.nn, make PyTorch the tool of choice for building language models from scratch.

A.1

Tensors

A tensor is PyTorch's core data structure: a multi-dimensional array that supports GPU acceleration and automatic differentiation. Scalars are 0-dimensional tensors, vectors are 1-dimensional, matrices are 2-dimensional, and neural network activations (batch × sequence × features) are typically 3-dimensional. PyTorch provides a rich set of tensor creation functions — torch.tensor for explicit values, torch.zeros and torch.ones for constant initialisation, torch.randn for Gaussian random values — and all standard mathematical operations apply element-wise or broadcast over dimensions following NumPy semantics.

The dtype of a tensor matters significantly in LLM work. Model weights are typically float32 (or float16/bfloat16 for mixed precision), while token IDs are int64. Mixing dtypes in an operation raises an error, so explicit casting with .float() or .long() is often necessary. Shape manipulation — view and reshape — are among the most frequently used operations in transformer code: they reorganise the underlying data without copying it (in most cases). The difference between the two is that view requires the tensor to be contiguous in memory, while reshape will make a copy if necessary.

Indexing and slicing follow Python/NumPy conventions. x[0] selects the first element along dimension 0, x[:, 2] selects column 2 across all rows, and x[..., -1] selects the last element along the final dimension (a pattern used constantly in transformer attention code). Matrix multiplication — the workhorse of neural networks — can be expressed as x @ y (Python operator), torch.matmul(x, y), or torch.bmm(x, y) for batched 3D tensors.

import torch
x = torch.tensor([[1.0, 2.0], [3.0, 4.0]])
print(x.shape)   # torch.Size([2, 2])
print(x.dtype)   # torch.float32
y = x.T          # transpose
z = x @ y        # matrix multiply → [[5,11],[11,25]]

A.2

Automatic Differentiation (Autograd)

PyTorch's autograd system is what makes training neural networks practical. When you create a tensor with requires_grad=True, PyTorch begins recording every operation performed on it in a dynamic computation graph. When you call .backward() on the loss tensor, PyTorch traverses this graph in reverse (backpropagation) and computes the gradient of the loss with respect to every leaf tensor that has requires_grad=True. These gradients are stored in the .grad attribute and used by the optimizer to update the weights.

The computation graph is dynamic — it is rebuilt fresh for each forward pass. This means you can use regular Python control flow (if statements, loops) inside your model's forward() method and PyTorch will correctly differentiate through whatever execution path was taken. This is a major advantage over static graph frameworks like early TensorFlow. The trade-off is slightly higher memory overhead, since the graph must be retained until .backward() is called.

A critical detail: PyTorch accumulates gradients rather than replacing them. Each call to .backward() adds to the existing .grad value. In a standard training loop where you want fresh gradients per step, you must call optimizer.zero_grad() (or explicitly set .grad = None) at the start of each iteration. Forgetting this is one of the most common PyTorch bugs: your gradients will silently sum across steps, producing incorrect parameter updates.

x = torch.tensor([2.0], requires_grad=True)
y = x ** 2 + 3 * x + 1   # y = x² + 3x + 1
y.backward()              # compute dy/dx
print(x.grad)             # dy/dx = 2x + 3 = 7.0 at x=2

A.3

Building a Simple Training Loop

All PyTorch neural networks subclass nn.Module. You define two methods: __init__, where you declare the learnable layers (e.g., nn.Linear, nn.Embedding), and forward, where you specify the computation that transforms the input into the output. When you call model(input), PyTorch internally calls model.forward(input) while also handling hooks and autograd bookkeeping — which is why you should always call the model as a callable rather than invoking forward() directly.

The training loop follows an invariant four-step pattern for every batch. Step 1: call optimizer.zero_grad() to clear accumulated gradients from the previous step. Step 2: run the forward pass by calling model(x) to get predictions, then compute the loss. Step 3: call loss.backward() to compute gradients for all parameters with requires_grad=True. Step 4: call optimizer.step() to update each parameter by subtracting lr * grad (or the equivalent Adam update). This four-step cycle repeats for every batch in every epoch.

The optimizer accumulates state beyond just the learning rate. torch.optim.SGD optionally maintains momentum buffers. torch.optim.Adam maintains per-parameter first and second moment estimates that adapt the effective learning rate for each weight. This is why Adam typically converges much faster than vanilla SGD for transformer models — the adaptive per-parameter learning rates compensate for the wide variation in gradient magnitudes across different parts of the network.

import torch.nn as nn
import torch.nn.functional as F

class SimpleNet(nn.Module):
    def __init__(self):
        super().__init__()
        self.fc1 = nn.Linear(2, 4)
        self.fc2 = nn.Linear(4, 1)
    def forward(self, x):
        return self.fc2(torch.relu(self.fc1(x)))

model = SimpleNet()
optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)

for x, y in dataloader:
    optimizer.zero_grad()                    # 1. clear gradients
    loss = F.mse_loss(model(x), y)           # 2. forward + loss
    loss.backward()                           # 3. backprop
    optimizer.step()                          # 4. update weights
The 4-Step Training Loop
1. zero_grad() Clear accumulated gradients from last step
↓
2. model(x) → loss Forward pass + compute loss function
↓
3. loss.backward() Backpropagate — fill .grad for all parameters
↓
4. optimizer.step() Update weights: w -= lr * w.grad
This four-step cycle is identical for every model — from a two-layer MLP to GPT-2. The loss function changes; the loop structure does not.
A.4

GPU Usage

Neural network training is dominated by large matrix multiplications — exactly the kind of massively parallel operation that GPUs are designed for. A modern GPU like the NVIDIA A100 can perform 312 teraflops of FP16 matrix operations per second, compared to roughly 1 teraflop for a CPU. For a model like GPT-2 (117M parameters), a single forward pass on CPU takes ~3 seconds; on GPU, the same pass takes ~15 milliseconds — 200× faster. Training without GPU access would make the experiments in this book infeasible in reasonable time.

In PyTorch, moving a tensor or model to the GPU is a single operation: .to(device) where device is a torch.device object. The canonical pattern is to detect the best available device at the top of your script and move everything there consistently. The critical rule is that all tensors participating in an operation must reside on the same device. Mixing CPU tensors with GPU tensors raises a RuntimeError — the fix is always to check that both x.device and model.device match before the operation.

Apple Silicon Macs (M1/M2/M3/M4) support a third device: 'mps' (Metal Performance Shaders). MPS provides GPU acceleration via the Mac's unified memory architecture and is compatible with most PyTorch operations used in this book. For multi-GPU setups, torch.nn.DataParallel replicates the model on each GPU and splits the batch, while DistributedDataParallel (DDP) is the recommended approach for serious multi-GPU training, as it avoids the GIL bottleneck of DataParallel.

device = torch.device('cuda' if torch.cuda.is_available() else 'cpu')
# On Mac: device = torch.device('mps') if torch.backends.mps.is_available()

model = model.to(device)  # move all parameters to GPU

for x, y in dataloader:
    x, y = x.to(device), y.to(device)  # move batch to GPU
    logits = model(x)                   # GPU computation
    loss = F.cross_entropy(logits, y)
    loss.backward()
    optimizer.step()
Device Management Pattern
Detect device cuda → mps → cpu (fallback chain)
↓
model.to(device) Move all parameters and buffers
↓
x.to(device), y.to(device) Move each batch from CPU RAM to GPU VRAM
↓
model(x) → loss → backward → step Entire training step runs on GPU
The model moves once; batches move every step. Forgetting to move either the model or a batch tensor to the same device is the most common GPU-related error.
A.5

Datasets and DataLoaders

PyTorch's torch.utils.data.Dataset is an abstract base class that provides a uniform interface for data sources. To create a custom dataset, you subclass it and implement exactly two methods: __len__ (returns the total number of samples) and __getitem__(idx) (returns the sample at index idx). The rest of the data pipeline — batching, shuffling, parallel loading — is handled automatically by the DataLoader wrapper. This separation of concerns (Dataset defines what the data is; DataLoader handles how it is delivered to the model) is one of the cleanest design patterns in the PyTorch ecosystem.

For language modelling, the Dataset typically stores a flat array of token IDs (the entire training corpus concatenated after tokenisation) and returns overlapping windows of length seq_len. The input x is tokens [i:i+seq_len] and the target y is tokens [i+1:i+seq_len+1] — a one-position shift. This means the model is trained to predict every next token in the sequence simultaneously, making maximum use of each training example. A sequence of length 1024 provides 1024 training signal tokens in a single forward pass.

The DataLoader's num_workers parameter controls how many CPU worker processes are used to load and preprocess data in parallel while the GPU is busy computing. Setting num_workers=4 (or matching your CPU core count) keeps the GPU fed and avoids the common bottleneck where training is CPU-bound rather than GPU-bound. The shuffle=True argument ensures batches are drawn in a random order each epoch, which is critical for generalisation — training on the same sequence order repeatedly leads to poor learning.

from torch.utils.data import Dataset, DataLoader

class TextDataset(Dataset):
    def __init__(self, token_ids, seq_len):
        self.data = token_ids
        self.seq_len = seq_len
    def __len__(self):
        return len(self.data) - self.seq_len
    def __getitem__(self, idx):
        x = torch.tensor(self.data[idx : idx+self.seq_len])
        y = torch.tensor(self.data[idx+1 : idx+self.seq_len+1])
        return x, y

dataset = TextDataset(token_ids, seq_len=1024)
loader = DataLoader(dataset, batch_size=32, shuffle=True, num_workers=4)
Key Insight

The sliding-window approach to language modelling is extremely data-efficient. A 1024-token sequence yields 1024 training examples simultaneously — every token predicts the next. A dataset with 1 billion tokens, tokenised and sliced into 1024-length windows, provides roughly 1 million training batches per epoch with a batch size of 1000.

Appendix A — Key Takeaways

  • Tensors are PyTorch's fundamental data structure — multi-dimensional arrays with GPU acceleration and autograd support. Dtype matters: float32 for weights, int64 for token IDs.
  • Autograd tracks operations and computes gradients automatically via .backward(). PyTorch accumulates gradients — call optimizer.zero_grad() every step to avoid incorrect gradient accumulation.
  • All models subclass nn.Module — define layers in __init__, computation in forward(). Always call model(input), never model.forward(input) directly.
  • Always move both model and data to the same device before operations. Use device = torch.device('cuda' if torch.cuda.is_available() else 'cpu') for portable code.
  • DataLoader handles batching, shuffling, and parallel data loading for efficient training. Implement __len__ and __getitem__ in your custom Dataset; let DataLoader handle the rest.
