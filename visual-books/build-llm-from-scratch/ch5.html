Chapter 5

Pretraining on Unlabeled Data

7 Lessons 7-Question Quiz ~22 min read Training

Chapter 4 left us with a fully assembled GPT model — every layer wired up, every weight randomly initialised. At this point the model exists but knows nothing; it will produce incoherent nonsense for any prompt you give it. Pretraining is the process that turns random noise into a model capable of fluent, coherent language. This chapter covers the entire pretraining loop from end to end: computing a meaningful loss signal, running gradient updates that push weights in the right direction, tracking progress with both training and validation curves, and finally generating text using several different decoding strategies once training is underway.

By the end of the chapter you will also know how to checkpoint your model mid-training (so you can resume if something crashes) and how to load OpenAI's publicly released GPT-2 weights into your own custom architecture — giving you a shortcut to a high-quality starting point without running the full pretraining yourself. These skills form the practical backbone of everything that follows in the book, from supervised fine-tuning to RLHF.

5.1

Text Generation Recap

Before diving into the training machinery, it is worth revisiting exactly what the GPT model produces during a forward pass. Given a batch of token-ID sequences of shape [B, T], the model outputs a tensor of logits with shape [B, T, vocab_size]. Each of the T positions holds a raw, unnormalised score for every token in the vocabulary — essentially the model's "vote" for what token should come next. At inference time, you take the logits at the final position (index T-1), apply softmax to turn the scores into a probability distribution, then either pick the argmax (greedy decoding) or sample from the distribution to obtain the next token ID.

During training, however, we exploit a critical advantage: we already know the correct next token at every position in the sequence. We can therefore compute T separate next-token predictions in a single forward pass and compare each one against its ground-truth label. This technique is called teacher forcing. Concretely, if your input token sequence is [t₀, t₁, t₂, …, t_{T-1}], the corresponding target sequence is shifted one position right: [t₁, t₂, t₃, …, t_T]. The model sees the full input, predicts a distribution at each position, and is immediately supervised against the next real token — all in one vectorised operation.

Teacher Forcing: Input vs Target Alignment
Teacher forcing: the model predicts the next token at every position. At position 0, input "The" should predict target "cat"; at position 1, "cat" predicts "sat", etc. All T predictions happen in one forward pass.

Teacher forcing is highly efficient but it does introduce a subtle training-inference mismatch. During inference the model conditions on its own (potentially imperfect) previous outputs, whereas during training it always has the perfect ground-truth context. For large language models this so-called exposure bias is rarely a practical problem because the enormous diversity of the training corpus means the model sees enough variation to generalise, but it is worth keeping in mind when analysing failure modes of smaller models trained on narrow datasets.

5.2

Cross-Entropy Loss

Cross-entropy loss is the standard objective for next-token prediction, and understanding it intuitively helps you debug training runs. The formula for a single token is simple: −log(p_correct_token), where p_correct_token is the probability the model assigned to the token that actually appeared next in the text. If the model is perfectly confident and correct, it assigns probability 1.0 and the loss contribution is −log(1) = 0. Any uncertainty or incorrect prediction pushes the loss above zero. Because the logarithm grows very steeply near zero, assigning even a tiny probability to the wrong token incurs a large penalty — this is by design, as it strongly encourages the model to concentrate probability mass on the correct answer.

A useful sanity check is to compute the expected loss at initialisation. A freshly initialised model with random weights will distribute probability roughly uniformly across all 50,257 tokens in GPT-2's vocabulary, giving each token a probability of about 1/50257 ≈ 1.99 × 10⁻⁵. Plugging that into the formula: −log(1.99 × 10⁻⁵) ≈ 10.82. Whenever you start training a GPT-style model you should see an initial loss very close to this value — if the starting loss is dramatically higher or lower, something is wrong with the initialisation or the data pipeline. In practice we flatten the [B, T, V] logit tensor to [B*T, V] so PyTorch's F.cross_entropy can vectorise the computation across all token positions in a single efficient call.

One subtlety is how PyTorch handles the reduction. By default F.cross_entropy averages the per-token losses across the entire flattened batch, giving a single scalar. This means the effective loss scale depends on the sequence length and batch size — two hyperparameters that may change across experiments. Some practitioners use reduction='sum' instead and divide manually to get more explicit control, but for most purposes the default mean reduction works well and matches the convention used by OpenAI's original GPT training code.

logits = model(input_ids)               # [B, T, vocab_size]
loss = F.cross_entropy(
    logits.view(-1, logits.size(-1)),   # [B*T, vocab_size]
    targets.view(-1)                    # [B*T]
)
6-Step Training Pass
⊕ Forward pass → logits [B, T, vocab_size]
⊕ Flatten logits & targets [B*T, vocab_size] and [B*T]
⊕ Cross-entropy: −log(p[correct]) one scalar loss per token position
⊕ Mean over batch single scalar loss value
⊕ loss.backward() compute gradients for every parameter
⊕ optimizer.step() update weights in direction of −∇loss
Every training step follows this 6-stage cycle. Steps 1-4 are the forward pass; 5-6 are the backward pass and parameter update.
Cross-Entropy Loss Curve: −log(p)
The −log(p) curve shows why even a small drop in predicted probability causes a large loss spike. At p=1 the loss is zero; at p=0.01 the loss is 4.6 — a steep penalty for low-confidence correct predictions.

5.3

Train / Validation Loss Curves

Tracking loss on a held-out validation set is one of the most important habits in machine learning, and it becomes especially critical during LLM pretraining where a single run can take days or weeks. The validation set is a slice of text that is never used to compute gradients — it exists purely as an unbiased measuring stick for how well the model generalises to unseen data. You evaluate it periodically (say, every 100 or 500 steps) and record both the training loss (averaged over recent batches) and the validation loss. Plotting both gives you the famous train/val loss curve that tells the complete story of your training run at a glance.

The relationship between the two curves reveals whether training is healthy. Ideally both curves slope downward together, indicating the model is genuinely learning patterns that transfer to new text rather than memorising the specific sequences it was trained on. When the training loss continues to fall but the validation loss stops improving or begins to creep back up, the model has entered the overfitting regime — it is getting better at the training data specifically rather than at language in general. This is also called the generalisation gap, and a growing gap is a clear signal to stop training or apply regularisation. At true LLM pretraining scale (hundreds of billions of tokens), both curves typically improve together for a very long time, because the massive dataset dwarfs the model's capacity to memorise individual examples.

In the book's toy example you will see the gap emerge relatively quickly because the training corpus is tiny (a few chapters of a novel). This is intentional — it lets you witness overfitting in a short training run so you can recognise the pattern. One practical trick: log loss in nats (base-e) rather than bits (base-2) to stay consistent with PyTorch's default, and convert to perplexity (exp(val_loss)) for a more interpretable metric. A perplexity of 10 means the model is, on average, as uncertain as if it were choosing uniformly among 10 equally likely tokens at each position.

Step-through: Diagnosing Loss Curves Step 1 / 4

Healthy Training

Both train and val curves decline together with only a small gap. The model is genuinely learning generalisable language patterns. No action needed — keep training.

Overfitting Detected

Train loss keeps falling but val loss rises. The model is memorising training data. Action: stop training, reduce model capacity, increase dropout, or add more data.

Underfitting

Both curves plateau at a high value and stop improving. The model cannot learn the data well enough. Action: increase model size, increase learning rate, or train longer.

Loss Spike (Exploding Gradients)

A sudden sharp increase in loss indicates exploding gradients. Action: apply gradient clipping (clip_grad_norm_ = 1.0), reduce learning rate, or check for data corruption.

5.4

The Training Loop

model.train()
for epoch in range(num_epochs):
    for input_ids, target_ids in train_loader:
        optimizer.zero_grad()
        logits = model(input_ids)
        loss = F.cross_entropy(
            logits.view(-1, logits.size(-1)),
            target_ids.view(-1)
        )
        loss.backward()
        torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
        optimizer.step()

The training loop above may look simple, but every single line is load-bearing. The call to optimizer.zero_grad() at the top is essential because PyTorch, by default, accumulates gradients into each parameter's .grad attribute rather than overwriting them. This behaviour exists to support advanced techniques like gradient checkpointing, where you deliberately sum gradients across multiple micro-batches before stepping. In a standard per-batch loop, however, you must explicitly zero the accumulated gradients at the start of each step, or you will be mixing gradient signals from the current batch with leftover signals from previous batches — a subtle bug that causes training to diverge in unpredictable ways.

Gradient clipping via clip_grad_norm_ guards against a different failure mode: exploding gradients. Early in training, or whenever the loss surface has very high curvature, the backward pass can compute gradient vectors with extremely large magnitudes. A single step of unconstrained gradient descent in this situation can fling the parameters far from any sensible region of the loss landscape, often destroying all the progress made in previous steps. Gradient clipping rescales the entire gradient vector (across all parameters at once) so that its L2 norm never exceeds the threshold — 1.0 is the value used in the GPT-2 paper and is almost universally adopted. Crucially, it rescales proportionally rather than clipping per-parameter, so the direction of the update is preserved; only the magnitude is capped.

The choice of AdamW as the optimizer deserves its own explanation. Adam tracks both the first moment (mean gradient) and the second moment (variance of gradients) for each parameter, which produces adaptive per-parameter learning rates that work far better than a single global learning rate for the heterogeneous parameters in a Transformer. However, vanilla Adam's original weight decay implementation is flawed at scale — weight decay is applied to the update that already includes the adaptive learning rate scaling, which means parameters with small gradients receive less regularisation than parameters with large gradients, an unintended asymmetry. AdamW fixes this by applying weight decay directly to the raw parameter values, completely decoupled from the gradient update. For large models this distinction meaningfully affects both final quality and training stability.

Step-through: One Training Iteration Step 1 / 6

Step 1: optimizer.zero_grad()

Clear all accumulated gradients from the previous step. Without this, PyTorch adds new gradients to old ones, corrupting the update direction.

Step 2: Forward Pass

Pass input tokens through all 12 transformer blocks. The output is a tensor of shape [B, T, 50257] — one score per vocabulary token per position.

Step 3: Compute Cross-Entropy Loss

Flatten logits and targets, then compute −log(p) for every correct token. Average across all positions to get a single scalar loss.

Step 4: loss.backward()

Reverse-mode automatic differentiation traverses the computation graph backwards, computing ∂loss/∂W for every parameter in the model.

Step 5: Gradient Clipping

If the global gradient norm exceeds the threshold (1.0), all gradients are rescaled proportionally. The update direction is preserved; only its magnitude is capped.

Step 6: optimizer.step()

AdamW updates each parameter using its adaptive learning rate (from momentum estimates), then applies weight decay directly to the parameter values. One step complete.

5.5

Decoding Strategies

Once training is underway, you will want to periodically generate text from the model to qualitatively assess progress. The strategy you use to pick the next token — the decoding strategy — has a profound effect on what you observe, entirely independent of the model's learned weights. Understanding the trade-off between determinism and diversity is therefore critical both for evaluating your model during training and for building useful generation pipelines afterwards. The same model can sound robotically repetitive or refreshingly creative depending entirely on how you decode its output distribution.

Greedy decoding is the simplest strategy: always select the token with the highest probability, i.e. argmax over the logits. This is fully deterministic — the same prompt always produces the same output — and it is fast because no sampling is needed. The downside is a strong tendency toward repetitive loops. Once the model finds a locally high-probability phrase it tends to repeat it indefinitely, because at each step the same token keeps winning the argmax. Temperature sampling addresses this by dividing the raw logits by a scalar T before applying softmax. When T < 1, differences between logit values are amplified, making the distribution sharper (the highest-probability tokens dominate even more), which produces more focused but still stochastic outputs. When T > 1, logit differences are compressed, flattening the distribution so lower-probability tokens get a real chance — outputs become more varied and surprising, though also potentially less coherent. Setting T = 1.0 leaves the original distribution untouched.

probs    = F.softmax(logits / temperature, dim=-1)
idx_next = torch.multinomial(probs, num_samples=1)

Top-k sampling constrains the sampling pool to only the k most probable tokens at each step, setting all other logits to -infinity (which becomes zero after softmax). This prevents the model from ever accidentally sampling a low-probability token that happens to be grammatically or factually wrong. A typical value is k = 50. Temperature and top-k are commonly combined: apply top-k first to restrict the candidate set, then apply temperature to reshape the distribution over those k candidates. This layered approach gives fine-grained control — you can have creative diversity (high temperature) while still excluding obvious garbage tokens (low k). A related technique, top-p (nucleus) sampling, dynamically adjusts the cutoff by taking only enough of the most probable tokens to cover a cumulative probability mass of p (e.g. 0.9), which adapts better to situations where the model is very confident (few tokens needed to reach 0.9) versus very uncertain (many tokens needed).

In practice, production systems almost always combine temperature with either top-k or top-p, and sometimes both. The exact values to use are not determined by theory — they are tuned empirically by running human evaluations on model outputs. For the purposes of this book, starting with temperature = 1.0 and top_k = 50 gives reasonable outputs from a partially trained model and makes qualitative progress easy to observe as training continues.

top_k_logits, _ = torch.topk(logits, k)
logits[logits < top_k_logits[..., -1:]] = float('-inf')
probs = F.softmax(logits, dim=-1)

5.6

Saving and Loading Weights

Pretraining runs can last hours or days even on reasonably powerful hardware, which makes checkpointing — periodically saving the model's state to disk — absolutely essential. The most common disaster scenario is a job crash or GPU out-of-memory error partway through training: without checkpoints you would have to restart from scratch. Beyond crash recovery, checkpoints let you compare models from different points in training (sometimes an earlier checkpoint generalises better to your specific downstream task than the final one), and they are the artifacts you share with collaborators or deploy to production. The habit of saving checkpoints regularly is one of the most valuable practices you can build from your very first training run.

There is an important detail about what to save. It is tempting to save only the model's state_dict (the parameter tensors), but this is insufficient if you ever want to resume training rather than just run inference. The optimizer — especially AdamW — maintains internal momentum buffers for every parameter, representing a running average of past gradients. These buffers encode "momentum" that took many steps to build up. If you restart training without them, the optimizer effectively starts cold and you often see a brief but harmful spike in the loss before it recovers. Always save both the model state and the optimizer state, along with the current epoch and loss value for bookkeeping. The map_location='cpu' argument when loading ensures the checkpoint can be read on any machine, regardless of whether a GPU is present — PyTorch will move tensors to the appropriate device after loading.

What Goes Into a Checkpoint
Always save model + optimizer states together, plus metadata (epoch, loss). Without optimizer state, resuming training causes a loss spike because AdamW's momentum buffers are lost.

For production-scale models there are more sophisticated checkpointing strategies. Large models (tens of billions of parameters) have checkpoints too big to save after every step, so practitioners save every few thousand steps and use rolling checkpoints that keep only the last N saved states to avoid filling up disk. Frameworks like DeepSpeed and FSDP have their own sharded checkpoint formats for distributed training, where parameters are spread across multiple GPUs. The principles remain the same — save model state, optimizer state, and training metadata — but the implementation details become more complex as you scale up.

# Save
torch.save({
    'model_state':     model.state_dict(),
    'optimizer_state': optimizer.state_dict(),
    'epoch':           epoch,
    'loss':            loss
}, 'gpt2_checkpoint.pt')

# Load
checkpoint = torch.load('gpt2_checkpoint.pt', map_location='cpu')
model.load_state_dict(checkpoint['model_state'])
5.7

Loading OpenAI GPT-2 Weights

Training a GPT-2-scale model from scratch on a consumer GPU would take weeks or months. Fortunately, OpenAI released the GPT-2 weights publicly in 2019, and loading them into your own from-scratch implementation is a powerful shortcut. Rather than initialising your model randomly and pretraining from zero, you can start with weights that already encode a rich understanding of English language, factual knowledge, and basic reasoning. From this standing start you can fine-tune on your own task in a matter of hours. This is one of the earliest and most important demonstrations of the value of publicly releasing pretrained language model weights — a practice that has since become a cornerstone of the open-source AI ecosystem.

The mechanics of loading external weights require careful attention to naming conventions. When you build a GPT model from scratch using your own module names, the keys in your state_dict will look different from the keys in OpenAI's checkpoint. For example, OpenAI stores the combined query-key-value projection as h.0.attn.c_attn.weight using a 1D convolution representation, whereas your implementation might use a standard nn.Linear with a name like blocks.0.attn.qkv.weight. You need to write a mapping function that iterates through both dictionaries and copies tensor values from matching layers by hand. An additional gotcha: OpenAI uses a transposed weight convention for some linear layers (a legacy of their Conv1D implementation), so you will need to transpose certain tensors before copying them or your matrix multiplications will compute the wrong thing. The book walks through this mapping procedure step by step, which is a useful exercise for building intuition about how model internals connect to saved checkpoint files.

Once the weights are loaded successfully, you can verify the transfer worked by running the model on a simple prompt and checking that it produces fluent, coherent text immediately — without any training on your machine. You can also compare the validation loss against the baseline: a correctly loaded GPT-2 Small should achieve a validation perplexity consistent with OpenAI's published evaluation results. Any significant deviation signals that some weights were mapped incorrectly. This sanity-check habit — verify that loaded weights produce expected metrics before doing anything else — is standard practice in the field and will save you from many hours of confusing debugging downstream.

from transformers import GPT2Model
gpt2 = GPT2Model.from_pretrained('gpt2')

Chapter 5 — Key Takeaways

  • Cross-entropy loss measures prediction quality: −log(p_correct). Random baseline ≈ 10.8 for a 50k-token vocabulary. Lower is better.
  • Training loop: forward → loss → backward → clip → step → repeat. This is the complete cycle for every batch.
  • AdamW + gradient clipping (norm = 1.0) is the standard pretraining setup. AdamW decouples weight decay; clipping prevents exploding gradients.
  • Temperature controls output randomness; top-k restricts the sampling vocabulary at each decoding step. Both can be combined.
  • Always track both train and val loss to detect overfitting early. Healthy pretraining shows both curves declining together.
  • GPT-2 weights are public and reusable. Loading pretrained weights via weight-name mapping gives your custom model a strong starting point for fine-tuning.
