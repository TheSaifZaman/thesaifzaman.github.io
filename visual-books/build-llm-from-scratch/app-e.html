Appendix E

LoRA Fine-tuning

5 Lessons 6-Question Quiz ~20 min read LoRA PEFT Fine-tuning

LoRA (Low-Rank Adaptation) is arguably the most practically impactful technique to emerge from LLM research since GPT-2. It makes fine-tuning large language models accessible on consumer hardware — a single GPU — without sacrificing meaningful performance. Before LoRA, fine-tuning a 7B-parameter model required storing full model weights, optimizer states, and gradients: roughly 56 GB in fp32, far exceeding what consumer GPUs can hold. After LoRA, the same 7B model can be fine-tuned on a 24 GB GPU with results that approach full fine-tuning quality.

This appendix explains the mathematical foundation of LoRA, how to implement it in PyTorch, the training workflow, and what the empirical results look like compared to full fine-tuning. By the end you will be able to apply LoRA to any transformer model and understand the rank–quality trade-off that governs its behaviour.

E.1

Why LoRA? The Parameter Efficiency Problem

Fine-tuning GPT-3 (175 billion parameters) in full float32 precision requires roughly 700 GB of GPU memory for model weights alone — before accounting for optimizer states (Adam requires 2× the weights for momentum and variance estimates, another 1.4 TB) and activation gradients. In practice, the total memory requirement for full fine-tuning GPT-3 exceeds 2 TB. Even with bfloat16 mixed precision and modern distributed training, you need dozens of A100 GPUs. This is simply not accessible to individuals, startups, or research labs without cloud-scale infrastructure.

LoRA (Low-Rank Adaptation, Hu et al. 2021) solves this by observing that the weight changes during fine-tuning are low-rank — they lie in a much smaller subspace than the full weight matrix. Instead of updating W (a large matrix), LoRA freezes W and adds a small trainable decomposition ΔW = A·B, where A and B are tiny matrices. Only A and B are trained, reducing the trainable parameter count by 100–10,000× depending on the rank chosen.

The practical impact is dramatic. A 7B-parameter model fine-tuned with LoRA (rank=8) on the Query and Value projections adds roughly 33 million new trainable parameters — just 0.47% of the original. This fits easily on a single 24 GB GPU. Optimizer states for those 33M parameters require only ~250 MB, compared to ~56 GB for the full model. The fine-tuned model achieves results that are competitive with full fine-tuning across a wide range of tasks.

Key Insight

A 7B-parameter model fine-tuned with LoRA (rank=8) requires only ~33M new parameters — 240× fewer than full fine-tuning. The pretrained weights are frozen and stored once; only the tiny A and B matrices are trained, stored in optimizer states, and saved as the fine-tuned model checkpoint.

E.2

Low-Rank Decomposition

Every weight matrix W in a neural network is an m×n matrix. A low-rank approximation of a change ΔW factors it as ΔW = A · B where A ∈ ℝ^(m×r) and B ∈ ℝ^(r×n), with rank r much smaller than min(m, n). For a typical attention projection in GPT-2 where m=n=768, a full ΔW would have 768×768 = 589,824 parameters. With rank r=8, A has 768×8 = 6,144 parameters and B has 8×768 = 6,144 parameters — a total of 12,288, representing a 48× reduction.

During the forward pass, the computation becomes: output = xW + x(AB) = x(W + AB). W is frozen — it never changes. A and B are trainable. The addition W + AB is what the fine-tuned model computes, without any architectural change to the network structure. At initialisation, A is sampled from a Gaussian distribution and B is set to all zeros, which ensures that ΔW = A·B = 0 at the start of training. This means the LoRA layers initialise as exact copies of the pretrained model, with zero perturbation — a critical property for stable fine-tuning.

The rank r is the key hyperparameter. Higher rank means more expressive power (ΔW can capture more complex adaptations) but more trainable parameters and memory. Lower rank is more parameter-efficient but may underfit complex tasks. Typical choices are r=4 for fast, lightweight fine-tuning; r=8 as the balanced default; r=64 for tasks that need near-full-fine-tuning quality. A common heuristic: start with r=8 and only increase if you observe clear underfitting on the validation set.

E.3

Implementing LoRA Layers

Implementing LoRA requires replacing the target linear layers in the model with a custom LoRALinear module that wraps the original layer. The implementation is compact: freeze the original weight, add two small parameter matrices lora_A and lora_B, and add their contribution to the forward pass. The scaling factor alpha/rank is applied to the LoRA output to keep the update magnitude stable regardless of which rank value you choose. Without this scaling, using rank=8 vs rank=64 would require retuning the learning rate — the scaling factor normalises the update magnitude across rank choices.

The initialisation of lora_A and lora_B follows a specific convention: A is initialised with a small Gaussian (standard deviation of 0.01) and B is initialised to exactly zero. Because B=0, the initial ΔW = A·B = 0 regardless of A's values. This is mathematically equivalent to starting fine-tuning from the exact pretrained model weights, with zero perturbation. If B were initialised non-zero, the model's behaviour at the start of fine-tuning would be different from the pretrained model — potentially harming performance on tasks the model already handles well.

In practice, LoRA is typically applied to the Query (W_q) and Value (W_v) projection matrices in the attention mechanism. Hu et al. found that this configuration alone captures most of the fine-tuning benefit. Adding LoRA to W_k and W_o provides marginal improvements on some tasks but roughly doubles the number of LoRA parameters. The feed-forward layers are usually left unchanged, as they appear to be less sensitive to task-specific adaptation — though this remains an active area of study.

class LoRALinear(nn.Module):
    def __init__(self, in_features, out_features, rank=8, alpha=16):
        super().__init__()
        self.linear = nn.Linear(in_features, out_features, bias=False)
        self.linear.weight.requires_grad = False  # freeze original

        # LoRA matrices
        self.lora_A = nn.Parameter(torch.randn(rank, in_features) * 0.01)
        self.lora_B = nn.Parameter(torch.zeros(out_features, rank))  # B=0
        self.scale = alpha / rank  # normalisation factor

    def forward(self, x):
        # base output + scaled LoRA update
        return self.linear(x) + self.scale * (x @ self.lora_A.T @ self.lora_B.T)
E.4

Training with LoRA

The LoRA training workflow involves four steps. First, load the full pretrained model — all parameters at their pretrained values. Second, replace the target layers (W_q and W_v in each attention block) with LoRALinear instances. Third, freeze all original weights by setting requires_grad=False — the original W matrices will never receive gradient updates. Fourth, run the standard training loop; only the A and B matrices have requires_grad=True, so the backward pass only computes gradients for them. The Adam optimizer only maintains momentum and variance for these small matrices.

Memory savings come from two sources. First, the frozen weights do not need gradient storage (saving one copy of the weights per parameter). Second, the optimizer states are only maintained for the LoRA parameters (33M instead of 7B for a 7B model), saving another ~250 MB vs ~56 GB. The forward and backward activations are similar in size to full fine-tuning (they depend on batch size and sequence length, not the number of trainable parameters), but with a smaller batch size — which fits in the reduced memory budget — the total memory can drop by 10–20× compared to full fine-tuning.

After training, the LoRA weights can be merged back into the base weights to eliminate any inference overhead. The merged weight is simply W_merged = W + (alpha/rank) × B·A. This merged model is architecturally identical to the original — no LoRA layers, no extra computation at inference. The only difference is the values of the W_q and W_v matrices, which now encode both the pretrained representations and the fine-tuned adaptations. Serving the merged model has exactly the same latency as the base model.

def apply_lora(model, rank=8, alpha=16):
    for block in model.trf_blocks:
        d = block.attn.d_model
        # Replace Q and V projections with LoRA versions
        block.attn.W_q = LoRALinear(d, d, rank, alpha)
        block.attn.W_v = LoRALinear(d, d, rank, alpha)
    # Collect only trainable parameters (A and B matrices)
    trainable = [p for p in model.parameters() if p.requires_grad]
    return trainable

# Training loop — identical to full fine-tuning
trainable_params = apply_lora(model, rank=8, alpha=16)
optimizer = torch.optim.Adam(trainable_params, lr=2e-4)

E.5

Results Comparison

LoRA consistently matches or closely approaches full fine-tuning quality across a wide range of tasks — instruction tuning, code generation, GLUE benchmarks, summarisation — while requiring dramatically fewer resources. Hu et al. demonstrated this on GPT-3 scale (175B) and the results generalise to smaller models. For instruction tuning a 7B-parameter model, LoRA with rank=8 applied to Q and V projections achieves performance within 2–3% of full fine-tuning on most benchmarks while using 50× less training memory.

The rank hyperparameter governs the quality–efficiency trade-off. Rank 4 is extremely lightweight — fewer than 15 million parameters for a 7B model — and works well for simple task adaptations where the pretrained model already has relevant capabilities. Rank 8 is the practical sweet spot for most instruction tuning and domain adaptation tasks, providing most of the benefit of higher ranks at much lower cost. Rank 64 approaches full fine-tuning quality for complex tasks but at the cost of more parameters and memory — at this point, the savings versus full fine-tuning are less dramatic, though still significant.

One empirical finding that is particularly important: applying LoRA to more weight matrices (Q, K, V, O, and possibly FFN) can improve results on tasks that require broad adaptation, but the returns diminish quickly. For most instruction tuning scenarios, W_q + W_v is sufficient. The intuition is that Q and V control what information is retrieved and how it is processed, while K is less critical for adapting to new task formats, and the FFN layers already capture much of the task-relevant knowledge during pretraining.

Key Insight

After training, merge the LoRA weights into the base model: W_merged = W + (alpha/rank) × B·A. The merged model has exactly the same architecture and inference cost as the base model — no LoRA overhead, no adapter loading, no latency penalty. LoRA's efficiency benefits apply only at training time, not deployment time.

Appendix E — Key Takeaways

  • LoRA (Low-Rank Adaptation) freezes pretrained weights and injects trainable low-rank decomposition matrices A·B into attention layers, reducing trainable parameters by 100–10,000×.
  • B is initialised to zero, ensuring ΔW = A·B = 0 at the start of training — the model begins fine-tuning from the exact pretrained state with no perturbation.
  • The alpha/rank scaling factor normalises the LoRA update magnitude so that changing rank does not require retuning the learning rate.
  • Apply LoRA to W_q and W_v in attention — this captures most fine-tuning benefit. Adding W_k and W_o provides marginal gains; FFN layers are usually left frozen.
  • Merge weights after training: W_merged = W + (alpha/rank) × B·A. The merged model has zero inference overhead — same architecture and latency as the base model.
