Chapter 2

Working with Text Data

8 Lessons 7-Question Quiz ~35 min read Text Processing

Before a neural network can understand or generate text, that text must be transformed into numbers. This chapter walks through every step of that pipeline — from raw characters all the way to the combined token-plus-position embedding vectors that feed into a transformer's first layer. You will build an intuition for why each step exists, how different design choices (vocabulary size, window stride, positional encoding scheme) affect model behavior, and how GPT-2 specifically implements each of these components.

2.1

Word Embeddings: Why We Need Them

At the most fundamental level, a neural network is a composition of linear algebra operations — matrix multiplications, additions, and non-linear activations. All of these operations require floating-point numbers as inputs. Raw text, a sequence of Unicode characters, is not a number. Therefore the very first challenge of building a language model is answering the question: how do we represent words as numbers in a way that preserves meaning?

The naive answer is one-hot encoding: assign each word in your vocabulary an integer index, then represent that word as a vector of all zeros with a single 1 at the word's index. A vocabulary of 10,000 words would give each word a 10,000-dimensional vector that is 99.99% zeros. This approach has two fatal weaknesses. First, it is catastrophically memory-inefficient — GPT-2's vocabulary of 50,257 tokens would require a 50,257-dimensional sparse vector per token. Second, and more importantly, one-hot vectors carry zero semantic information: the vectors for "cat" and "kitten" are just as far apart as the vectors for "cat" and "universe". The model has no way to leverage the fact that synonyms, related words, or grammatically similar words should behave similarly.

Dense embeddings solve both problems. Instead of a sparse binary vector of length V (vocabulary size), each word is mapped to a compact, real-valued vector of length d (the embedding dimension, often 128–1024). These vectors are learned: during training, the model adjusts them so that words appearing in similar contexts end up with similar vectors. In the resulting embedding space, "cat" and "kitten" are geometrically nearby, arithmetic relationships like king − man + woman ≈ queen emerge naturally, and the model can generalise: if it learns something about "cats", some of that knowledge transfers to "kittens".

Classical embedding methods like Word2Vec (Mikolov et al., 2013) and GloVe (Pennington et al., 2014) produced standalone, context-independent embedding tables trained on unlabeled corpora. Modern LLMs go further: they learn their own embedding tables end-to-end as part of training. Every weight update that improves next-token prediction also refines the embeddings. The result is context-aware representations — the same token "bank" gets a different effective representation depending on whether the surrounding context is about rivers or finance, because the attention mechanism modulates the embedding before it reaches the output layer.

Key Insight: The embedding layer is the bridge between the discrete world of tokens (integers) and the continuous world of neural network mathematics (real-valued vectors). Everything downstream — attention, feed-forward layers, layer normalisation — operates on these vectors.

2.2

Tokenizing Text

Before we can assign integer IDs to words, we must first decide what counts as a "word" — the process of tokenization. At first glance, splitting by whitespace seems obvious: "The cat sat" → ["The", "cat", "sat"]. But real text is messier. Punctuation needs to be separated ("Hello," → ["Hello", ","]), contractions behave differently across languages ("don't" → ["don", "'", "t"] or ["don't"]?), and some languages like Chinese have no whitespace between words at all. The tokenizer's job is to convert a raw Unicode string into a clean, well-defined sequence of string tokens.

A simple rule-based tokenizer might split on whitespace and on punctuation, handling edge cases with regular expressions. For example, the sentence "Hello, world!" could be split by scanning character-by-character: accumulate alphabetic characters into a word token, then emit any punctuation as its own token. This produces ["Hello", ",", " world", "!"] — five distinct tokens that together faithfully reconstruct the original string when concatenated.

Building a vocabulary then means scanning a large corpus, collecting every unique token encountered, and assigning each a consecutive integer ID. The result is two dictionaries: word_to_id (encode: string → int) and id_to_word (decode: int → string). Encoding turns text into a list of integers; decoding turns a list of integers back into text. The vocabulary must be fixed before training — tokens not seen in the training corpus would be "out-of-vocabulary" (OOV).

Simple word-level tokenizers have a fundamental weakness: out-of-vocabulary words. If the vocabulary was built from the training corpus and a test sentence contains a proper noun, a brand name, or a misspelling not seen during training, the tokenizer has no token for it. One workaround is a special [UNK] token, but this loses all information about the unknown word. Sub-word tokenization methods — the most important being Byte-Pair Encoding (BPE), covered in Lesson 2.5 — solve this problem by decomposing unknown words into smaller known pieces.

Rule of Thumb: In the GPT-2 tokenizer, a token is roughly 4 characters on average. A 1,000-word essay is approximately 750–800 tokens. LLM context windows and pricing are measured in tokens, not words, so understanding tokenization is practically important.

import re

class SimpleTokenizer:
    def __init__(self, vocab):
        self.word_to_id = vocab
        self.id_to_word = {v: k for k, v in vocab.items()}

    def tokenize(self, text):
        # Split on whitespace, then isolate punctuation
        tokens = re.findall(r"\b\w+\b|[^\s\w]", text)
        return tokens

    def encode(self, text):
        tokens = self.tokenize(text)
        return [self.word_to_id[t] for t in tokens]

    def decode(self, ids):
        return " ".join(self.id_to_word[i] for i in ids)

2.3

Token IDs: Converting Text to Numbers

Once we have a fixed vocabulary, encoding is the process of replacing every token string with its corresponding integer ID. The result is a sequence of integers — often called a token ID sequence or simply input IDs — that the model actually consumes. The inverse operation, decoding, converts a sequence of integer IDs back into human-readable text by looking up each ID in the reverse vocabulary mapping.

GPT-2 uses a vocabulary of 50,257 BPE tokens, so every token ID is an integer in the range [0, 50256]. When you encode the string "Hello world" using the GPT-2 tokenizer, you get a specific sequence: [15496, 995]. Token 15496 is the BPE subword piece corresponding to "Hello" and 995 corresponds to " world" (note the leading space is encoded as part of the token — GPT-2's BPE vocabulary treats "world" and " world" as two distinct tokens). Decoding [15496, 995] back produces exactly "Hello world".

The vocabulary acts as a two-way lookup table. Building it efficiently requires sorting all unique tokens from the training corpus and assigning IDs in a deterministic order (often alphabetical or by frequency). During inference, the model produces a probability distribution over all 50,257 token IDs, then samples or argmax-selects the next token ID, which is decoded to produce the next character of output text.

One important subtlety: whitespace handling. Some tokenizers strip all whitespace and add it back on decode. Others — like GPT-2's tiktoken — include leading spaces as part of tokens (e.g., "Ġcat" where Ġ represents a leading space). This means the same word encoded at the start of a sentence may have a different token ID than the same word in the middle of a sentence. This is not a bug; it is a deliberate design choice that makes decoding perfectly lossless — the tokenized form can always be decoded back to the exact original string.

Actual GPT-2 IDs: You can verify token IDs with tiktoken. "Hello" → 15496, " world" → 995, "." → 13, " the" → 262, " cat" → 3797. Notice " the" and " cat" have leading spaces embedded in the token — GPT-2 BPE includes spacing as part of the token representation.

2.4

Special Tokens: [BOS], [EOS], [PAD], [UNK]

In addition to regular word tokens, language model vocabularies include a set of special tokens — reserved entries that serve structural or control purposes rather than representing actual words. These tokens are essential for training stability, batch processing, and instructing the model about the boundaries of meaningful content. Different model families use different special tokens with slightly different conventions, so it is important to understand what each one does.

The [BOS] (Beginning of Sequence) token is prepended to every input sequence to signal to the model "this is the start of a new text". Without it, the model would need to infer from context whether it is in the middle of a document or at its very beginning — information that changes how it should behave. Conversely, the [EOS] (End of Sequence) token is appended to every training example. The model learns that generating an EOS token means "I am done generating" — which is how autoregressive models know when to stop at inference time.

The [PAD] (Padding) token is a technical necessity for batch processing. When training on batches of sequences, all sequences in the same batch must have the same length to fit into a tensor. Shorter sequences are right-padded with [PAD] tokens to match the length of the longest sequence. The attention mask tells the model to ignore padded positions during the attention computation so they don't affect gradients. Without padding, you would have to train one sequence at a time (batch size = 1), which is extremely slow.

The [UNK] (Unknown) token is used by word-level tokenizers to handle out-of-vocabulary (OOV) words encountered at inference time. If a test sentence contains a word never seen during training, the tokenizer replaces it with [UNK]. GPT-2, however, uses BPE tokenization — since BPE can represent any string using individual bytes as a fallback, it has no concept of OOV and therefore needs no [UNK] token. GPT-2's vocabulary has only a single special token: <|endoftext|> (ID 50256), which serves as both BOS and EOS separator between documents during pretraining.

GPT-2 Simplicity: GPT-2's vocabulary has exactly one special token — <|endoftext|> with ID 50256. It appears between documents in the pretraining corpus (e.g. "...end of article. <|endoftext|> Next article begins...") and signals document boundaries. No [PAD], no [UNK], no [CLS] — BPE eliminates the need for most special tokens.

2.5

Byte-Pair Encoding (BPE)

Byte-Pair Encoding (BPE) is the tokenization algorithm that powers GPT-2 and most modern LLMs. It elegantly solves the OOV problem by operating at the sub-word level: instead of a fixed vocabulary of whole words, it builds a vocabulary of sub-word pieces — common character sequences that can be assembled into any word. Any word the model has never seen can be broken down into known sub-word pieces, and if all else fails, individual bytes always exist in the vocabulary as a last resort.

The BPE algorithm starts from a character-level vocabulary (every individual character in the training corpus gets its own token) and iteratively merges the most frequently co-occurring adjacent pair of tokens into a single new token. For example, if "e" and "s" appear next to each other 10,000 times in the corpus — far more than any other pair — BPE creates a new token "es" and replaces all occurrences of the pair "e s" with the single token "es". This merge step is repeated thousands of times until the vocabulary reaches the desired size.

Consider a toy corpus containing the words "low", "lower", "newest", and "widest". Starting from characters: l, o, w, e, r, n, s, t, i, d. BPE might first merge "e"+"s" → "es" (because "newest", "widest" both contain "-est"), then "es"+"t" → "est", then "l"+"o" → "lo", and so on. After enough merges, common words like "the" or "ing" become single tokens; rare or new words get split into their component morphemes. The vocabulary size is a hyperparameter — GPT-2 uses 50,000 merge rules, yielding a final vocabulary of 50,257 tokens (50,000 merges + 256 base byte tokens + 1 special token).

Why does this design work so well? Because natural language is highly morphological — words share prefixes ("un-", "pre-"), suffixes ("-ing", "-tion", "-est"), and roots. BPE automatically discovers these units from the data. A word like "uncharacteristically" might be tokenized as ["un", "char", "acter", "ist", "ically"] — each piece is a common sub-word that appears in many other words, allowing the model to generalise across them. GPT-2 uses the tiktoken library for fast BPE encoding at inference time.

BPE in Practice: The same word can tokenize differently depending on context (leading space) and model. GPT-2 typically encodes " cat" (with a space) as a single token 3797, but "cat" at the start of a sentence is token 9246. Always use the model's own tokenizer — never a different model's tokenizer — to ensure consistency.

2.6

Sliding Window Data Sampling

With a tokenized corpus in hand, the next step is creating the actual training examples that will teach the model to predict the next token. This is done using a sliding window approach: a window of fixed width moves along the token sequence, and at each position it produces one (input, target) training pair. The input is the window's tokens and the target is those same tokens shifted one position to the right — i.e., for each input token, the corresponding target is the token that comes immediately after it in the corpus.

Concretely, if the context window size (also called block size or max_seq_len) is 4 and the token sequence is ["The", "cat", "sat", "on", "the", "mat"], then the first window produces: input = ["The", "cat", "sat", "on"], target = ["cat", "sat", "on", "the"]. Slide the window by 1 (stride = 1) and the second sample is: input = ["cat", "sat", "on", "the"], target = ["sat", "on", "the", "mat"]. Each (input, target) pair actually encodes 4 individual next-token prediction tasks simultaneously — one per position — because the loss is computed at every output position.

The stride parameter controls how much the window moves between consecutive samples. A stride of 1 maximises data diversity (consecutive samples overlap by context_size − 1 tokens) but consumes a lot of memory to store all the samples. A stride equal to context_size produces non-overlapping samples, which is more memory-efficient but may miss important context boundaries. GPT-2 was pretrained with a stride equal to the full context size (1024 tokens) for efficiency.

During training, many windows are batched together. If batch size is 4 and context size is 1024, the model processes a tensor of shape [4, 1024] as input and [4, 1024] as target simultaneously, computing loss across 4 × 1024 = 4,096 individual next-token prediction tasks in a single forward pass. This parallelism — training on every position in every sequence simultaneously — is one of the key efficiency advantages of the transformer over recurrent architectures, which had to process positions sequentially.

Data Efficiency: A single sliding window of context_size C produces C individual training signals (one per position). A batch of B windows produces B × C signals. For GPT-2 with C=1024 and B=512, that is 524,288 next-token predictions per training step — from a single forward + backward pass.

2.7

Token Embeddings

After tokenization and ID assignment, the embedding layer performs the critical transformation: it maps each integer token ID into a dense real-valued vector. Mechanically, this is just a lookup table (also called a weight matrix) of shape [vocab_size, d_model]. Given a token ID, the embedding layer returns the corresponding row of this matrix. For a sequence of N token IDs, the output is an N × d_model matrix — one vector per token position.

For GPT-2 Small, vocab_size = 50,257 and d_model = 768. The embedding matrix therefore contains 50,257 × 768 = 38,597,376 parameters — roughly 38.6 million learnable floats just in the embedding layer. This is actually the single largest weight matrix in GPT-2 Small, containing more parameters than any individual transformer block. At 4 bytes per float32 parameter, this matrix occupies about 147 MB of memory. In practice, the output projection (the final linear layer that produces logits) often shares weights with the embedding matrix (called weight tying), cutting the parameter count significantly.

The embedding vectors are initialised randomly (typically from a normal distribution with small standard deviation) and then updated through backpropagation just like any other weight matrix. Over training, tokens that appear in similar contexts — and therefore produce similar losses when their embeddings are swapped — converge to similar vector locations. This is the mechanism by which the model implicitly learns semantic relationships: "Paris" and "London" end up nearby in embedding space not because someone told the model they are both capitals, but because sentences containing "Paris" and sentences containing "London" are statistically similar.

A useful way to inspect learned embeddings is to apply dimensionality reduction (PCA or t-SNE) to project the high-dimensional embedding space down to 2D for visualisation. In such plots, you typically see clusters of semantically related words — countries near each other, numbers near each other, verbs in one region, nouns in another. This emergent geometric structure is one of the most striking demonstrations that the model has learned something meaningful about language from raw next-token prediction.

Weight Tying: Many LLMs including GPT-2 "tie" the input embedding matrix and the output projection matrix — they are literally the same tensor. This halves the parameter count for these layers and often improves performance, since the model must produce embeddings in a space that is also good for predicting which tokens come next.

2.8

Positional Embeddings

The transformer's self-attention mechanism is fundamentally permutation-equivariant: if you shuffle the tokens in the input sequence, the output is shuffled in exactly the same way. Unlike RNNs or CNNs, a vanilla transformer has no built-in sense of which token comes first or last. This is a serious problem for language modelling — the sentence "the dog bit the man" and "the man bit the dog" contain the same words but mean something very different. Without positional information, the attention mechanism would produce identical outputs for both.

The solution is positional embeddings: a second set of learned (or computed) vectors — one per position in the sequence — that are added element-wise to the token embeddings before the first transformer block. The combined vector at position i is: input[i] = token_embedding[token_id[i]] + positional_embedding[i]. Both are [d_model]-dimensional vectors, so addition is straightforward. The result carries both semantic information (what word it is) and positional information (where in the sequence it sits).

GPT-2 uses learned absolute positional embeddings: a trainable weight matrix of shape [max_seq_len, d_model] = [1024, 768]. Each row is a learned vector for one absolute position (position 0, 1, 2, …, 1023). These vectors are initialised randomly and updated by backpropagation, just like the token embeddings. The downside of absolute positional embeddings is that the model cannot generalise beyond its training context length — position 1025 was never seen during training. More modern approaches like RoPE (Rotary Position Embeddings, used in LLaMA) and ALiBi address this by using relative position representations that can extrapolate beyond training length.

The original "Attention Is All You Need" paper proposed sinusoidal positional encodings — fixed mathematical functions (sine and cosine at different frequencies) rather than learned vectors. The intuition is that each position gets a unique "fingerprint" of sine/cosine values across dimensions, with lower dimensions corresponding to lower frequencies (global position information) and higher dimensions to higher frequencies (local position information). While sinusoidal encodings work well and require no additional parameters, GPT-2's learned positional embeddings tend to perform slightly better in practice because they can adapt to the specific patterns in the training data.

Modern Alternative — RoPE: LLaMA, Mistral, and most post-2022 LLMs use Rotary Position Embeddings (RoPE), which encode relative position by rotating the query and key vectors before the dot-product attention. RoPE extrapolates better to long sequences and is now the dominant approach in frontier models.

Token and positional embeddings are both [d_model]-dimensional vectors. Adding them element-wise creates a combined representation that encodes both what the token is (semantic content) and where it is in the sequence (positional context).

Chapter 2 — Key Takeaways

  • Text must be converted to tokens, then token IDs for neural network processing. Raw strings cannot be used as neural network inputs.
  • BPE tokenization handles any word without out-of-vocabulary issues by decomposing unknown words into common sub-word pieces. GPT-2 uses tiktoken for fast BPE encoding.
  • GPT-2 uses exactly 50,257 BPE tokens (50,256 sub-word pieces + 1 special <|endoftext|> token) and 768-dimensional embeddings.
  • Sliding window sampling creates training pairs: for each context window of K tokens, the target is the same window shifted right by 1 position, providing K next-token prediction tasks per sample.
  • Positional embeddings inject token order information into the model. GPT-2 uses learned absolute positional embeddings added element-wise to token embeddings before the first transformer block.
  • The final input to a transformer is token_embedding + positional_embedding — a combined vector encoding both semantic identity and sequence position.
